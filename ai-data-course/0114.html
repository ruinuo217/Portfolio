MySQL 語法實戰與權限管理

資料庫建置、使用者授權、CRUD 增刪改查與查詢優化。

Part 1: 資料庫建置與權限

權限設定 (Grant)

管理 
-- 建立最高權限使用者 (可從任何主機 %)
GRANT ALL PRIVILEGES ON *.* TO 'mydbo'@'%' WITH GRANT OPTION;

-- 讓設定立即生效
FLUSH PRIVILEGES;

-- 修改密碼 (相容舊版驗證)
ALTER USER 'mydbo'@'%' IDENTIFIED WITH mysql_native_password BY 'Pass123';

Part 2: CRUD 資料操作 (增刪改)

INSERT (新增)

寫入

單筆與多筆寫入範例。

-- 寫法 1: 指定欄位
INSERT INTO users (u_email, u_name) VALUES
('alex@example.com', 'Alex');

-- 寫法 2: 多筆寫入 (VS Code 範例)
INSERT INTO `users`(`u_name`, `u_email`, `status`) 
VALUES 
('nnn', 'frank@imusm.net', '已審核'),
('nnn2', 'frank2@imusm.net', '已審核');

UPDATE (更新)

修改
注意： 務必加上 WHERE 條件，否則會修改整張表的資料！ 
UPDATE `users` 
SET `u_email` = 'test@abc.com', 
    `u_birth` = '2025-01-01'
WHERE `u_name` = 'nnn';

DELETE 與 軟刪除 (Soft Delete)

危險

比較「真的刪除」與「標記刪除」。

1. 硬刪除 (Hard Delete)
DELETE FROM users WHERE u_name = 'nnnnn'; -- 資料真的消失
2. 軟刪除 (Soft Delete)

透過 UPDATE 更新 delete_at 時間戳記，資料還在但被視為已刪除。

UPDATE `users` 
SET `delete_at` = NOW() 
WHERE `u_name` = 'nnnnn';

Part 3: SELECT 查詢語法

SELECT * 的壞處

觀念
  • 浪費流量： 把不必要的欄位全部傳輸。
  • 拖慢速度： 增加硬碟 I/O 讀取負擔。
  • 索引失效： 無法利用 Covering Index 優化查詢。
好習慣： SELECT u_email, u_name FROM users ... (只抓需要的欄位)

模糊查詢 (LIKE %)

技巧
語法說明
'%值%'包含 (Contains)
'%值'結尾是 (Ends with)
'值%'開頭是 (Starts with)
-- 找名字裡面有 "Amy" 的人
SELECT * FROM users WHERE u_name LIKE '%Amy%';
