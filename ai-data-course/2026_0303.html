AWS 雲端平台（EC2 + S3）

3/3 課程筆記

IAM 身分與權限管理

IAM 核心概念

觀念

AWS 帳號就像租一整棟機房，root 是總鑰匙（什麼都能開），IAM User 是個人門禁卡（只能開被授權的門）。正式環境絕對不用 root 做日常操作，而是為每位工程師建立獨立 IAM 帳號。

專有名詞說明比喻
Root AccountAWS 帳號的最高權限擁有者大樓總鑰匙
IAM User獨立的操作身份，可追蹤行為個人門禁卡
PolicyJSON 格式的權限規則（Allow/Deny）門禁卡的權限設定
Group一組 User 共用同一套 Policy部門權限群組
MFA多因素認證，登入需額外驗證碼門禁卡 + 指紋辨識
重點：root 只做帳務與 IAM 初始設定，日常操作一律用 IAM User。離職時停用個人帳號即可，不影響其他人。

IAM 實作流程

實作

建立 student-admin IAM User 的完整 7 步驟：

  • 用 root 帳號登入 AWS Console
  • 進入 IAM → Groups → 建立 Group：training-admin-group
  • 為 Group 附加 AdministratorAccess Policy
  • 建立 User：student-admin，加入 training-admin-group
  • 啟用 Console 登入（設定初始密碼）
  • 登出 root，改用 student-admin 重新登入驗證
  • 為 student-admin 啟用 MFA 多因素認證
注意：正式環境不應長期保留 AdministratorAccess，應改為最小權限 Policy。課堂為了方便教學才使用全權限。

練習：IAM 操作驗證

練習

用 student-admin 建立一台 EC2（iam-check-ec2），確認 CloudTrail 可追蹤操作到個人身份。

優秀回答：我用 student-admin 建立了 iam-check-ec2，在 CloudTrail Event history 中搜尋 RunInstances 事件，可看到 userIdentity.userName 是 student-admin 而非 root。這代表操作可追蹤到個人身份，若發生異常變更能快速鎖定責任人。在正式環境中，每位工程師都應有獨立 IAM 帳號，離職時可單獨停權而不影響其他人。

EC2 核心知識地圖

EC2 與網路基礎

觀念

EC2 是 AWS 的雲端主機服務。要讓一台 EC2 能被外部存取，需理解以下核心概念：

概念說明角色
RegionAWS 全球資料中心區域（如 ap-northeast-1 東京）資源邊界
VPCVirtual Private Cloud，你在 AWS 裡的私有雲網路私有雲網路
SubnetVPC 內的子網路，分 Public / Private子網段
IGWInternet Gateway，VPC 連外的大門對外出口
Route Table決定封包流向的路由表（0.0.0.0/0 → igw-xxx）流量路由
Security Group虛擬防火牆，控制進出流量防火牆
Public IP / EIP對外 IP 位址（EIP 為固定 IP，需額外綁定）對外位址

EC2 可被外部存取的最小判斷 4 點：

  • EC2 放在 Public Subnet 中
  • EC2 有分配 Public IPv4 位址
  • Subnet 的 Route Table 有指向 IGW 的路由（0.0.0.0/0 → igw-xxx）
  • Security Group 有開放對應的 Port（如 22、80）

AMI / EBS / Snapshot

觀念

三者的關聯與概念流程：AMI 開 EC2 → 掛 EBS → Snapshot 備份 → 可從 Snapshot 建自有 AMI

項目本質用途
AMI作業系統樣板（映像檔）建立新 EC2 的模板
EBS系統磁碟（Block Storage）EC2 的持久化儲存
SnapshotEBS 的時間點備份（增量備份）災難復原、跨區複製
Delete on Termination：勾選 = 終止 EC2 時自動刪除磁碟（課堂用，避免遺留費用）；不勾選 = 磁碟保留（可能產生遺留成本）。

Instance Lifecycle 與計費

觀念

EC2 的三種主要狀態與計費關係：

狀態運算費用EBS 費用說明
Running計費中計費中主機運行中
Stopped不計費仍計費關機但 EBS 保留
Terminated不計費預設刪除刪除主機

計費模式比較：

模式價格適用場景特性
On-Demand標準價（按秒計費）教學、開發測試隨開隨用，無綁約
Spot最低可達 1-2 折可中斷的批次運算便宜但 AWS 可能回收
Reserved約 6-7 折（1-3 年）長期穩定的正式環境綁約承諾
注意：Stop 不等於零成本！EBS 磁碟、Snapshot、閒置 EIP 都可能持續計費。

EC2 建置實作

建置前檢核清單

觀念

開機前確認以下 5 點：

  • Console 右上角 Region 是否正確（如 ap-northeast-1 東京）
  • IAM 身分非 root（應使用 student-admin）
  • 準備好 Key Pair（.pem 檔下載到本機）
  • 確認 SSH 來源 IP（選 My IP，不要開 0.0.0.0/0）
  • 知道結束後要 Stop 還是 Terminate

EC2 建立步驟

實作

完整 10 步驟建立 EC2 主機：

  • Step 1：EC2 Dashboard → Launch instances
  • Step 2：Name = aws12h-u1-web-01
  • Step 3：AMI 選 Ubuntu 22.04 LTS
  • Step 4：Instance type = t3.micro
  • Step 5：Key pair → 建立 aws12h-u1-key（下載 .pem）
  • Step 6：Network → 預設 VPC / Public Subnet / Auto-assign public IP: Enable
  • Step 7：Create security group → aws12h-u1-web-sg
  • Step 8：入站規則：SSH:22（My IP）+ HTTP:80（Anywhere）
  • Step 9：Storage: 8 GiB gp3
  • Step 10：Launch instance → 等待 Status = Running

Security Group 規則

實作

入站規則（Inbound Rules）設定：

類型Port來源理由
SSH22你的 IP（My IP）只讓管理者連線
HTTP800.0.0.0/0讓網站可被外部瀏覽
不要這樣設：SSH:22 開 0.0.0.0/0（全球都能暴力破解）、MySQL:3306 對外開放（資料庫直接暴露）、所有 Port 全開（形同虛設）。

SSH 連線

實作

本機終端機連線到 EC2：

  • chmod 400 aws12h-u1-key.pem（設定金鑰權限為唯讀）
  • ssh -i aws12h-u1-key.pem ubuntu@<EC2_PUBLIC_IP>

常見錯誤：

  • Permission denied：key 權限不是 400、或使用者名稱打錯（應為 ubuntu）
  • Connection timed out：SG 沒開 22、IP 填錯、EC2 不在 Running

練習：SSH 驗證

練習

SSH 連線後執行 whoami 與 uname -a 驗證環境。

優秀回答：SSH 連線成功後 whoami 顯示 ubuntu，uname -a 顯示 Linux 核心版本與 x86_64 架構。架構圖：瀏覽器 → EC2:80(Nginx) → 127.0.0.1:5000(Flask) → MySQL:3306(僅本機)。SG 入站只開 22(限管理IP) 和 80(對外)，3306 不對外是因為資料庫只需本機 Flask 存取，對外會有 SQL Injection 與暴力破解風險。

環境安裝與 Flask 實作

安裝 Nginx + Python + MySQL

實作

SSH 進入 EC2 後依序執行：

  • sudo apt update && sudo apt upgrade -y（更新套件清單與升級）
  • sudo apt install -y nginx python3-venv python3-pip mysql-server
  • sudo systemctl enable --now nginx（啟動並設定開機自動啟動）
  • sudo systemctl enable --now mysql
  • 驗證：systemctl status nginx 確認 active (running)
  • 驗證：systemctl status mysql 確認 active (running)

Flask + MySQL 最小留言板

實作

建立專案目錄與虛擬環境：

  • mkdir -p ~/myapp && cd ~/myapp
  • python3 -m venv venv && source venv/bin/activate
  • pip install flask pymysql gunicorn

MySQL 初始化 SQL：

SQL 指令：
CREATE DATABASE myapp_db CHARACTER SET utf8mb4;
CREATE USER 'myapp_user'@'localhost' IDENTIFIED BY 'password';
GRANT ALL PRIVILEGES ON myapp_db.* TO 'myapp_user'@'localhost';
FLUSH PRIVILEGES;
USE myapp_db;
CREATE TABLE messages (id INT AUTO_INCREMENT PRIMARY KEY, author VARCHAR(50), content TEXT, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP);
INSERT INTO messages (author, content) VALUES ('admin', 'Hello World!');

Flask app.py 關鍵邏輯：

  • DB_CONFIG：host='localhost', user='myapp_user', database='myapp_db'，連本機 MySQL
  • get_conn()：每次請求建立新連線，用完關閉（簡單版）
  • index() GET /：SELECT * FROM messages ORDER BY id DESC，查詢並顯示所有留言
  • add_message() POST /add：INSERT INTO messages 使用 %s 參數化查詢防 SQL Injection，完成後 redirect('/')
重點：永遠用參數化查詢（%s），絕對不要用 f-string 或 + 拼接 SQL 字串，否則會有 SQL Injection 漏洞。

Nginx 反向代理

實作

為什麼不直接讓 Flask 對外？Flask 開發伺服器不適合生產環境 — 穩定性不足、無法處理大量併發連線、不支援 HTTPS。

  • 啟動 Gunicorn：gunicorn --bind 127.0.0.1:5000 app:app
  • Nginx config 核心：location / { proxy_pass http://127.0.0.1:5000; }
  • 建立軟連結：sudo ln -sf /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/
  • 移除預設：sudo rm /etc/nginx/sites-enabled/default
  • 測試語法：sudo nginx -t
  • 重載設定：sudo systemctl reload nginx

練習：502 排查

練習

故意停止 gunicorn（pkill gunicorn），觀察瀏覽器出現 502 Bad Gateway，再重啟驗證恢復。

優秀回答：停止 gunicorn 後出現 502 Bad Gateway，代表 Nginx 正常接請求但 127.0.0.1:5000 無服務。排查 SOP：(1) ps aux | grep gunicorn 確認進程 (2) ss -lntp | grep :5000 確認 port (3) curl -I http://127.0.0.1:5000 測本機 (4) tail /var/log/nginx/error.log 看日誌。重啟後恢復，一次只改一項才能定位問題。

排查 SOP 總結

四步排查順序

觀念
步驟問題指令
1. 服務在不在Process 有沒有跑起來ps aux | grep gunicorn
systemctl status nginx
2. Port 有沒有聽服務是否綁定正確的 portss -lntp | grep :5000
ss -lntp | grep :80
3. 本機請求通不通繞過外部網路測試curl -I http://127.0.0.1:5000
curl -I http://127.0.0.1:80
4. 日誌看什麼錯誤訊息的具體原因tail /var/log/nginx/error.log
journalctl -u mysql
黃金法則：一次只改一項，才能知道哪個設定影響結果。同時改多項會無法判斷是哪個修正生效。

S3 核心概念

為什麼圖片不該放 EC2

觀念

圖片直接存在 EC2 的 EBS 上會造成四大實務問題：

  • 磁碟膨脹：成本不可預期，上傳量成長後 EBS 不夠用
  • 搬遷成本高：主機重建時要手動搬所有檔案，漏搬就斷圖
  • 多台主機難同步：多台 EC2 做負載均衡時，圖片不會自動同步
  • 備份綁定：備份時把程式與靜態檔綁在一起，增加時間和成本

S3 最小名詞

觀念
名詞說明備註
Bucket最上層的儲存容器名稱全域唯一
Object存在 Bucket 裡的檔案本體含 metadata
KeyObject 的完整路徑字串如 uploads/photo.jpg
PrefixKey 的路徑前綴，用來做分類模擬資料夾結構
Storage Class儲存層級（Standard / IA / Glacier）先用 Standard

權限模型

觀念

S3 權限設計原則：先封閉，再按需求最小開放。

  • Block Public Access：預設先擋住所有公開存取
  • Bucket Policy：JSON 格式明確定義誰可以做什麼
重點：即使 Bucket Policy 寫了 Allow，如果 Block Public Access 開著，公開存取仍然會被攔截。先封閉再按需求最小開放。

S3 操作與 Flask 整合

S3 Console 基本操作

實作
  • 建立 Bucket：命名規則 aws12h-u2-basic-<account-id>（全小寫、無底線、全球唯一）
  • 建立 Prefix（模擬資料夾）：uploads/、public/、private/
  • 基本操作：上傳、下載、刪除 Object
  • 權限檢查：確認 Block Public Access 全部開啟

EC2 IAM Role 設定

實作

建立 aws12h-u2-ec2-s3-role，用最小權限 Policy 讓 EC2 存取 S3：

  • IAM → Roles → Create role → Trusted entity = EC2
  • s3:ListBucket — 限定 uploads/* prefix
  • s3:GetObject — 限定 uploads/*
  • s3:PutObject — 限定 uploads/*
  • EC2 → Actions → Security → Modify IAM Role → 綁定此 Role
重點：用 IAM Role 而不要把 Access Key 寫死在程式碼中。Role 透過 IMDS 自動輪替臨時憑證，無金鑰外洩風險。

Flask + S3 程式碼重點

實作

使用 boto3 在 Flask 中操作 S3：

  • 環境變數：AWS_REGION、S3_BUCKET_NAME
  • boto3.client('s3')：建立 S3 客戶端
  • make_s3_url()：產生 S3 Object 的公開 URL
  • upload_fileobj()：上傳檔案到 S3
  • list_objects_v2()：列出 Bucket 中的 Object 清單

改進版重點：

  • Lazy Init + @lru_cache：S3 client 首次使用才初始化，並快取避免重複建立
  • /health endpoint：提供健康檢查 API，回傳服務狀態與 S3 連線狀態
  • Graceful Degradation：S3 不可用時不讓整頁 500，而是顯示替代訊息
  • Timeout 設定：connect_timeout=3, read_timeout=5，避免請求卡死

練習：S3 上傳驗證

練習

透過 Flask 網頁上傳一張圖片，確認 Flask 頁面與 S3 Console 都看得到。

優秀回答：上傳成功後 Flask 頁面看到圖片卡片與 S3 key，S3 Console 也能看到 Object，EC2 本機沒有新檔案。使用 IAM Role 而非 Access Key：(1) Role 透過 IMDS 自動輪替臨時憑證 (2) 無金鑰外洩風險 (3) 停用只需解除 Role。最小權限 Policy 只允許 uploads/* 的 PutObject/GetObject，即使 EC2 被入侵也無法存取其他 prefix 或刪除檔案。

S3 Block Public Access 詳解（額外補充）

四個選項解析

觀念

Block Public Access 分為兩大類四個選項，分別攔截 ACL 和 Bucket Policy 的公開設定：

類別選項攔截範圍
ACL 類BlockPublicAcls阻擋新建的公開 ACL
IgnorePublicAcls忽略所有既有公開 ACL（不管新舊）
Bucket Policy 類BlockPublicPolicy阻擋新的公開 Bucket Policy
RestrictPublicBuckets忽略所有公開 Policy（限制只能被 AWS 服務存取）
補充提醒：隨身碟 FAT32 / exFAT 不支援權限系統，無法真正設定 chmod 400。建議金鑰存放在 NTFS 格式磁碟上。

Bucket Policy 範例

實作

讓 uploads/* 可被公開讀取的 JSON Policy（需先關閉對應的 Block Public Access 選項）：

JSON Policy：
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicReadUploads",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::aws12h-u2-assets/uploads/*"
  }]
}
  • Principal: "*" — 任何人（公開）
  • Action: s3:GetObject — 只能讀取，不能上傳或刪除
  • Resource 限定 uploads/* — 其他 prefix（如 private/）不受影響

Unit 1 思考題

練習

設計「可交接給同事」的最小上線說明，包含 SG 規則理由、三層架構角色、502 排查流程與技術債。

優秀回答：
SG 入站規則：SSH:22 限管理者 IP 防暴力破解，HTTP:80 開放讓用戶瀏覽，MySQL:3306 不外露只允許本機。

三層角色：Nginx 接請求做反向代理；Flask 處理業務邏輯；MySQL 持久化儲存。

502 排查：(1) ps aux 確認 gunicorn (2) ss -lntp 確認 5000 port (3) curl 測 Flask (4) nginx error.log 看錯誤。

技術債：圖片仍存 EBS，下一單元搬 S3 解決磁碟膨脹問題。

Unit 2 思考題

練習

公司決定將現有 EC2 上的 uploads/ 圖片全部遷移到 S3，請規劃遷移方案。

優秀回答：
盤點：du -sh 統計 uploads/ 大小，區分使用中（優先搬）與歸檔（後搬）。

避免失效：採雙寫策略，新上傳走 S3，舊檔分批 aws s3 cp 同步，Flask 同時支援 S3 URL 與本機 fallback。

回滾：搬移前做 EBS Snapshot，保留本機檔案至少 7 天。

權限：Block Public Access 全開，Bucket Policy 只允許 EC2 IAM Role 讀寫 uploads/*。
