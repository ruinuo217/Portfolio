Function Calling 與 RAG

4/1 課程筆記 — Gemini Function Calling、OpenAI API 對照、RAG 向量搜尋、Reranker 精準重排、LangChain Loader

Function Calling 概念

什麼是 Function Calling

觀念

你定義函式，LLM 決定要不要呼叫。整個流程是兩段式：LLM 選擇（回傳函式名稱 + 參數） → 你執行（把結果送回 LLM）。

  • 核心觀念：Gemini 只負責「決定」不負責「執行」——執行權在你手上
  • 完整流程：使用者需求 → Gemini 看可用函式 → 選擇呼叫 → 你執行 → 結果送回 → Gemini 再選擇或回傳文字
重點：LLM 不會直接操作系統，它只是「告訴你」要呼叫哪個函式、傳什麼參數。真正的執行永遠在你的程式碼裡。

定義工具

實作

Gemini 用 Python function 的 docstring 和 type hint 理解函式用途與參數。

def check_regulation(violation_type: str) -> str:
    """查詢違規類型對應的法規
    Args:
        violation_type: 違規類型
    """
    # 實作內容...

model = genai.GenerativeModel(
    "gemini-2.5-flash",
    tools=[check_regulation, send_alert, log_incident]
)
技巧：docstring 寫得越清楚，Gemini 越能正確判斷何時該呼叫這個函式。type hint 讓 Gemini 知道參數型別。

完整迴圈實作

實作

Function Calling 的核心是一個 while loop：持續檢查回應是否為 function_call，直到 LLM 回傳文字為止。

chat = model.start_chat()
response = chat.send_message("使用者輸入")

# func_map 是函式指標字典
func_map = {"check_regulation": check_regulation,
            "send_alert": send_alert,
            "log_incident": log_incident}

while True:
    part = response.candidates[0].content.parts[0]
    if hasattr(part, "function_call") and part.function_call.name:
        fc = part.function_call
        result = func_map[fc.name](**dict(fc.args))  # 執行函式
        response = chat.send_message(
            genai.protos.Content(parts=[
                genai.protos.Part(function_response=
                    genai.protos.FunctionResponse(
                        name=fc.name,
                        response={"result": result}))]))
    else:
        print(part.text)  # 最終文字回覆
        break

Workshop：違規處理助手

練習

整合 Function Calling 完整流程，建立一個工安違規處理助手。

import google.generativeai as genai

def check_regulation(violation_type: str) -> str:
    '''查詢違規類型對應的法規 Args: violation_type: 違規類型如no_helmet'''
    regs = {"no_helmet": "職安法第281條：應使勞工確實使用安全帽。",
            "no_vest": "職安法第21條：應提供反光背心。"}
    return regs.get(violation_type, "查無相關法規")

def send_alert(message: str, severity: str) -> str:
    '''發送告警 Args: message: 告警內容 severity: 嚴重程度high/medium/low'''
    print(f"[{severity.upper()}] {message}")
    return f"告警已發送: {message}"

def log_incident(description: str, location: str) -> str:
    '''記錄事件 Args: description: 事件描述 location: 發生地點'''
    print(f"{location} - {description}")
    return f"已記錄: {location} {description}"

model = genai.GenerativeModel("gemini-2.5-flash",
    tools=[check_regulation, send_alert, log_incident])
chat = model.start_chat()
response = chat.send_message("A區有人沒戴安全帽，幫我處理")

func_map = {"check_regulation": check_regulation,
            "send_alert": send_alert,
            "log_incident": log_incident}
while True:
    part = response.candidates[0].content.parts[0]
    if hasattr(part, "function_call") and part.function_call.name:
        fc = part.function_call
        print(f"呼叫: {fc.name}({dict(fc.args)})")
        result = func_map[fc.name](**dict(fc.args))
        response = chat.send_message(genai.protos.Content(parts=[
            genai.protos.Part(function_response=genai.protos.FunctionResponse(
                name=fc.name, response={"result": result}))]))
    else:
        print(f"{part.text}")
        break

Gemini 會依序呼叫：check_regulation → send_alert → log_incident → 最終文字回覆。跟 LangChain Tool 的關係：Function Calling 是原生 API，LangChain Tool 是框架封裝——學完原生再學框架就知道它在包裝什麼。

OpenAI Chat Completion

實作

OpenAI 的基本對話呼叫方式，使用 messages list 管理對話歷史。

from openai import OpenAI

client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "system", "content": "你是工安專家"},
        {"role": "user", "content": "什麼是PPE？"}
    ]
)
print(response.choices[0].message.content)
  • system：系統設定，定義 AI 角色與行為
  • user：使用者輸入
  • assistant：AI 回覆，多輪對話時需手動帶入歷史

OpenAI Function Calling

實作

OpenAI 的工具定義用 JSON Schema（比 Gemini 囉嗦但更明確）。

# 工具定義
tools = [{"type": "function", "function": {
    "name": "check_regulation",
    "parameters": {
        "type": "object",
        "properties": {
            "violation_type": {
                "type": "string",
                "description": "違規類型"
            }
        },
        "required": ["violation_type"]
    }
}}]

# 取得呼叫
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    tools=tools
)
tool_call = response.choices[0].message.tool_calls[0]

# 送回結果
messages.append({"role": "tool",
    "tool_call_id": tool_call.id,
    "content": result})

FC 差異對照

觀念

Function Calling 在兩個平台的語法差異與共同點。

項目GeminiOpenAI
工具定義Python docstringJSON Schema
定義風格簡單囉嗦但明確
對話歷史chat.send_message 自動帶歷史每次傳完整 messages list
共同點：兩段式流程完全相同——LLM 選擇 → 你執行 → 送回。邏輯一樣，只是語法不同。

Workshop：OpenAI 實作

練習

分三個任務實作 OpenAI API 並與 Gemini 比較。

任務 1 Chat Completion：client.chat.completions.create(model='gpt-4o-mini', messages=[...])。

任務 2 Function Calling：工具定義用 JSON Schema 格式（type / properties / required）。

任務 3 比較：同一問題兩邊跑——Gemini 免費額度大適合圖片分析，OpenAI 文字任務可能更穩定。兩種 API 都會用，根據任務選模型：圖片分析 → Gemini，文字生成 → 看效果和成本，Function Calling → 語法不同但邏輯一樣。

RAG 概念

什麼是 RAG

觀念

Retrieval-Augmented Generation：讓 LLM「先找資料再回答」。200 頁法規不可能全塞進 prompt → 只搜相關段落塞進去。

  • 流程：使用者問題 → Embedding 搜尋法規庫 → 找到相關段落 → 塞進 prompt → LLM 回答
  • 能引用真實文件：回答有依據，可附上來源
  • 不需重新訓練：不用 fine-tune 模型
  • 資料更新快：改知識庫即可，不用重訓
  • 減少幻覺：限定 LLM 只用提供的資料回答

RAG vs Prompt Engineering

觀念

兩種讓 LLM 使用外部知識的方式比較。

項目Prompt EngineeringRAG
資料來源資料直接放進 prompt經過 embedding 的外部資料
適用場景少量短期資料大量長期需語意搜尋
技術重點提示語設計向量資料管理
成本低稍高
選擇策略：靜態知識 → RAG，動態資訊 → Function Call，兩者都有 → 混合架構。

RAG 實作

Embedding 建索引

實作

文件切段 → 每段做 embedding → 存入向量資料庫（一次性建置）。

# 將文件內容做 embedding
client.models.embed_content(
    model="gemini-embedding-001",
    contents=["段落1的內容", "段落2的內容", ...]
)
重點：Embedding 是將文字轉成高維向量，語意相近的文字在向量空間中距離較近。

向量搜尋

實作

用 cosine similarity 計算查詢與所有段落的相似度，分數越高越相關。

from sklearn.metrics.pairwise import cosine_similarity

sims = cosine_similarity([query_emb], chunk_embeddings)[0]
# 取 top 3 最相關段落
top_idx = np.argsort(sims)[::-1][:3]

組合 RAG Pipeline

實作

搜尋 → 組 context → 塞進 prompt → Gemini 回答 → 附上來源。

# 組合 prompt
context = "\n".join([chunks[i]["content"] for i in top_idx])
prompt = f"""根據以下法規回答問題，不要編造：
{context}

問題：{query}"""

model = genai.GenerativeModel("gemini-2.5-flash")
response = model.generate_content(prompt)
print(response.text)

Workshop：RAG 法規問答

練習

完整實作 RAG Pipeline：從建索引到搜尋到回答。

import google.generativeai as genai
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity

client = genai.Client(api_key="your-key")
chunks = [
    {"id": "reg_001", "title": "安全帽規定", "content": "依據職安法第281條..."},
    {"id": "reg_002", "title": "反光背心規定", "content": "依據職安法第21條..."},
    # ...共8段
]
# 建索引
contents = [c["content"] for c in chunks]
result = client.models.embed_content(model="gemini-embedding-001", contents=contents)
chunk_embs = np.array(result["embedding"])

# 搜尋
query = "安全帽有什麼規定？"
q_result = client.models.embed_content(model="gemini-embedding-001", contents=query)
q_emb = np.array(q_result["embedding"])
sims = cosine_similarity([q_emb], chunk_embs)[0]
top_idx = np.argsort(sims)[::-1][:3]

# 組 prompt
context = "\n".join([chunks[i]["content"] for i in top_idx])
prompt = f"根據以下法規回答問題,只用提供的內容,不要編造:\n{context}\n\n問題：{query}"
model = genai.GenerativeModel("gemini-2.5-flash")
response = model.generate_content(prompt)
print(response.text)

RAG 的關鍵：「只用提供的法規回答，不要編造」這句 prompt 減少幻覺。top_k=3 就夠，不用全部塞。cosine_similarity 越高 = 越相關。

RAG 進階 — Reranker

Embedding 搜尋的問題

觀念

「客服」與「配送」的向量可能相近 → 搜尋「如何聯絡客服」可能排到「國際配送」前面。

  • Embedding：快速粗篩，捕捉語意空間中的距離
  • Reranker：精準重排，用 LLM 理解語意相關性
類比：Embedding 像是用關鍵字搜尋，Reranker 像是請專家逐一審閱搜尋結果後重新排序。

Reranker 用 Gemini 重新打分

實作

用 Gemini 直接評估「問題和 FAQ 內容的相關度」0-100 分。

  • 兩階段：Embedding 快速篩選（粗排） → Reranker 精準重排（精排）
  • 效果：修正後「如何聯絡客服」→ 正確排到第 1 名（100 分）

Workshop：Reranker 改善

練習

理解 Reranker 的運作原理與實務平衡。

Embedding 的限制：它捕捉的是「語意空間中的距離」，有時候不同主題但都跟「服務」相關的 FAQ 會被誤判為相似。

Reranker 解法：讓 LLM 逐一評估「這個 FAQ 是否直接回答了使用者的問題」——這是語意理解而非向量距離。

代價：Reranker 需要對每個候選都呼叫一次 LLM，速度較慢（N 個候選 = N 次 API 呼叫）。

實務平衡：Embedding 粗篩取 top 5 → Reranker 精排取 top 2 → 只送最相關的給 LLM 回答。

LangChain Document Loader（補充）

各種格式讀取

觀念

LangChain 提供多種 Document Loader，支援不同格式的文件讀取。

  • PyPDFLoader：讀取 PDF 文件
  • CSVLoader：讀取 CSV / Google Sheet
  • WebBaseLoader：讀取網頁內容
  • TextSplitter：切段（chunk_size=500, chunk_overlap=50 避免切斷語意）
完整 LangChain RAG Pipeline：讀文件 → 切段 → embedding → 向量 DB → 搜尋 → LLM 回答。LangChain 把這些步驟都封裝好了，但底層邏輯跟我們今天手刻的一模一樣。

課程回顧

今日學習架構

觀念

今天從原生 API 到框架封裝，完整走過 LLM 整合的核心技術。

  • Function Calling：讓 LLM 決定呼叫函式，兩段式流程（LLM 選擇 → 你執行）
  • OpenAI API：第二種 API 選擇，語法不同但邏輯相同
  • RAG：先找再答，減少幻覺，讓 LLM 能引用真實文件
  • Reranker：精準重排，解決 Embedding 粗篩的誤判問題
  • LangChain：框架封裝，把手刻流程包成現成元件
核心觀念：LLM 不直接操作系統，透過 Function Calling + RAG 間接使用外部資源。Function Calling 處理「動態操作」，RAG 處理「靜態知識」，兩者結合就是完整的 LLM 應用架構。
