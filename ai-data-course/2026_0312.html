FastAPI 基礎與事件驅動 API

3/12 課程筆記

什麼是事件 API

函式呼叫 vs 事件思維

觀念

函式：參數不對改了重跑，可控可重來。API：request 送出事情就已發生，不能撤回。

比較項目函式呼叫事件 API
範圍你的程式裡跨系統跨網路
可重來可重跑送出就發生
錯誤處理try-except需記錄 + 回溯
記錄不記錄每次都該記錄

把 API 看成事件

觀念

每個 API 呼叫都該問三個問題：什麼事件（endpoint 名稱）、內容是什麼（request body）、系統怎麼回應（response）。

  • POST /login = 使用者嘗試登入
  • POST /upload-image = 上傳圖片
  • GET /recommendations = 請求推薦
重點：連 GET 都是事件 —— 「使用者在這個時間點請求了這個資料」值得記錄。

為什麼 AI 系統需要事件思維

觀念

如果只是「呼叫 → 回傳」，事後分析全做不到；每次 request 當事件記錄 = 完整 audit trail。

  • 模型會錯 → 需知道當時輸入
  • 模型會改版 → 需舊資料重跑比較
  • 行為會漂移 → 需長期追蹤

練習：事件分析

練習

題目：POST /transactions/evaluate、GET /users/123/history、POST /alerts/dismiss 各是什麼事件？不記錄的風險？

回答：
  • POST /transactions/evaluate = 交易風險評估請求事件，內容含 user_id / amount / timestamp，不記錄 → 無法回溯模型當時輸入，出錯時無法復盤。
  • GET /users/123/history = 查詢使用者歷史事件，內容含 query params（時間範圍），不記錄 → 不知道誰在什麼時候查了什麼（資安稽核空白）。
  • POST /alerts/dismiss = 關閉警報事件，內容含 alert_id / reason，不記錄 → 無法追蹤誰關了警報為什麼關（責任歸屬不明）。
  • 連 GET 都該記錄的原因：追蹤存取模式、偵測異常查詢行為、滿足合規要求。

FastAPI 快速上手

安裝與最小 API

實作

FastAPI = 框架（定義怎麼處理 request），Uvicorn = ASGI server（負責收發）。

  • 安裝：pip install fastapi uvicorn
  • 最小 3 行：app = FastAPI()、@app.get("/")、return {"message": "alive"}
  • 啟動：uvicorn main:app --reload

Swagger UI 自動文件

實作

http://127.0.0.1:8000/docs 自動產生互動式 API 文件。

  • 不用 curl / Postman，直接瀏覽器測
  • 後續每個 workshop 都可用 Swagger 驗證

POST endpoint 接收資料

實作

@app.post("/echo") def echo(data: dict = Body()): return {"received": data} — POST /echo 接收 JSON body 原封不動回傳 = 最簡單的「收事件」。

  • JSON 解析：body → dict（自動）
  • JSON 序列化：dict → JSON response（自動）
  • HTTP status code：200 / 422（自動）

練習：FastAPI 啟動

練習

題目：啟動服務、Swagger 測 GET /、POST /echo 送 {"event_type": "test", "user_id": 1}

回答：
  • 啟動：uvicorn main:app --reload，瀏覽器開 http://127.0.0.1:8000 看到 {"message": "alive"}。
  • Swagger：http://127.0.0.1:8000/docs，點 GET / → Try it out → Execute，回傳 {"message": "alive"}。
  • POST /echo：點 POST /echo → Try it out → 填入 {"event_type": "test", "user_id": 1} → Execute，回傳 {"received": {"event_type": "test", "user_id": 1}}。
  • --reload 是開發模式，改 code 自動重啟。FastAPI 自動做 JSON 解析（body → dict）和序列化（return dict → JSON）。422 是 FastAPI 的驗證失敗代碼，格式不對會自動回 422。

HTTP Request 三層結構

Path — 資源的身份證

觀念

@app.get("/drinks/{drink_id}") — 判斷標準：拿掉這個參數 API 還有意義嗎？沒有 → 放 Path。

  • 多層 Path：/menu/{item_id}/reviews/{review_id}

Query — 篩選與選項

觀念

@app.get("/drinks") def list_drinks(category: str = None, max_price: int = None) — 判斷：拿掉還能用嗎？可以 → 放 Query。

  • FastAPI 規則：有預設值 + 不在 URL path → 自動變 Query

Body — 寄送資料包裹

觀念

@app.post("/orders") def create_order(order: dict = Body(...)) — 不是在「看」是在「送」資料。

  • 三層一起用：@app.post("/drinks/{drink_id}/customize") path = 哪杯, query = debug 選項, body = 客製化內容
層級用途判斷依據
Path身份證拿掉不知講誰
Query篩選選項拿掉還能用
Body資料包裹要送資料給 server

練習：學生餐廳 API

練習

題目：GET /menu, GET /menu/{item_id}, GET /menu/{item_id}/reviews/{review_id}, GET /menu?category=飯類&max_price=75, POST /orders

回答： 
from fastapi import FastAPI, Body
app = FastAPI()
MENU = {"fried-rice": {"name": "炒飯", "price": 80, "category": "飯類"},
        "chicken-cutlet": {"name": "雞排", "price": 60, "category": "炸物"}}

@app.get("/menu")
def list_menu(category: str = None, max_price: int = None):
    results = {k: v for k, v in MENU.items()
               if (not category or v["category"] == category)
               and (not max_price or v["price"] <= max_price)}
    return results

@app.get("/menu/{item_id}")
def get_item(item_id: str, lang: str = "zh"):
    item = MENU[item_id]
    if lang == "en":
        return {"item_id": item_id, "name": "Fried Rice", "price": item["price"]}
    return {"item_id": item_id, **item}

@app.get("/menu/{item_id}/reviews/{review_id}")
def get_review(item_id: str, review_id: int):
    return {"item_id": item_id, "review_id": review_id,
            "comment": "好好吃！", "rating": 5}

@app.post("/orders")
def create_order(order: dict = Body(...)):
    item = MENU[order["item_id"]]
    return {"message": "訂餐成功", "order_id": 1,
            "total": item["price"] * order["quantity"]}
  • Path 放 item_id 因為拿掉不知道在問哪道菜。
  • Query 放 category / max_price 因為拿掉菜單還是菜單只是沒篩選。
  • Body 放訂單內容因為是完整資料包裹。

非同步 — 不要卡住 API

Event Loop 與阻塞

觀念

FastAPI 跑在 event loop 上 = 單線道，所有 request 排隊。time.sleep(5) = 堵死 5 秒誰都不能動。await asyncio.sleep(5) = 暫時讓出，其他 request 可以先走。

比較項目time.sleepawait asyncio.sleep
Event Loop堵住繼續跑
其他 request等 5 秒正常回應
整體影響整個 API 卡死只有這個 request 等

async def 不等於非阻塞

觀念

async def 只是告訴 Python「可以被 await」，如果裡面用 time.sleep（同步 DB / 同步 HTTP request）一樣堵住。

  • 規則：如果在 async function 裡呼叫的東西不是 await，它就是阻塞的
陷阱：async def sync_api() 明明有 async 為什麼還阻塞？因為裡面的 time.sleep 不是 await 操作。

練習：阻塞測試

練習

題目：兩個 terminal 測 /sync + /ping 和 /async + /ping

回答：
  • /sync + /ping → 打 /sync 時 /ping 要等 5 秒才回（被堵住）。
  • /async + /ping → 打 /async 時 /ping 馬上回 pong（event loop 沒被堵）。
  • 修正方法：把 time.sleep(5) 改成 await asyncio.sleep(5)。
  • 實務應用：呼叫外部 API 用 httpx.AsyncClient 而非 requests（requests 是同步會阻塞）；資料庫操作用 asyncpg 而非 psycopg2（同步驅動會阻塞）。
  • 判斷法則：在 async def 裡，所有 IO 操作都要用 await 版本。

Pydantic Schema — 合約

Schema = 合約

觀念

dict 什麼都吃 → usre_id: "abc", amout: -100 照收 → 下游 AI 炸掉。Schema 宣告：user_id（int）、amount（float）、currency（str）三個都不能少。

  • 定義：class TransactionEvent(BaseModel): user_id: int, amount: float, currency: str
  • Fail-Fast：驗證在 handler 前執行，不合法資料永遠碰不到商業邏輯
  • 合法 → 200，缺欄位 / 型別錯 → 422

Swagger 自動文件

觀念

定義 Schema → Swagger 自動顯示欄位名稱 / 型別 / required。不用另外寫 API 文件，Schema 就是文件。

  • handler 裡可放心用資料，不需 if event.user_id is None 防禦

練習：Schema 定義

練習

題目：定義 TransactionEvent, POST /transactions, Swagger 測試合法 / 缺欄位 / 型別錯

回答： 
from pydantic import BaseModel

class TransactionEvent(BaseModel):
    user_id: int
    amount: float
    currency: str

@app.post("/transactions")
def receive(event: TransactionEvent):
    return {"status": "accepted", "event": event.model_dump()}
  • 合法 {"user_id": 123, "amount": 500, "currency": "TWD"} → 200。
  • 缺 currency → 422 {"detail": [{"loc": ["body", "currency"], "msg": "Field required"}]}。
  • user_id = "abc" → 422 型別錯。
  • Schema 的價值：handler 執行時 event.user_id 保證是 int，不需防禦性程式碼。Swagger 自動產生文件不用另外維護。

Validation — 擋語意

Schema vs Validation

觀念

Schema 問「格式對嗎」（型別 / 是否存在），Validation 問「合理嗎」（數值範圍 / 長度）。amount = -9999 格式正確但語意不合理。

  • Field(gt=0) = 必須 > 0
  • Field(min_length=3, max_length=3) = 剛好 3 字元
  • Optional[str] = None = 可不給

Field 驗證規則

實作

Validation 也在 handler 前執行，與 Schema 一起構成兩層防線。

規則說明範例
gt=0最小值金額 > 0
lt=1000000最大值單筆上限
min_length=3 / max_length=3字串長度貨幣代碼
pattern=r"^[A-Z]{3}$"正則三碼大寫

練習：Validation

練習

題目：加上 user_id > 0, amount > 0, currency 三字元，測試 4 種情境

回答： 
from pydantic import BaseModel, Field
from typing import Optional

class TransactionEvent(BaseModel):
    user_id: int = Field(gt=0)
    amount: float = Field(gt=0)
    currency: str = Field(min_length=3, max_length=3)
    note: Optional[str] = None
  • 合法 {"user_id": 123, "amount": 500, "currency": "TWD"} → 200。
  • user_id = -1 → 422 "Input should be greater than 0"。
  • amount = -9999 → 422。
  • currency = "TD"（2 字元）→ 422 "String should have at least 3 characters"。
  • Schema 擋結構（型別 / 存在），Validation 擋語意（範圍 / 長度）—— 兩層防線都在 handler 前執行，商業邏輯永遠收到乾淨資料。

Workshop — 設計事件型 API

Workshop 需求

實作

AI 行為分析公司：統一事件入口 API。POST /events/{event_type}?debug=false

  • Path：event_type（click / transaction / login）
  • Query：debug（選填）
  • Body：EventPayload（user_id: int > 0, timestamp: str, payload: dict）
  • debug=true 回傳 payload，false 不回傳

練習：完整實作

練習
回答： 
from fastapi import FastAPI, Body, HTTPException
from pydantic import BaseModel, Field
from typing import Any, Dict

app = FastAPI()
SUPPORTED = ["click", "transaction", "login"]

class EventPayload(BaseModel):
    user_id: int = Field(gt=0)
    timestamp: str = Field(min_length=1)
    payload: Dict[str, Any]

@app.post("/events/{event_type}")
def receive_event(event_type: str, debug: bool = False,
                  event: EventPayload = Body()):
    if event_type not in SUPPORTED:
        raise HTTPException(status_code=400,
                            detail=f"Unsupported: {event_type}")
    response = {"status": "accepted", "event_type": event_type,
                "user_id": event.user_id, "timestamp": event.timestamp}
    if debug:
        response["payload"] = event.payload
    return response
  • event_type 放 Path 因為它定義這是什麼事件，拿掉就不知道在處理什麼。
  • debug 放 Query 因為是選項，拿掉 API 還能用。
  • payload 放 Body 因為是完整資料包裹。
  • 三層各司其職：Path = 身份，Query = 選項，Body = 資料。
