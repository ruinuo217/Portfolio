FastAPI 進階與 AI 系統整合

3/13 課程筆記

專案結構 — 模組化開發

四層架構

觀念

main.py(建立 app 掛 router) → routers/(定義 endpoint) → schemas/(Pydantic model) → services/(商業邏輯)。

層級職責不做的事
main.py建立 app、掛載 router不寫 endpoint
router定義 endpoint、接參數不寫邏輯
schema定義資料結構不寫邏輯
service處理商業邏輯不碰 HTTP

目錄結構：app/main.py + routers/events.py + schemas/event.py + services/event_service.py

為什麼要拆

觀念

拆開不是為了好看，是為了可讀、可測、可協作。

情境單檔模組化
改 Schemadiff 顯示 main.pydiff 顯示 schemas/event.py
兩人同改conflict各改各的
測 serviceimport 整個 app只 import service function
找 bug在 800 行翻看檔名就知道在哪

練習：重構為模組化

練習

題目：將單檔 FastAPI 應用重構為四層模組化結構。

回答：四個檔案各司其職：

schemas/event.py — 只定義結構：

from pydantic import BaseModel, Field
from typing import Any, Dict

class EventPayload(BaseModel):
    user_id: int = Field(gt=0)
    timestamp: str = Field(min_length=1)
    payload: Dict[str, Any]

services/event_service.py — 只處理邏輯：

def handle_event(event_type, event, debug):
    response = {"status": "accepted", "event_type": event_type,
                "user_id": event.user_id, "timestamp": event.timestamp}
    if debug:
        response["payload"] = event.payload
    return response

routers/events.py — 只定義 endpoint：

from fastapi import APIRouter, Body
from schemas.event import EventPayload
from services.event_service import handle_event

router = APIRouter(prefix="/events")

@router.post("/{event_type}")
def receive_event(event_type: str, debug: bool = False,
                  event: EventPayload = Body()):
    return handle_event(event_type, event, debug)

main.py — 只負責組裝：

from fastapi import FastAPI
from routers.events import router

app = FastAPI(title="Event API")
app.include_router(router)

驗收：API 行為完全不變 — 同樣 request 同樣 response。重構的目的是改善內部結構，不是改變外部行為。

Error Handling — 系統邊界的錯誤處理

錯誤的層級

觀念

Request → Schema(422) → Validation(422) → Service(400/404) → 未預期(500)。FastAPI 自動處理前兩層(422)，Service 層需自己處理。

  • 「世界的錯」：400(事件不合理) / 404(對象不存在) / 422(格式不對)
  • 「系統的錯」：500(系統 bug)
  • 重要：500 不是正常錯誤回應，代表系統出問題

自訂 Exception

實作

class EventRejected(Exception): reason, event_type。raise 不是 return — 錯誤是中斷不是回傳。

  • 註冊 handler：@app.exception_handler(EventRejected) 回傳統一 JSON
  • 覆寫 422：@app.exception_handler(RequestValidationError) 統一格式

統一錯誤格式

觀念

所有錯誤同一結構：{"status":"rejected", "event_type":"...", "reason":"...", "code":"EVENT_TYPE_NOT_SUPPORTED"}

  • 下游系統只看 status 就知道結果

練習：Error Handling

練習

題目：實作三種錯誤的統一處理格式。

回答： 
class EventRejected(Exception):
    def __init__(self, reason: str, event_type: str = "unknown"):
        self.reason = reason
        self.event_type = event_type

class EventNotFound(Exception):
    def __init__(self, event_id: str):
        self.event_id = event_id

@app.exception_handler(EventRejected)
async def handle_rejected(request, exc):
    return JSONResponse(status_code=400,
        content={"status": "rejected", "event_type": exc.event_type,
                 "reason": exc.reason})

@app.exception_handler(EventNotFound)
async def handle_not_found(request, exc):
    return JSONResponse(status_code=404,
        content={"status": "rejected",
                 "reason": f"Event {exc.event_id} not found"})

@app.exception_handler(RequestValidationError)
async def handle_validation(request, exc):
    return JSONResponse(status_code=422,
        content={"status": "rejected",
                 "reason": "Event does not match schema",
                 "details": exc.errors()})

三種錯誤統一格式：Schema 錯(422) → status:rejected + details。邏輯錯(400) → status:rejected + reason。找不到(404) → status:rejected + reason。正常(200) → status:accepted。下游只看 status 欄位即可判斷。

AI 系統的防呆層 (Guardrail)

模型負責算，系統負責決定

觀念

Guardrail 在模型後面，審查輸出不是過濾輸入。模型回傳的 confidence 是參考數字不是決策。

  • 門檻分流：≥0.9 自動處理 / 0.6~0.9 人工審核 / <0.6 不做判斷
  • 門檻值由系統決定：根據風險承受度，同模型用在醫療 vs 推薦門檻不同

Pydantic 定義決策結構

實作

兩個 model 分開：Prediction 是模型的事，Decision 是系統的事。

  • Prediction：label: str, confidence: float (0~1)
  • RiskLevel enum：HIGH / MEDIUM / LOW
  • Action enum：AUTO_BLOCK / REQUIRE_REVIEW / DEFER
  • DecisionResponse：prediction / confidence / risk_level / action / reason

evaluate() 純函式

實作

THRESHOLDS = {"auto": 0.9, "review": 0.6} 集中管理。純函式不碰 HTTP 不碰 DB。

  • 決策也是事件：DecisionEvent 記錄每次 AI 判斷供追蹤回溯

練習：Guardrail

練習

題目：實作 AI 防呆層的門檻分流邏輯。

回答： 
from enum import Enum

class RiskLevel(str, Enum):
    HIGH = "high"
    MEDIUM = "medium"
    LOW = "low"

class Action(str, Enum):
    AUTO_BLOCK = "auto_block"
    REQUIRE_REVIEW = "require_review"
    DEFER = "defer"

THRESHOLDS = {"auto": 0.9, "review": 0.6}

def evaluate(prediction):
    if prediction.confidence >= THRESHOLDS["auto"]:
        return DecisionResponse(
            risk_level=RiskLevel.HIGH,
            action=Action.AUTO_BLOCK,
            reason="信心值足夠，系統自動處理", ...)
    elif prediction.confidence >= THRESHOLDS["review"]:
        return DecisionResponse(
            risk_level=RiskLevel.MEDIUM,
            action=Action.REQUIRE_REVIEW,
            reason="信心值不足以自動處理，轉人工審核", ...)
    else:
        return DecisionResponse(
            risk_level=RiskLevel.LOW,
            action=Action.DEFER,
            reason="信心值過低，系統不做判斷", ...)
  • confidence=0.95 → auto_block（高風險自動處理）
  • confidence=0.68 → require_review（轉人工）
  • confidence=0.42 → defer（不做判斷）
  • 門檻集中在 THRESHOLDS，調整時只改一處
  • evaluate() 是純函式可單獨 unit test
  • Guardrail 回 200 不是錯誤 — 「我不確定」是合法回應

事件資料流 — FastAPI x Pandas

Event vs DataFrame

觀念

Event = 一次性事實（發生過就存在）。DataFrame = 觀察角度（同事件不同整理方式）。

  • 流程：User 查詢 → FastAPI → Service → Event Store/CSV → Pandas 整理 → Response
  • Pandas 不直接面對使用者，在 service 裡安靜整理好交給上層回傳

為什麼先存再整理

觀念

Event 是永久的，DataFrame 是暫時的。

  • 系統會重啟：記憶體 DataFrame 消失
  • 規則會改：需用新規則重跑歷史
  • 出事需 audit：回放每筆事件

Pandas 整理歷史

實作

get_user_summary(): pd.read_csv → df[df["user_id"]==user_id] → len/sum/mean。

  • 邊界處理：user_df.empty → 回傳安全結構（不是 error）
  • get_event_breakdown()：groupby("event_type").agg(count/sum/mean)

練習：Pandas 整合

練習

題目：實作使用者歷史事件摘要 API。

回答： 
import pandas as pd

def get_user_summary(user_id: int) -> dict:
    df = pd.read_csv("events.csv")
    user_df = df[df["user_id"] == user_id]
    if user_df.empty:
        return {"user_id": user_id, "total_events": 0,
                "total_amount": 0, "avg_confidence": None,
                "note": "no historical events"}
    return {"user_id": user_id,
            "total_events": len(user_df),
            "total_amount": float(user_df["amount"].sum()),
            "avg_confidence": round(float(
                user_df["confidence"].mean()), 2)}

@app.get("/users/{user_id}/summary")
def user_summary(user_id: int):
    return get_user_summary(user_id)
  • User 123（有資料）：{"total_events":3, "total_amount":2500.0, "avg_confidence":0.79}
  • User 999（無資料）：{"total_events":0, "note":"no historical events"}
  • 查不到不是 error 是「系統還沒紀錄」
  • Pandas 操作全在 service，endpoint 只呼叫 + 回傳
  • groupby + agg 是最常用組合：一行搞定按 X 分組算 Y 的總和/平均/次數

LLM 管道 — FastAPI x AI Pipeline

AI 是使用者不是主人

觀念

Web App / Mobile / Bot / AI 都是 client，透過 API 使用系統。AI 不該有特權，走同一套 Schema / Validation / 防呆。

方式直接碰 DB透過 API
驗證跳過驗證完整驗證
控管無法控管Service 控制
紀錄難 audit有紀錄
安全AI 出錯直接影響資料被防呆攔住

三種 Pipeline

觀念

每個 endpoint 背後不同 pipeline，對 AI 來說都只是「呼叫 API」。

類型用途工具輸出
推論 Inference判斷風險AI 模型risk_level + action
摘要 Summary回顧歷史Pandas + LLM統計 + 敘述
任務觸發 Action請系統做事Servicetask_id + status
  • 推論：prepare_features → ai_predict → guardrail
  • 摘要：prepare_features → 可選 LLM 語言化（?llm=true）
  • 任務：回傳 pending 不是 done，系統排程後才執行

三條 Endpoint

實作

每個 endpoint 背後不同 pipeline，對 AI 來說都只是「呼叫 API」。

  • POST /risk/evaluate — 推論
  • GET /users/{id}/summary?llm=false — 摘要
  • POST /tasks/review — 任務觸發

練習：AI Pipeline

練習

題目：說明三種 Pipeline 的實作重點。

回答：三種 Pipeline 實作重點：
  • 推論型：prepare_features 從 CSV 整理特徵 → fake_ai_predict 模擬模型 → evaluate 防呆（confidence ≥0.8 → auto_flag, ≥0.6 → require_review, <0.6 → defer）。AI 拿到整理好的特徵不是 raw event，系統控制 AI 看到什麼。
  • 摘要型：?llm=true 才觸發 LLM 語言化，結構化資料永遠回傳，自然語言是加值。LLM 不碰 DB / 不碰 Pandas，只是最後一層翻譯。
  • 任務型：回傳 pending 不是 done，系統排程後才執行。AI 只能請求不能執行 — 跟人類使用者的權限一樣。
  • 共通原則：所有 Pipeline 走同一套 Schema + Validation + Error Handling，AI 不該有特權通道。

綜合 Workshop — AI 交易風險評估系統

系統需求

實作

5 個階段串起所有學過的概念：

  • (1) 專案結構：模組化 4 層
  • (2) 事件入口：POST /transactions/{tx_type}/evaluate, Schema + Validation
  • (3) 歷史分析：Pandas 從 CSV 讀取 user 歷史
  • (4) 風險評分：rule-based: base = amount/10000, 歷史 risk > 0.5 加 0.2, 交易 < 3 筆加 0.1
  • (5) 決策防呆：risk < 0.3 → auto_approve, 0.3~0.7 → require_review, > 0.7 → auto_block

預期輸出範例

實作

不同情境的 API 回應：

  • 低風險（user456, amount 200）：risk_score = 0.12, action = auto_approve
  • 高風險（user123, amount 50000）：risk_score = 0.95, action = auto_block
  • 不支援 type：status = rejected, 400

練習：完整系統

練習

題目：實作 AI 交易風險評估系統。

回答：開發順序建議：先結構 → 先 hardcode → 加歷史 → 加評分 → 加防呆，每階段 Swagger 測試確認。核心程式碼： 
# services/risk_service.py
def calculate_risk(amount, history):
    score = min(amount / 10000, 0.5)
    if history.get("avg_risk", 0) > 0.5:
        score += 0.2
    if history.get("tx_count", 0) < 3:
        score += 0.1
    return min(score, 1.0)

def decide(risk_score):
    if risk_score > 0.7:
        return "auto_block", "高風險自動擋下"
    if risk_score > 0.3:
        return "require_review", "轉人工審核"
    return "auto_approve", "低風險自動通過"

這串起了所有學過的概念：Schema 擋格式 → Validation 擋語意 → Error Handling 統一錯誤 → Pandas 查歷史 → 風險評分 → 防呆決策 → 事件記錄。

補充 — Flask vs FastAPI 與現代 API 框架

Flask vs FastAPI

觀念

兩者各有適用場景，選擇取決於專案需求。

面向FlaskFastAPI
定位成熟穩定現代高效
非同步同步為主原生 async
文件手動寫文件自動 Swagger
驗證手動驗證Pydantic 自動驗證
資源學習資源豐富效能更好
適合傳統 Web 應用AI/ML 系統 + 微服務
  • Flask 優點：生態系龐大（Flask-Login / Flask-SQLAlchemy）、部署方案成熟、社群大
  • FastAPI 優點：型別提示 = 文件 = 驗證、async 原生、效能接近 Node.js

現代 API 框架概覽

觀念

不同語言生態的主流 API 框架比較：

框架語言適合場景特點
FlaskPython傳統 Web成熟穩定
FastAPIPythonAI + 微服務高效能 + 自動文件
Django RESTPython企業級全功能但重
Express.jsNode.js全端 JavaScript靈活輕量
GinGo高併發極高效能
Spring BootJava企業級重但完整
  • 選擇建議：AI/ML 系統 → FastAPI, 傳統 Web → Flask/Django, 高併發 → Gin/FastAPI, 企業 Java 生態 → Spring Boot

練習：框架選擇

練習

題目：如何選擇適合的 API 框架？

回答：選擇 API 框架的決策邏輯：
  • (1) AI/ML 系統選 FastAPI — Pydantic 自動驗證 AI 輸入輸出，async 處理模型推論不阻塞，Swagger 讓資料科學家自助測試
  • (2) 傳統 Web 應用選 Flask — 生態成熟（Flask-Login / SQLAlchemy / WTForms），模板渲染方便，學習資源最多
  • (3) 需要 GraphQL 選 FastAPI + Strawberry 或 Node.js + Apollo
  • (4) 超高併發選 Go Gin 或 Rust Actix
  • Flask → FastAPI 遷移成本低：路由語法類似（@app.route → @app.get），主要改動是加 Pydantic model 和改 async
