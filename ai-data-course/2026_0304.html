AWS 雲端平台（CloudFront + ALB）

3/4 課程筆記

為什麼需要 CDN

S3 直連的四個問題

觀念

當 S3 作為唯一的靜態資源來源時，會遇到以下四個瓶頸。

  • 跨區延遲高：S3 在東京，歐洲用戶繞半個地球，延遲 200-500ms
  • 無快取機制：同一張圖片請求 100 次就回源 100 次，流量成本持續累積
  • Origin 壓力大：尖峰時所有請求直衝 S3
  • 無路徑控制：所有檔案同一套規則，無法分別處理靜態 vs 動態

CloudFront 解決方案

觀念

AWS CloudFront 是全球 CDN 服務，透過邊緣節點解決上述問題。

  • 全球 400+ 邊緣節點：就近取資源，大幅降低延遲
  • 快取機制：首次回源後 Edge Cache 返回，後續請求不再回源
  • 降低 Origin 壓力：大量請求由邊緣節點攔截
  • 路徑分流（Multiple Behaviors）：不同路徑可套用不同規則

CloudFront 核心名詞

五大概念

觀念

建立 CloudFront 前必須掌握的五個核心名詞。

名詞說明
Distribution分發配置，自動分配 d123abc.cloudfront.net
Origin內容來源，本單元使用 S3
Behavior請求行為規則，按路徑前綴處理
Cache Key快取鍵，URL path + query string
Edge Location全球邊緣節點

Origin 選擇比較

觀念

CloudFront 可搭配多種 Origin 類型，依需求選擇。

Origin 類型適用場景特性
S3 Bucket靜態資源穩定、自動擴展
EC2 / ELB動態 API需自行管理
ALB多 EC2 負載平衡高可用
Custom HTTP外部伺服器彈性最大
Lambda Function URLServerless免管理主機
Tip: 靜態用 S3，動態用 ALB + EC2，混合用 Multiple Behaviors 分流。

OAC 與快取失效

OAC (Origin Access Control)

觀念

讓 S3 只接受 CloudFront 請求，拒絕直接訪問。

  • 工作原理 4 步：
    1. CloudFront 建立 OAC identity
    2. S3 Bucket Policy 設定只允許 OAC
    3. CloudFront 帶簽名請求
    4. S3 驗證通過才回應
  • 不設 OAC 的風險：繞過 CDN 快取、盜連消耗流量
  • OAC vs 舊版 OAI：OAC 是新版推薦，支援 SSE-KMS 加密

快取失效 (Invalidation)

觀念

情境：更新 S3 的 hero.jpg 但用戶仍看舊版（Edge Cache 未過期）。每月前 1000 次免費。

方式做法成本/延遲適用情境
Invalidation保持原檔名，手動清除快取成本高、有延遲緊急修正
版本化檔名hero-v2.jpg無成本、即時生效正常發布

CloudFront 建立實作

Step A: 建立 Distribution

實作

建立 CloudFront Distribution 並設定 OAC。

  • Origin domain：選 S3 bucket
  • Origin access：選 OAC（Create new OAC: aws12h-u3-s3-oac, Sign requests）
  • Default Cache Behavior：Redirect HTTP to HTTPS, GET/HEAD only, CachingOptimized policy
  • 等待部署：3-10 分鐘
  • 記下 domain：d123abc.cloudfront.net

Step B: 更新 S3 Bucket Policy

實作

整合兩個 Statement：EC2 Role 的 PutObject/GetObject（Unit2 既有）+ CloudFront 的 GetObject（新增，用 Condition 限定 SourceArn）。

Tip: 重點：是整合不是覆蓋！

Step C: 確認 Block Public Access

實作

4 個選項全部 On。

  • 測試：CloudFront URL = 200 OK
  • 測試：S3 直連 URL = 403 Forbidden

Step D: Flask 改用 CloudFront URL

實作

在 Flask 應用中整合 CloudFront domain。

  • 新增環境變數：CLOUDFRONT_DOMAIN
  • 新函數 make_cdn_url()：優先使用 CloudFront domain，fallback 到 make_s3_url()

Step E: Cache Invalidation 實作

實作

CloudFront Console → Distribution → Invalidations → Create：路徑填 /uploads/* 或 /*

練習：CloudFront 驗證

練習

測試 CloudFront URL 可顯示圖片（200），S3 直連被拒（403）。

優秀回答：CloudFront URL 回傳 200 OK，Response Header 中 X-Cache 顯示 Miss from cloudfront（首次）或 Hit from cloudfront（快取命中）。S3 直連回傳 403 AccessDenied，代表 OAC 保護生效。這確保：(1) 所有流量經過 CloudFront 邊緣節點加速 (2) S3 不被直接訪問消耗流量 (3) 可統一管理存取策略與快取。若 CloudFront 仍 403，排查：確認 Distribution ID 正確 → Resource ARN 有 /* 結尾 → Bucket Policy JSON 語法正確 → 等待 1-2 分鐘傳播。

高可用架構概念（Unit 4）

單機架構的天花板

觀念

單機架構在面對真實流量時會遇到四個核心問題。

  • 單點故障：主機掛掉全站不可用
  • 無法水平擴展：流量超過單機極限
  • 零停機困難：更新部署需停機
  • 無法跨 AZ 容災：單一可用區域故障即全掛

Scale Up vs Scale Out

觀念

兩種擴展策略的差異與選擇。

  • Scale Up（垂直擴展）：加大單機（換更大 instance），有物理極限
  • Scale Out（水平擴展）：增加機器數量，理論上無上限
  • ALB + ASG 實現 Scale Out：自動增減 EC2 數量應對流量變化

ALB 核心概念

ALB 三大核心元件

觀念

Application Load Balancer 的核心架構與流量路徑。

元件說明
ALBApplication Load Balancer，接收流量入口
Listener監聽 Port，如 80 / 443
Rule路由規則，決定轉發到哪個 Target Group
Target Group一群 EC2，被分配流量
  • 流量路徑：Client → ALB DNS → Listener:80 → Rule 判斷 → Target Group → EC2s

Target Group 設定與健康檢查機制。

設定項值
Target typeinstance

Unhealthy threshold3

Launch Template

觀念

啟動範本定義了 EC2 的建立規格，ASG 依此建立新 EC2。

設定項說明

Auto Scaling Group (ASG)

觀念

ASG 透過 Target Group 與 ALB 串接，自動管理 EC2 數量。

關鍵數字建議值說明
Minimum1最少幾台
Desired2期望幾台
Maximum4最多幾台
Scaling 策略說明特性

ALB + ASG 建置實作

Step A: 建立 ALB Security Group

實作

名稱：aws12h-u4-alb-sg

  • 入站規則：HTTP:80 from 0.0.0.0/0

Step B: 修改 EC2 Security Group

實作

Port 80 來源改為 ALB SG（不再直接對外），這樣只有 ALB 能轉發到 EC2。

Step C: 建立 Target Group

實作

名稱：aws12h-u4-tg

  • Target type：instance
  • Health check：/health
  • 註冊現有 EC2

Step D: 建立 ALB

實作

名稱：aws12h-u4-alb

  • Scheme：Internet-facing
  • Subnet：選 2+ 個 AZ 的 Subnet
  • Security Group：aws12h-u4-alb-sg
  • Listener：HTTP:80 → aws12h-u4-tg
  • 記下 ALB DNS Name 測試

練習：ALB + ASG 驗證

練習

用 ALB DNS 訪問網站，確認正常。

優秀回答：透過 ALB DNS 訪問網站成功，代表流量路徑為：瀏覽器 → ALB(aws12h-u4-alb) → Listener:80 → Target Group(aws12h-u4-tg) → EC2。與單機架構的差異：(1) ALB 提供統一入口，後端 EC2 可動態增減 (2) Health Check 自動偵測不健康的 EC2 並停止轉發 (3) 搭配 ASG 可根據 CPU/流量自動擴縮 (4) 跨 AZ 部署提供容災能力。EC2 的 SG 改為只接受 ALB SG 來源，確保流量必須經過 ALB，不能直接連 EC2。
