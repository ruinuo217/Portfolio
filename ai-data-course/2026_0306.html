GCP 無伺服器架構

3/6 課程筆記 — Cloud Functions、Storage 觸發器、Cloud Run 容器化、Signed URL、前後端整合、無伺服器成本分析

無伺服器架構入門

傳統架構 vs 無伺服器

觀念

「無伺服器」不是沒有伺服器，而是開發者不需要管理伺服器。雲端平台自動處理所有基礎設施，開發者只需專注於程式碼。

比較項目 傳統架構 (VM) 無伺服器 (Serverless)
伺服器啟動 手動建立 VM 自動
OS 更新 手動 平台處理
擴展 手動增加 VM 自動擴展到數千
閒置成本 持續計費 趨近 $0
部署方式 SSH 上傳 直接上傳程式碼
適用場景 長時間運行 事件驅動 / 流量不穩定

為什麼需要無伺服器

觀念

以相簿系統為範例，說明無伺服器的核心價值。

  • 流量不穩定問題：平常 10 req/hr = VM 閒置 99%；爆紅 10,000 req/hr = VM 當機。無伺服器自動擴展，完美應對兩種情境
  • 自動化任務：上傳照片自動產生縮圖 — 只在事件發生時執行，其他時間成本 $0

Cloud Functions vs Cloud Run

觀念

GCP 提供兩種無伺服器運算服務，各有適用場景。

比較項目 Cloud Functions Cloud Run
程式結構 單一函數 完整應用
觸發方式 事件驅動 任何框架 (Flask / FastAPI / Express)
執行時間上限 最長 9 分鐘 最長 60 分鐘
記憶體上限 最大 8GB 支援 Docker 容器
適用場景 圖片處理、通知等 Web API、資料處理

課程情境 — 智慧相簿系統

系統架構設計

觀念

智慧相簿系統的完整無伺服器架構流程。

  • (1) 使用者瀏覽相簿 → 靜態網頁 (Cloud Storage)
  • (2) 上傳照片 → 呼叫 Cloud Run API
  • (3) API 儲存原圖 → 原圖 Bucket
  • (4) Storage 事件觸發 → Cloud Functions 縮圖處理
  • (5) 縮圖儲存 → 縮圖 Bucket
  • (6) 網頁顯示縮圖

VM 方案 vs 無伺服器方案

觀念

兩種方案的全面比較，突顯無伺服器的優勢。

比較項目 VM 方案 無伺服器方案
建置時間 10-15 分鐘 2-5 分鐘
擴展 手動 自動 0 到數千
閒置成本 $7/月 (e2-micro) $0
維護 需更新 OS、套件 無需
流量暴增 可能當機 自動應對
部署 SSH 上傳 直接上傳 / Git 整合

Cloud Functions 基礎

函數組成三要素

觀念

每個 Cloud Function 由三個核心元素組成。

  • (1) 觸發器：HTTP 觸發（接收 HTTP 請求）、Storage 觸發（檔案上傳/刪除時）、Pub/Sub 觸發（接收訊息）
  • (2) 執行環境：Python 3.11、Node.js 20、Go、Java 等
  • (3) 資源配置：記憶體 128MB ~ 8GB，逾時最長 540 秒（9 分鐘）

冷啟動 vs 熱啟動

觀念

Cloud Functions 的啟動機制直接影響回應速度。

  • 冷啟動：首次呼叫或閒置後，需 1-3 秒建立環境
  • 熱啟動：重複呼叫使用現有實例，延遲 <100ms
  • 回收機制：閒置 10-15 分鐘後回收實例

第一個 Cloud Function 實作

建立圖片資訊讀取函數

實作

建立一個 Cloud Function，讀取 Cloud Storage 中的圖片並回傳圖片資訊。

  • 準備：建立 Bucket [專案ID]-photos，上傳測試圖片
  • 函數名稱：get-image-info
  • 世代：Gen2
  • Region：asia-east1
  • Runtime：Python 3.11
  • Entry point：get_image_info
  • 驗證：允許公開存取
  • Memory：256 MiB
  • Timeout：60s

程式碼重點

實作

Cloud Function 的核心程式碼邏輯與依賴套件。

  • @functions_framework.http 標記 HTTP 觸發
  • request.args.get('file') 解析 URL 參數
  • storage.Client() 連接 Cloud Storage
  • blob.download_as_bytes() 下載到記憶體
  • Image.open(io.BytesIO()) 用 Pillow 讀取圖片資訊
  • requirements.txt：google-cloud-storage、Pillow、functions-framework

測試與驗證

實作

部署完成後進行功能測試。

  • 瀏覽器打開 https://FUNCTION-URL?file=test.jpg
  • 預期回傳 JSON 含 filename / size_bytes / width / height / format

練習：Cloud Function

練習

實作 Cloud Function 並驗證圖片資訊讀取功能。

優秀回答： 建立 get-image-info 函數後，用瀏覽器呼叫 ?file=test.jpg 回傳 JSON 包含圖片寬高、大小、格式。Cloud Functions Gen2 自動建立 HTTPS endpoint，部署後狀態為 Active。關鍵程式碼：@functions_framework.http 標記 HTTP 觸發，storage.Client() 透過服務帳號自動取得權限（不需手動設定金鑰）。冷啟動約 1-3 秒，連續呼叫走熱啟動 <100ms。常見錯誤：Bucket not found（名稱錯）、Access denied（服務帳號缺 Storage 權限）、Module not found PIL（requirements.txt 要寫 Pillow 不是 PIL）。

常見錯誤排查

觀念

Cloud Function 開發常見錯誤與解決方案。

錯誤訊息 原因
Bucket not found Bucket 名稱錯誤
Access denied 服務帳號缺 Storage 權限
Module not found PIL requirements.txt 要寫 Pillow 不是 PIL
Function timeout 增加逾時或優化程式碼
檔案不存在 檔名錯誤或未上傳

自動縮圖處理 — Storage 觸發器

為什麼需要縮圖

觀念

縮圖大幅減少頁面載入時間與流量消耗。

  • 原圖：每張 3-5MB，50 張相簿 = 200MB 載入 >30 秒
  • 縮圖：每張 50KB，50 張 = 2.5MB 載入 <3 秒
  • 效果：節省 98% 流量

Storage 觸發器原理

觀念

Cloud Storage 事件可自動觸發 Cloud Functions 執行。

  • finalize：上傳完成（最常用）
  • delete：刪除
  • archive：封存
  • metadataUpdate：更新
重要提醒： 避免無限迴圈！縮圖存到同一個 Bucket 會再次觸發事件 → 必須檢查路徑跳過 thumbs/ 前綴的檔案。

縮圖函數程式碼重點

實作

自動縮圖 Cloud Function 的核心邏輯。

  • @functions_framework.cloud_event 標記 Storage 觸發
  • cloud_event.data 取得 bucket / name / contentType
  • 跳過 thumbs/ 前綴（防無限迴圈）
  • 只處理 image/ 類型
  • image.thumbnail((300, 300)) 保持長寬比
  • RGBA 轉 RGB（JPEG 不支援透明）
  • 儲存到 thumbs/ 前綴
  • 資源配置：記憶體 512 MiB，逾時 120 秒

建立步驟（教師示範）

實作

在 GCP Console 建立自動縮圖 Cloud Function。

  • 函式名稱：generate-thumbnail
  • 觸發：Cloud Storage finalize/create
  • Bucket：[專案ID]-photos
  • Runtime：Python 3.11
  • Entry point：generate_thumbnail
  • 測試：上傳新圖片 → 等 5-10 秒 → 查看 thumbs/ 資料夾確認縮圖生成

練習：縮圖處理

練習

實作 Storage 觸發的自動縮圖功能。

優秀回答： 上傳 test.jpg（2.3MB）後約 5 秒在 thumbs/ 看到縮圖（45KB），壓縮率 98%。Storage 觸發器使用 finalize 事件，每次檔案上傳完成自動執行。防無限迴圈的關鍵：if file_name.startswith('thumbs/'): return — 因為縮圖存到同 Bucket 的 thumbs/ 會再觸發事件，不跳過就會無限執行。Pillow 的 thumbnail() 保持長寬比，800x600 原圖縮成 300x225。JPEG 不支援透明所以 RGBA 要先 convert('RGB')。記憶體設 512MiB 因為圖片處理比文字處理吃記憶體。

Cloud Run 相簿 API

Cloud Run 基礎概念

觀念

容器化無伺服器平台，可執行任何語言與框架。

比較項目 Cloud Functions Cloud Run
程式結構 單一函數 完整應用
框架 限制多 任何（Flask / FastAPI）
執行時間 9 分鐘 60 分鐘
容器 不需 Docker
冷啟動 較快 需載入容器

相簿 API 功能

實作

使用 Flask 框架建立相簿 API，透過 Cloud Console 直接部署（Source-based deploy 不需 Docker）。

  • GET /photos：列出所有照片，回傳 JSON 含原圖 URL + 縮圖 URL
  • POST /photos：產生 Signed URL 讓前端直傳 Cloud Storage

Signed URL 概念

觀念

讓前端直接上傳到 Cloud Storage，不經過 API 伺服器，節省頻寬。

  • API 只產生臨時上傳 URL（有效 15 分鐘）
  • 前端拿 URL 直接 PUT 到 Storage
  • 安全性：URL 過期就失效，限定 Bucket / 路徑 / 操作

部署步驟

實作

透過 Cloud Console 部署 Cloud Run 服務。

  • Cloud Run → 建立服務
  • 名稱：album-api
  • Region：asia-east1
  • 從原始碼部署
  • 允許未經驗證的叫用
  • 取得 URL：https://album-api-xxxx.asia-east1.run.app

前後端整合

靜態前端頁面

實作

HTML + CSS + JavaScript 部署到 Cloud Storage 靜態網站。

  • JavaScript 用 fetch() 呼叫 Cloud Run API 取得照片列表
  • 用 Signed URL 直接上傳到 Storage
  • 網格展示縮圖，點擊查看原圖

完整架構回顧

觀念

智慧相簿系統的完整無伺服器架構。

  • 使用者 → Cloud Storage 靜態網頁 → Cloud Run API → Cloud Storage 原圖 Bucket → Cloud Functions 自動縮圖 → Cloud Storage 縮圖 Bucket → 網頁顯示
  • 所有元件都是無伺服器：無請求時成本 $0、自動擴展、免維護

成本分析

無伺服器成本計算

觀念

無伺服器方案在小型應用中的成本優勢極為明顯。

  • Cloud Functions：每月 200 萬次免費，超過 $0.40/百萬次
  • Cloud Run：每月 200 萬次免費，CPU/記憶體按秒計費
  • Cloud Storage：$0.020/GB/月（Standard）
  • 對比 VM 方案：e2-micro 持續運行 = $7/月，無伺服器小型應用 ≈ $0-2/月
Tip: 無伺服器特別適合流量不穩定的應用 — 沒人用時真的 $0。

整合思考題

練習

假設你要為一家餐廳建立線上點餐系統，請設計使用 GCP 服務的架構方案。

優秀回答： (1) 前端：Cloud Storage 託管靜態菜單頁面（HTML/CSS/JS），搭配 Cloud CDN 加速。(2) 後端 API：Cloud Run 部署 Flask/FastAPI 點餐 API，處理訂單 CRUD，連接 Cloud SQL（MySQL）。(3) 圖片處理：菜品照片存 Cloud Storage，上傳時 Cloud Functions 自動生成縮圖。(4) 通知：Cloud Functions 監聽新訂單事件，透過 Pub/Sub 觸發推播通知（LINE Notify / Email）。(5) 成本：非用餐時段流量低 → Cloud Run 自動縮至 0 實例 → 成本趨近 $0。(6) 與全 VM 方案比較：不需維護 OS、自動擴展尖峰（午餐時段）、閒置不計費，預估月費從 $50+（2台VM）降至 $5-10。
