PostgreSQL 基礎與資料處理

3/23 課程筆記 — PostgreSQL vs MySQL、層級結構與 Schema、COPY 大數據匯入、COALESCE/CAST/TO_CHAR、CTE 與 JSONB

PostgreSQL vs MySQL — 為什麼選 PostgreSQL

四大優勢場景

觀念

PostgreSQL 在以下四大場景中明顯優於 MySQL，是企業級分析的首選。

#場景PostgreSQL 優勢
1複雜查詢視窗函數 + CTE 完整支援
2JSON 半結構化JSONB 比 MySQL JSON 快 20-50 倍
3陣列型別原生 TEXT[] / INTEGER[] 支援
4精細權限Schema 隔間，部門級隔離
注意：MySQL 8.0 才補上 CTE 和視窗函數，但仍有限制（不支援 GROUPS / EXCLUDE / IGNORE NULLS）。

MySQL → PostgreSQL 差異速查

觀念

從 MySQL 轉換到 PostgreSQL 時，需要注意以下語法差異。

項目MySQLPostgreSQL
大小寫不區分區分 → 全用小寫
日期格式化DATE_FORMAT()TO_CHAR()
自動遞增AUTO_INCREMENTSERIAL
字串引號單引號或雙引號必須單引號
字串連接CONCAT()|| 運算子

決策表

觀念

根據不同需求場景，選擇合適的資料庫。

需求場景推薦
簡單 CRUD都可以
複雜分析PostgreSQL
JSON 處理PostgreSQL
高效能簡單查詢MySQL
企業級應用PostgreSQL

MySQL vs PostgreSQL 選擇考量

觀念

根據團隊現狀與未來發展方向做出最佳選擇。

  • 適合 MySQL：標準 Web CRUD / 讀寫單純 / 團隊既有經驗
  • 適合 PostgreSQL：交易 + 分析混合 / CTE 視窗函數 / JSONB / 彈性索引 / 長期演進
結論：PostgreSQL 同一套引擎可同時做交易與分析，SQL 完整可擴充，開源不被綁死。

練習：PostgreSQL vs MySQL

練習
優秀回答：「選擇 PostgreSQL 的核心理由：(1) 視窗函數完整支援——計算排名、累計、環比不需要複雜子查詢 (2) CTE 讓複雜查詢模組化——80 行巢狀子查詢變成 3 個各 10 行的清晰步驟 (3) JSONB 比 MySQL JSON 快 20-50 倍——預分類 + GIN 索引 (4) Schema 層級權限隔離——銷售部看不到物流部資料。MySQL 8.0 雖補上視窗函數但仍有限制（不支援 GROUPS / EXCLUDE / IGNORE NULLS）。兩者 SQL 語法 90% 相同，主要差異：大小寫敏感（全用小寫避免）、TO_CHAR 取代 DATE_FORMAT、SERIAL 取代 AUTO_INCREMENT、字串必須單引號（安全特性防 SQL 注入）。」

PostgreSQL 必知基礎

觀念

用公司比喻理解 PostgreSQL 的六層架構，vs MySQL 無 Schema 隔間。

PG 層級公司比喻

資料型別

觀念

PostgreSQL 獨有的進階型別，讓資料處理更靈活。

  • JSONB：直接存取 + 查詢 JSON，建 GIN 索引，使用 -> / ->> 操作子
  • TEXT[] / INTEGER[]：原生陣列型別，用 ANY() 查詢
  • 基本型別：INT / TEXT / VARCHAR(255) / TIMESTAMP / DECIMAL / BOOLEAN

使用者與權限

觀念

透過 CREATE USER 和 GRANT 精細控管每位使用者的存取範圍。

指令用途
CREATE USER建立新使用者
GRANT SELECT授予查詢權限
GRANT INSERT授予新增權限
REVOKE撤銷權限
ALTER ROLE修改角色屬性
  • 場景：分析師只給 SELECT，部門間用 Schema 隔離

表的生命週期

實作

從建表到查詢的完整流程。

  • CREATE TABLE：定義欄位、PRIMARY KEY、FOREIGN KEY
  • ALTER TABLE：ADD COLUMN / DROP COLUMN 修改結構
  • INSERT INTO：新增資料
  • SELECT...WHERE：條件查詢
觀念：PRIMARY KEY = 唯一識別碼，像身份證號一樣確保每筆資料都能被精確定位。

索引與 EXPLAIN

觀念

CREATE INDEX 可加速查詢 100 倍，EXPLAIN 用來診斷效能。

  • EXPLAIN 診斷：Seq Scan（慢，全表掃描）→ Index Scan（快，索引查找）
  • 何時建索引：常用 WHERE / JOIN / ORDER BY 的欄位

練習：PostgreSQL 基礎

練習
優秀回答：「PostgreSQL 層級結構的實務價值：Database（銷售部）→ Schema（零售團隊 / 批發團隊）→ Table（日報 / 客戶檔案）。分析師小王只有銷售部的 SELECT 權限，(1) 可查詢銷售紀錄 ✅ (2) 不能刪除訂單 ❌（無 DELETE 權限）(3) 看不到物流部 ❌（無該 DB 權限）。PRIMARY KEY 像身份證號——100 個叫『張三』的客戶中用 customer_id=1001 精確定位。索引像書籍目錄：無索引 = 逐頁翻（10 秒），有索引 = 查目錄直接跳（0.1 秒）。EXPLAIN 輸出 Seq Scan 代表需要建索引。」

第一堂課 — 建立數據基地與大數據匯入

Olist 電商資料集

觀念

巴西最大電商平台的真實資料，涵蓋 100,063 筆訂單 / 32,951 位客戶 / 73,099 件評論，時間跨度 2016-2018。

資料表主要欄位
ordersorder_id / customer_id / status / timestamps
customerscustomer_id / city / state
order_itemsorder_id / product_id / price
order_paymentsorder_id / payment_type / payment_value
order_reviewsorder_id / review_score / comment
  • 關聯結構：customers ← orders → items / payments / reviews

環境設定

實作

三種環境擇一建置 PostgreSQL 實作環境。

  • GCP Cloud SQL（推薦）：全託管服務，免維護
  • Docker：docker run postgres:15，本地快速啟動
  • GCP VM：apt install postgresql，完整控制

COPY vs INSERT

觀念

批量匯入與逐筆寫入的效能差異高達 3000 倍。

  • INSERT 逐筆：每筆讀取 → 驗證 → 寫入 → 確認，100K 筆 = 6+ 小時
  • COPY 批量：一次讀取整個 CSV，100K 筆 = 5-10 秒，快 3000 倍
  • 語法：COPY table FROM '/path.csv' WITH (FORMAT csv, HEADER true)
匯入順序重要！先 customers → orders → items / payments / reviews（外鍵參照）

建表 SQL

實作

7 個 CREATE TABLE 含 PRIMARY KEY / FOREIGN KEY / VARCHAR / TIMESTAMP / DECIMAL。

  • Cloud SQL：使用 GCS Import 匯入資料
  • 常見錯誤：Permission denied / Invalid encoding / 欄位不符

練習：資料匯入

練習
優秀回答：「COPY vs INSERT 效能差異的根本原因：INSERT 每筆都經歷完整的讀取 → 解析 → 驗證 → 寫入 → 確認循環（5 個步驟 x 100,000 筆 = 500,000 次 IO 操作）。COPY 批量處理：一次讀取整個檔案到記憶體 → 批量解析 → 批量驗證 → 批量寫入（只做 4 次大型 IO）。匯入順序遵循外鍵依賴：customers 先（被 orders 參照）→ orders 次（被 items / payments / reviews 參照）。建表時 FOREIGN KEY (customer_id) REFERENCES olist_customers(customer_id) 確保每筆訂單的 customer_id 都存在於客戶表——防止資料孤兒。Cloud SQL 匯入用 GCS bucket：上傳 CSV → Console Import → 選擇目標表。」

第二堂課 — 數據清洗與半結構化資料

COALESCE 空值替代

實作

NULL 不等於 0、不等於空字串、不等於 false，它是「沒有值」。NULL 具有傳染性：100 + NULL = NULL。

  • COALESCE(值1, 值2, ...)：按順序找第一個非 NULL 的值
  • 範例：COALESCE(delivered_date, estimated_date, approved_at, purchase_timestamp)

CAST 型別轉換

實作

確保數值運算正確，避免文字拼接的陷阱。

  • 語法：CAST(value AS DECIMAL) 或 value::DECIMAL
  • 不轉換：'299.99' + '150.00' = '299.99150.00'（文字拼接）❌
  • 轉換後：299.99 + 150.00 = 449.99 ✅

TO_CHAR 日期格式化

實作

將時間戳轉換為可讀的日期字串格式。

  • 語法：TO_CHAR(ts, 'YYYY-MM') → '2023-03'
  • 季度：'YYYY"Q"Q' → '2023Q1'
格式碼說明範例
YYYY四位年份2023
MM月份03
DD日期23
HH2424 小時制14
MI分鐘30
SS秒45

CTE (WITH 語法)

觀念

巢狀子查詢像俄羅斯套娃（難讀 / 難維護），CTE 讓查詢模組化（清晰 / 可逐步驗證）。

  • 語法：WITH cte1 AS (...), cte2 AS (...) SELECT ...
  • 實戰：3 步 CTE 計算客戶購買次數 + 總支出

JSONB 操作

觀念

PostgreSQL 獨有的 JSONB 型別，讓半結構化資料查詢效能飛躍。

  • -> 提取 JSONB，->> 提取文字
  • 巢狀存取：specs->'cpu'
  • 陣列查詢：@> 包含查詢
  • GIN 索引：<50ms vs JSON 純文字 5000ms+
  • JSONB vs JSON：JSONB 二進制預處理（快），JSON 純文字每次解析（慢）

練習：數據清洗

練習
優秀回答：「數據清洗三大工具實戰：(1) COALESCE 處理 NULL——COALESCE(delivered_date, estimated_date, purchase_timestamp) 確保永遠有日期值，避免報表出現空白。實務中 80% 訂單有 delivery 日期，15% 只有 estimated，5% 只有 purchase (2) CAST 防止型別錯誤——payment_value 可能被存為 TEXT，'299.99' + '150.00' 會變成文字拼接不是數學加法 (3) TO_CHAR 統一日期格式——GROUP BY TO_CHAR(ts, 'YYYY-MM') 做月度統計。CTE 的威力：把 SELECT...FROM (SELECT...FROM (SELECT...)) 三層巢狀拆成 WITH step1 AS (...), step2 AS (...), step3 AS (...) 三個清晰步驟，每步可獨立執行驗證。JSONB 效能關鍵：建 GIN 索引後查詢 <50ms，不建索引可能 5000ms+，差 100 倍。」
