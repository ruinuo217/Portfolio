Git 分支與進階技巧

分支管理、合併策略、rebase、tag、Git Flow 與實用工具

Part 1: 分支概念與操作

為什麼需要分支

重要概念

分支 (Branch) 是 Git 最強大的功能之一，讓團隊能高效協作開發：

  • 平行開發 (Parallel Development)：多位開發者可以同時在不同分支上開發不同功能，互不干擾
  • 隔離環境 (Isolation)：每個分支是獨立的工作環境，實驗性修改不會影響主線程式碼
  • 團隊協作 (Collaboration)：透過 Pull Request / Merge Request 進行程式碼審查，確保品質
  • 敏捷開發 / CI-CD：分支策略與持續整合、持續部署流程緊密結合，實現快速迭代
分支本質上是一個指向 commit 的指標 (pointer)，建立分支幾乎不花時間與空間。

git branch 基本指令

必學
指令 說明
git branch <name> 新建分支（不會自動切換）
git switch <name> 切換分支（Git 2.23+ 推薦用法）
git checkout <name> 切換分支（傳統用法）
git switch -c <name> 新建分支並切換（推薦）
git checkout -b <name> 新建分支並切換（傳統）
git branch -D <name> 強制刪除本地分支
git push origin --delete <name> 刪除遠端分支
git fetch --prune 清理已刪除的遠端分支參考
git log --graph --oneline 以圖形方式查看版本線圖 
# 常用操作流程
git branch                     # 列出所有本地分支
git branch -a                  # 列出所有分支（含遠端）
git switch -c feature/login    # 建立並切換到新分支
git branch -d feature/login    # 刪除已合併的分支（-d 安全刪除）
git branch -D feature/login    # 強制刪除（不管是否合併）

Part 2: git merge 合併分支

合併流程

基本操作

合併的核心概念：先切換到目標分支，再將來源分支合併進來。

# 將 feature/login 合併到 main
git switch main                # 1. 切換到目標分支
git merge feature/login        # 2. 合併來源分支
git branch -d feature/login    # 3. 刪除已合併的分支
記住口訣：「站在目標，拉進來源」。你在哪個分支上執行 merge，就是把另一個分支合併到當前分支。

衝突處理 (Merge Conflict)

重要

當兩個分支修改了同一個檔案的同一個區域，Git 無法自動合併，就會產生衝突。

# 衝突標記格式
<<<<<<< HEAD
這是目標分支的內容
=======
這是來源分支的內容
>>>>>>> feature/login

衝突解決步驟：

  • 打開衝突檔案，手動選擇要保留的內容（刪除衝突標記）
  • 使用 git add <file> 標記衝突已解決
  • 使用 git commit 完成合併提交
# 如果想放棄合併
git merge --abort              # 回到合併前的狀態

# 解決衝突後
git add .                      # 標記所有衝突已解決
git commit                     # 完成合併 commit

Fast-Forward 機制

進階

當目標分支沒有新的 commit 時，Git 預設會使用 Fast-Forward 合併，不產生合併 commit。

# Fast-Forward 前：main 沒有新 commit
main:     A --- B
                 \
feature:          C --- D

# Fast-Forward 後：main 指標直接移動
main:     A --- B --- C --- D
# 強制產生合併 commit（保留分支歷史）
git merge --no-ff feature/login

# 只允許 Fast-Forward（無法 FF 時會失敗）
git merge --ff-only feature/login
建議使用 --no-ff 合併，可以在 log 中清楚看到分支的起始與結束點，方便追蹤開發歷史。

Part 3: git rebase 變基

rebase vs merge 比較

重要觀念
特性 git merge git rebase
歷史紀錄 保留完整分支歷史，產生合併 commit 線性歷史，看起來像在同一條線上開發
Commit 變化 不改變現有 commit 重新建立新的 commit（hash 值改變）
適用場景 公開分支、團隊協作 個人分支、整理歷史
衝突處理 一次性解決所有衝突 逐一 commit 解決衝突
安全性 較安全，不改寫歷史 較危險，會改寫歷史

基本用法

實作

在 feature 分支上執行 rebase，將分支的基底移到 main 的最新 commit：

# rebase 前
main:     A --- B --- E
                 \
feature:          C --- D

# 在 feature 分支上執行 rebase
git switch feature
git rebase main

# rebase 後：C、D 變成 C'、D'（新的 hash）
main:     A --- B --- E
                       \
feature:                C' --- D'
# 如果 rebase 過程中產生衝突
# 1. 解決衝突檔案
git add <file>
git rebase --continue          # 繼續 rebase

# 或放棄 rebase
git rebase --abort             # 回到 rebase 前的狀態

Interactive Rebase

進階

互動式 rebase 可以重新編排、合併、修改最近的 commit：

git rebase -i HEAD~3           # 編輯最近 3 個 commit
命令 縮寫 說明
pick p 保留此 commit（預設）
reword r 保留 commit 但修改訊息
edit e 保留 commit 但暫停以修改內容
squash s 合併到上一個 commit，保留訊息
drop d 刪除此 commit
絕對禁止：不要對已經推送 (push) 到共用分支的 commit 執行 rebase！這會改寫歷史，導致其他協作者的程式碼產生嚴重衝突。

Part 4: git tag 標籤管理

Tag vs Branch 比較

觀念
特性 Tag Branch
用途 標記特定版本（如 v1.0.0） 開發工作線
是否移動 固定不動，永遠指向同一個 commit 隨著新 commit 不斷移動
典型命名 v1.0.0、v2.1.3-beta main、feature/login
生命週期 通常永久保留 合併後通常刪除

Tag 常用指令

必學
指令 說明
git tag 列出所有 tag
git tag -a v1.0.0 -m "Release 1.0" 建立附註標籤（推薦）
git tag v1.0.0 建立輕量標籤
git tag -d v1.0.0 刪除本地 tag
git push origin --tags 推送所有 tag 到遠端
git show v1.0.0 查看 tag 詳細資訊
git checkout -b hotfix v1.0.0 從 tag 建立新分支

Lightweight vs Annotated Tags

進階

輕量標籤 (Lightweight)：僅是一個指向 commit 的指標，不包含額外資訊。

git tag v1.0.0                 # 建立輕量標籤

附註標籤 (Annotated)：包含建立者、日期、訊息等完整資訊，儲存為獨立的 Git 物件。

git tag -a v1.0.0 -m "First stable release"
git show v1.0.0                # 可查看完整標籤資訊
正式發佈版本建議使用 Annotated Tag，方便記錄發佈時間與說明。版本號建議遵循 Semantic Versioning：MAJOR.MINOR.PATCH（如 v2.1.3）。

Part 5: Git Flow 分支策略

Git Flow 工作流程

團隊必備

Git Flow 是一套成熟的分支管理策略，適用於有明確發佈週期的專案：

分支名稱 用途 來源 合併目標 生命週期
main / master 正式發佈版本，每個 commit 都是可部署的 - - 永久
develop 開發主線，整合所有功能分支 main release, main 永久
feature/* 功能開發（如 feature/login） develop develop 短期
bugfix/* 修正開發中的 bug develop develop 短期
hotfix/* 修正線上緊急 bug main main + develop 短期
release/* 發佈準備，最後測試與修正 develop main + develop 短期

推薦的工作流程：

# 1. 開始新功能
git switch -c feature/shopping-cart develop

# 2. 開發完成後合併回 develop
git switch develop
git merge --no-ff feature/shopping-cart
git branch -d feature/shopping-cart

# 3. 準備發佈
git switch -c release/1.0.0 develop
# ... 測試、修正 bug ...

# 4. 發佈到 main
git switch main
git merge --no-ff release/1.0.0
git tag -a v1.0.0 -m "Release 1.0.0"

# 5. 同步回 develop
git switch develop
git merge --no-ff release/1.0.0
git branch -d release/1.0.0
小型團隊或快速迭代的專案，也可以考慮較簡化的 GitHub Flow（只有 main + feature branches）或 Trunk-Based Development。

Part 6: 實用工具

.gitignore

必學

透過 .gitignore 檔案告訴 Git 哪些檔案不需要追蹤：

# .gitignore 基本規則
*.log              # 忽略所有 .log 檔案
build/             # 忽略整個資料夾
!important.log     # 例外：不忽略此檔案
doc/*.pdf          # 忽略 doc 下的 pdf
**/temp            # 忽略任何路徑下的 temp

# Python 專案常見
__pycache__/
*.pyc
.venv/
.env

# Node.js 專案常見
node_modules/
.env
dist/
# 如果檔案已經被追蹤，需要先移除快取
git rm --cached <file>          # 從 Git 移除但保留本地檔案
git rm -r --cached <folder>    # 移除整個資料夾的追蹤

git stash 暫存

實用

暫時保存未完成的修改，讓工作目錄恢復乾淨狀態：

git stash                      # 暫存目前的修改
git stash save "WIP: login form" # 附帶說明訊息
git stash list                 # 列出所有暫存
git stash pop                  # 恢復最近的暫存並刪除
git stash apply                # 恢復最近的暫存但保留
git stash drop                 # 刪除最近的暫存
常見場景：正在開發功能時，突然要切換分支修 bug，就先 git stash 保存進度。

git cherry-pick

進階

從其他分支挑選特定 commit 套用到當前分支：

# 挑選單一 commit
git cherry-pick abc1234

# 挑選多個 commit
git cherry-pick abc1234 def5678

# 挑選一個範圍（不包含起始 commit）
git cherry-pick abc1234..def5678

# 加上 -x 旗標，自動在 commit 訊息加上來源 hash
git cherry-pick -x abc1234

# 衝突時
git cherry-pick --abort        # 放棄
git cherry-pick --continue     # 解決衝突後繼續

git reflog 參考日誌

救命工具

reflog 記錄了 HEAD 的所有移動歷史，是 Git 的「後悔藥」：

git reflog                     # 查看 HEAD 移動紀錄

# 找回被 reset 掉的 commit
git reflog                     # 找到目標 commit 的 hash
git reset --hard HEAD@{3}      # 回到 3 步之前的狀態

# 找回被刪除的分支
git reflog                     # 找到分支最後的 commit hash
git switch -c recovered-branch abc1234
reflog 只記錄本地操作，預設保留 90 天。只要 commit 還在 reflog 中，就有救！

git blame 追溯

實用

逐行查看檔案的最後修改者與 commit 資訊：

# 查看整個檔案的 blame
git blame <file>

# 只查看特定行數範圍（第 10~20 行）
git blame -L 10,20 <file>

# 顯示完整格式（含作者 email、時間戳記）
git blame -p <file>
git blame 常用於追蹤某行程式碼是誰、在哪個 commit 修改的，方便溝通與除錯。

Part 7: git fetch / git pull

git fetch vs git pull

重要區別
特性 git fetch git pull
動作 只下載遠端更新，不修改工作目錄 下載遠端更新 + 自動合併 (fetch + merge)
安全性 較安全，可先檢視再決定合併 較方便但可能產生意外衝突
適用場景 想先查看遠端變更再決定如何合併 確定要直接同步最新程式碼 
# fetch：下載但不合併
git fetch origin               # 下載遠端所有分支更新
git log origin/main --oneline  # 查看遠端 main 的新 commit
git merge origin/main          # 手動合併

# pull：下載並合併（等同 fetch + merge）
git pull origin main

Pull 的 Fast-Forward 策略

進階

在 pull 時也可以控制合併策略：

# 只允許 Fast-Forward（推薦，避免不必要的合併 commit）
git pull --ff-only

# 使用 rebase 代替 merge（保持線性歷史）
git pull --rebase

# 設定為預設行為
git config pull.ff only        # 預設 --ff-only
git config pull.rebase true    # 預設 --rebase
建議在個人分支使用 git pull --rebase 保持乾淨的線性歷史；在共用分支使用 git pull --ff-only 確保安全。
