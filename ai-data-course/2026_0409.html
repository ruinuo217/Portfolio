LangChain 與 AI Agent 整合

4/9 課程筆記 — LangChain Chain/Tool/Agent、ReAct 決策模式、LangChain RAG、State 狀態管理、LangGraph 概念

行為層 — Orchestration 概念

誰決定AI下一步

觀念

系統偵測到違規——要查法規？發通知？記錄？步驟的順序和條件由行為層決定。

  • Chain（固定流程）：預先定義好的步驟順序，A → B → C
  • Agent（動態決策）：LLM 根據情境自己決定下一步
  • Tool（LLM 與系統的邊界）：LLM 不該直接控制系統——透過 Tool 間接操作
核心原則：LLM 不該直接控制系統——透過 Tool 間接操作，確保安全與可控性。

Chain vs Agent

觀念

兩種編排策略的核心差異比較：

特性ChainAgent
流程固定步驟動態決策
執行方式A → B → CLLM 決定下一步
穩定性可預測靈活但不穩定
適用場景流程明確的任務需要推理的任務
  • Chain 適合：報表生成 / 資料清洗 / 固定流程
  • Agent 適合：問答系統 / 複雜判斷 / 需要工具的任務

LangChain 基礎

LangChain 是什麼

觀念

LLM 應用開發框架，提供統一介面串接各種 LLM 與外部工具。

  • Model：呼叫 LLM（Gemini / OpenAI / HuggingFace）
  • Prompt Template：動態生成 prompt
  • Chain：串接多步驟
  • Tool：讓 LLM 呼叫外部功能
  • Agent：讓 LLM 自己決定用什麼工具
  • Memory：對話記憶

安裝：

pip install langchain langchain-google-genai langchain-openai

Prompt Template

實作

動態插入變數而非硬編 prompt，讓同一個模板適用不同情境。

from langchain.prompts import ChatPromptTemplate

template = ChatPromptTemplate.from_messages([
    ("system", "你是{role}"),
    ("human", "{question}")
])

prompt = template.invoke({
    "role": "工安專家",
    "question": "安全帽規定"
})
重點：用變數取代硬編，讓 prompt 可重複使用、易於維護。

LangChain Chain

實作

用 LCEL（LangChain Expression Language）語法，把 prompt 生成 → 模型呼叫 → 輸出解析串成一條線。

# LCEL 語法：用 | 串接各元件
chain = prompt | model | output_parser

# 一行搞定：prompt 生成 → 模型呼叫 → 輸出解析
result = chain.invoke({
    "role": "工安專家",
    "question": "安全帽規定"
})

LangChain Tool

Tool 定義

實作

用 @tool 裝飾器把普通 function 變成 LangChain Tool。

from langchain.tools import tool

@tool
def check_regulation(violation_type: str) -> str:
    """查詢違規法規"""
    # 實作查詢邏輯
    return f"{violation_type} 的相關法規..."

跟 Gemini Function Calling 的關係：Tool 是框架封裝，底層邏輯一樣（LLM 選擇 → 你執行 → 送回）。

自訂 Tool 範例

實作

跟 Gemini 原生 FC 的差異：@tool 取代 docstring + type hint，LangChain 自動處理兩段式流程（不需手動 while loop）。

@tool
def check_regulation(violation_type: str) -> str:
    """查詢違規法規"""
    regulations = {
        "未戴安全帽": "職安法第6條...",
        "未穿反光背心": "營造安全衛生設施標準第11條..."
    }
    return regulations.get(violation_type, "查無相關法規")

@tool
def send_alert(message: str) -> str:
    """發送告警通知"""
    return f"已發送告警: {message}"

Workshop：Tool 實作

練習

LangChain Tool vs Gemini Function Calling 比較：

關鍵差異：

Gemini 原生：手動定義 docstring + 手動 while loop 處理兩段式 + 手動送回 function_response。

LangChain：@tool 裝飾器自動從 docstring 提取 schema + Agent 自動執行工具 + 自動處理結果。

LangChain 的價值：把 Gemini FC 那 20 行 while loop 簡化成 3 行。但你必須先懂原生 FC，才能 debug LangChain 出問題時發生了什麼。

LangChain Agent

Agent 概念

觀念

Agent = LLM + Tools + 決策循環

ReAct 模式的運作流程：

  • Thought（思考）：分析目前的狀況
  • Action（呼叫工具）：選擇並執行適當的工具
  • Observation（觀察結果）：接收工具回傳的結果
  • 重複：直到有最終答案為止
from langchain.agents import create_react_agent

agent = create_react_agent(model, tools)

Agent 實作

實作

Agent 自動決定用哪個工具、什麼順序。

from langchain.agents import create_react_agent

agent = create_react_agent(
    model,
    tools=[check_regulation, send_alert]
)

result = agent.invoke({
    "input": "A區有人沒戴安全帽"
})
# Agent 自動：
# 1. 思考 → 需要查法規
# 2. 呼叫 check_regulation("未戴安全帽")
# 3. 觀察結果 → 需要發通知
# 4. 呼叫 send_alert("A區違規...")
# 5. 回傳最終結果

Agent 的風險

觀念

Agent 雖然靈活，但在生產環境需要特別注意以下風險：

  • 不穩定：同一問題不同次可能選不同工具 / 不同順序
  • 成本高：每個決策都呼叫 LLM，Token 消耗大
  • 安全風險：Agent 可能呼叫不該呼叫的工具

緩解策略：

  • 限制可用工具數量與範圍
  • 設定最大步數（max_iterations）
  • 加入人工審核（Human-in-the-loop）
重要：Agent 適合探索性任務，不適合需要 100% 一致性的生產系統——生產系統用 Chain 更穩定。

LangChain RAG

LangChain RAG Pipeline

實作

LangChain 提供完整的 RAG 管線，從讀文件到生成回答一條龍。

# 1. 讀文件
from langchain.document_loaders import PyPDFLoader, CSVLoader, WebBaseLoader
loader = PyPDFLoader("safety_regulations.pdf")
docs = loader.load()

# 2. 切段
from langchain.text_splitter import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(docs)

# 3. 向量 DB
from langchain.vectorstores import Chroma  # 或 FAISS
vectorstore = Chroma.from_documents(chunks, embedding_model)

# 4. 檢索
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 3}
)

# 5. 生成
from langchain.chains import RetrievalQA
qa_chain = RetrievalQA.from_chain_type(
    llm=model,
    retriever=retriever
)

vs 手寫 RAG

觀念

兩種 RAG 實作方式的比較：

特性手寫 RAGLangChain RAG
相似度計算自己算 cosine similarity自動處理 embedding + 搜尋
Prompt 組裝自己組 prompt自動處理 prompt
優點靈活但繁瑣簡潔但黑盒
建議：先學手寫理解原理，再用 LangChain 提高效率。出問題時才知道該從哪裡 debug。

狀態與流程層概念

State 是什麼

觀念

LLM 沒有記憶——每次呼叫都是獨立的。系統需要 State 追蹤：

  • 累積的違規紀錄
  • 對話歷史
  • 流程進度
class SafetyState:
    violations: list    # 違規紀錄列表
    alert_count: int    # 告警次數

# Function Calling 結果存進 State
# → 系統有記憶

Chain vs Graph

觀念

從線性流程進化到有狀態的圖形結構：

  • Chain：A → B → C 固定流程
  • Graph：根據狀態決定下一步（可能 A → B → A → C）
  • LangGraph：有狀態的 Agent——每個節點是一個步驟，邊是條件判斷

適合：多步驟流程 / 需要回退 / 需要人工審核的系統。

語意角色 vs Chatbot

觀念

兩種不同的 AI 互動模式：

  • Chatbot：開放式聊天，難以控制品質
  • 語意角色（Semantic Role）：引導使用者完成特定流程（報稅 / 申請 / 檢核）

語意角色的系統設計：

  • 明確的狀態：使用者在哪一步
  • 明確的動作：下一步該做什麼
  • 明確的邊界：什麼時候需要人工介入

五層架構回顧

觀念

從底層到頂層的完整 LLM 系統架構：

  • 模型層：LLM 能做什麼、不能做什麼
  • 感知層：多模態輸入轉結構化
  • 行為層：Function Calling / RAG 讓 LLM 與系統互動
  • 編排層：Chain / Agent / Tool 決定流程
  • 狀態層：State / Graph 管理記憶與流程
核心觀念：LLM 不是系統，是系統中的元件。

整合思考題 + HW

練習

題目：設計一個「工地安全 AI 助手」的完整系統架構。

完整系統設計：

(1) 感知層：YOLO 偵測安全帽 / 背心 → Event JSON（class / confidence / bbox / timestamp）

(2) 模型層：Gemini 分析圖片內容（Structured Output 保證 JSON 格式）；Embedding 將法規轉向量存入向量 DB

(3) 行為層：Function Calling 定義 3 個工具（check_regulation / send_alert / log_incident）；RAG 搜尋相關法規段落塞進 prompt

(4) 編排層：用 Chain 固定流程（偵測 → 查法規 → 發告警 → 記錄）而非 Agent（生產系統要穩定）；每步結果存入 SafetyState

(5) 狀態層：SafetyState 追蹤累積違規 / 告警次數 / 處理歷史。N 次違規升級告警嚴重度。

(6) API 層：FastAPI 接收圖片（POST /detect）→ Schema 驗證 → Service 處理 → Guardrail 防呆 → 回傳決策

(7) 防護：System prompt 設定角色邊界 + 輸入過濾攻擊模式 + 輸出檢查敏感資訊

技術選型：YOLO（感知）+ Gemini（多模態分析 + FC）+ RAG（法規搜尋）+ FastAPI（API 入口）+ Pydantic（Schema 驗證）+ LangChain Chain（流程編排）。

核心原則：LLM 不直接操作系統，透過 Tool 間接使用；生產系統用 Chain 不用 Agent；Schema 驗證在 handler 前執行；每個決策都是事件，都要記錄。
