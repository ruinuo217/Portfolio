LangChain 與 AI Agent 整合

LangChain Chain/Tool/Agent、ReAct 決策模式、LangChain RAG、State 狀態管理、LangGraph 概念

行為層 — Orchestration 概念

誰決定AI下一步

觀念

系統偵測到違規——要查法規？發通知？記錄？步驟的順序和條件由行為層決定。

  • Chain（固定流程）：預先定義好的步驟順序，A → B → C
  • Agent（動態決策）：LLM 根據情境自己決定下一步
  • Tool（LLM 與系統的邊界）：LLM 不該直接控制系統——透過 Tool 間接操作
核心原則：LLM 不該直接控制系統——透過 Tool 間接操作，確保安全與可控性。

Chain vs Agent

觀念

兩種編排策略的核心差異比較：

特性ChainAgent
流程固定步驟動態決策
執行方式A → B → CLLM 決定下一步
穩定性可預測靈活但不穩定
適用場景流程明確的任務需要推理的任務
  • Chain 適合：報表生成 / 資料清洗 / 固定流程
  • Agent 適合：問答系統 / 複雜判斷 / 需要工具的任務

LangChain 基礎

LangChain 是什麼

觀念

LLM 應用開發框架，提供統一介面串接各種 LLM 與外部工具。

  • Model：呼叫 LLM（Gemini / OpenAI / HuggingFace）
  • Prompt Template：動態生成 prompt
  • Chain：串接多步驟
  • Tool：讓 LLM 呼叫外部功能
  • Agent：讓 LLM 自己決定用什麼工具
  • Memory：對話記憶

安裝：

pip install langchain langchain-google-genai langchain-openai

Prompt Template

實作

動態插入變數而非硬編 prompt，讓同一個模板適用不同情境。

from langchain.prompts import ChatPromptTemplate

template = ChatPromptTemplate.from_messages([
    ("system", "你是{role}"),
    ("human", "{question}")
])

prompt = template.invoke({
    "role": "工安專家",
    "question": "安全帽規定"
})
重點：用變數取代硬編，讓 prompt 可重複使用、易於維護。

LangChain Chain

實作

用 LCEL（LangChain Expression Language）語法，把 prompt 生成 → 模型呼叫 → 輸出解析串成一條線。

# LCEL 語法：用 | 串接各元件
chain = prompt | model | output_parser

# 一行搞定：prompt 生成 → 模型呼叫 → 輸出解析
result = chain.invoke({
    "role": "工安專家",
    "question": "安全帽規定"
})

LangChain Tool

Tool 定義

實作

用 @tool 裝飾器把普通 function 變成 LangChain Tool。

from langchain.tools import tool

@tool
def check_regulation(violation_type: str) -> str:
    """查詢違規法規"""
    # 實作查詢邏輯
    return f"{violation_type} 的相關法規..."

跟 Gemini Function Calling 的關係：Tool 是框架封裝，底層邏輯一樣（LLM 選擇 → 你執行 → 送回）。

自訂 Tool 範例

實作

跟 Gemini 原生 FC 的差異：@tool 取代 docstring + type hint，LangChain 自動處理兩段式流程（不需手動 while loop）。

@tool
def check_regulation(violation_type: str) -> str:
    """查詢違規法規"""
    regulations = {
        "未戴安全帽": "職安法第6條...",
        "未穿反光背心": "營造安全衛生設施標準第11條..."
    }
    return regulations.get(violation_type, "查無相關法規")

@tool
def send_alert(message: str) -> str:
    """發送告警通知"""
    return f"已發送告警: {message}"

Workshop：Tool 實作

練習

LangChain Tool vs Gemini Function Calling 比較：

關鍵差異：

Gemini 原生：手動定義 docstring + 手動 while loop 處理兩段式 + 手動送回 function_response。

LangChain：@tool 裝飾器自動從 docstring 提取 schema + Agent 自動執行工具 + 自動處理結果。

LangChain 的價值：把 Gemini FC 那 20 行 while loop 簡化成 3 行。但你必須先懂原生 FC，才能 debug LangChain 出問題時發生了什麼。

LangChain Agent

Agent 概念

觀念

Agent = LLM + Tools + 決策循環

ReAct 模式的運作流程：

  • Thought（思考）：分析目前的狀況
  • Action（呼叫工具）：選擇並執行適當的工具
  • Observation（觀察結果）：接收工具回傳的結果
  • 重複：直到有最終答案為止
from langchain.agents import create_react_agent

agent = create_react_agent(model, tools)

Agent 實作

實作

Agent 自動決定用哪個工具、什麼順序。

from langchain.agents import create_react_agent

agent = create_react_agent(
    model,
    tools=[check_regulation, send_alert]
)

result = agent.invoke({
    "input": "A區有人沒戴安全帽"
})
# Agent 自動：
# 1. 思考 → 需要查法規
# 2. 呼叫 check_regulation("未戴安全帽")
# 3. 觀察結果 → 需要發通知
# 4. 呼叫 send_alert("A區違規...")
# 5. 回傳最終結果

Agent 的風險

觀念

Agent 雖然靈活，但在生產環境需要特別注意以下風險：

  • 不穩定：同一問題不同次可能選不同工具 / 不同順序
  • 成本高：每個決策都呼叫 LLM，Token 消耗大
  • 安全風險：Agent 可能呼叫不該呼叫的工具

緩解策略：

  • 限制可用工具數量與範圍
  • 設定最大步數（max_iterations）
  • 加入人工審核（Human-in-the-loop）
重要：Agent 適合探索性任務，不適合需要 100% 一致性的生產系統——生產系統用 Chain 更穩定。

LangChain RAG

LangChain RAG Pipeline

實作

LangChain 提供完整的 RAG 管線，從讀文件到生成回答一條龍。

# 1. 讀文件
from langchain.document_loaders import PyPDFLoader, CSVLoader, WebBaseLoader
loader = PyPDFLoader("safety_regulations.pdf")
docs = loader.load()

# 2. 切段
from langchain.text_splitter import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(docs)

# 3. 向量 DB
from langchain.vectorstores import Chroma  # 或 FAISS
vectorstore = Chroma.from_documents(chunks, embedding_model)

# 4. 檢索
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 3}
)

# 5. 生成
from langchain.chains import RetrievalQA
qa_chain = RetrievalQA.from_chain_type(
    llm=model,
    retriever=retriever
)

vs 手寫 RAG

觀念

兩種 RAG 實作方式的比較：

特性手寫 RAGLangChain RAG
相似度計算自己算 cosine similarity自動處理 embedding + 搜尋
Prompt 組裝自己組 prompt自動處理 prompt
優點靈活但繁瑣簡潔但黑盒
建議：先學手寫理解原理，再用 LangChain 提高效率。出問題時才知道該從哪裡 debug。

狀態與流程層概念

State 是什麼

觀念

LLM 沒有記憶——每次呼叫都是獨立的。系統需要 State 追蹤：

  • 累積的違規紀錄
  • 對話歷史
  • 流程進度
class SafetyState:
    violations: list    # 違規紀錄列表
    alert_count: int    # 告警次數

# Function Calling 結果存進 State
# → 系統有記憶

Chain vs Graph

觀念

從線性流程進化到有狀態的圖形結構：

  • Chain：A → B → C 固定流程
  • Graph：根據狀態決定下一步（可能 A → B → A → C）
  • LangGraph：有狀態的 Agent——每個節點是一個步驟，邊是條件判斷

適合：多步驟流程 / 需要回退 / 需要人工審核的系統。

語意角色 vs Chatbot

觀念

兩種不同的 AI 互動模式：

  • Chatbot：開放式聊天，難以控制品質
  • 語意角色（Semantic Role）：引導使用者完成特定流程（報稅 / 申請 / 檢核）

語意角色的系統設計：

  • 明確的狀態：使用者在哪一步
  • 明確的動作：下一步該做什麼
  • 明確的邊界：什麼時候需要人工介入

LangGraph 簡介 + State + Node

為什麼需要 LangGraph

觀念

LangChain 的 Chain 是線性的（A → B → C），不能分支、不能回頭、不能追蹤狀態。需求一複雜就卡住：

  • 需求：「高風險通知主管，低風險記錄就好，通知失敗要重試」
  • Chain 做不到：線性流程無法分流、無法迴圈
  • LangGraph 解法：把 Chain 升級為 Graph（狀態圖）
項目LangChainLangGraph
流程線性 A→B→C圖（可分支可迴圈）
狀態每步獨立State 跨步共享
分支不支援conditional_edges
迴圈不支援節點可指回自己

四個核心概念

觀念
概念說明類比
State狀態資料（全流程共享）共用白板
Node處理節點（一個函式）工作站
Edge節點間的連線輸送帶
Conditional Edge條件分支分岔路口

安裝：

pip install langgraph langchain-google-genai python-dotenv

Hello World：計數器

範例 
from langgraph.graph import StateGraph, START, END
from typing import TypedDict

# 1. 定義 State
class CounterState(TypedDict):
    count: int

# 2. 定義 Node（接 State → 回傳要更新的欄位）
def increment(state: CounterState) -> CounterState:
    return {"count": state["count"] + 1}

# 3. 建立 Graph
graph = StateGraph(CounterState)
graph.add_node("increment", increment)
graph.add_edge(START, "increment")
graph.add_edge("increment", END)

# 4. 編譯 + 執行
app = graph.compile()
result = app.invoke({"count": 0})
print(result)  # {"count": 1}
  • Node 不用回傳整個 State：只回傳要更新的欄位
  • State 是共用白板：所有 Node 讀寫同一份，不是 Node 之間直接傳值

Annotated 累加語法

技巧

普通欄位會覆蓋，Annotated[list, add] 會累加：

from typing import Annotated
from operator import add

class SafetyState(TypedDict):
    raw_event: str
    event_type: Optional[str]
    severity: Optional[str]
    messages: Annotated[list, add]   # ← 累加
    status: str
  • 普通欄位：Node A 寫 "parsed"、Node B 寫 "alerted" → 結果是 "alerted"
  • Annotated 累加：A 寫 ["分析完成"]、B 寫 ["告警發送"] → ["分析完成", "告警發送"]
  • 實用場景：記錄完整處理歷程

三種 Node

觀念
Node 類型做什麼場景
邏輯 Node純 Python 計算/判斷if-else、格式轉換
LLM Node呼叫 AI 模型語意理解、文字生成
Tool Node存取外部系統查 DB、呼叫 API
關鍵：每個 Node 只做一件事，透過 State 共用白板串起來。

條件分支：根據狀態選路線

路由函式（Router）

觀念

路由函式讀 State 決定走哪條路，回傳值用 Literal 限定可能路線：

from typing import Literal

def route_by_severity(state) -> Literal["high_handler", "low_handler"]:
    if state["severity"] == "高":
        return "high_handler"
    return "low_handler"
  • 只讀 State：路由函式不修改 State
  • Literal 型別：幫 IDE 和 LangGraph 知道有哪些可能路線

add_conditional_edges

範例 
graph.add_conditional_edges(
    "classify",           # 來源節點
    route_by_severity,    # 路由函式
    {
        "high_handler": "high_handler",  # 回傳值 → 節點
        "low_handler": "low_handler",
    }
)
graph.add_edge("high_handler", END)
graph.add_edge("low_handler", END)

第三個參數是對照表：路由函式回傳值 → 實際節點名稱。

完整流程示意

流程 
parse → classify → route ─┬─→ high_handler → 通知主管 → END
                          └─→ low_handler  → 記錄事件 → END

# no_helmet → 高風險 → 通知主管
# no_vest   → 低風險 → 記錄事件

多條件分支：可以同時考慮多個因素，例如「高風險 + 重試超過 2 次 → 緊急路線」。

迴圈重試：失敗了再來一次

迴圈 = 條件分支指回自己

觀念

LangGraph 沒有 for / while。迴圈就是讓條件分支的一條路指回來源節點：

send_notification ──→ should_retry ──→ send_notification（再來一次）
                              │
                              ├──→ finish（成功）
                              └──→ error（放棄）
重要：一定要有終止條件，否則無限迴圈！

重試實作範例

範例 
class RetryState(TypedDict):
    task: str
    retry_count: int
    success: bool
    messages: Annotated[list, add]

def should_retry(state) -> Literal["retry","done","give_up"]:
    if state.get("success"):
        return "done"
    if state["retry_count"] >= 3:
        return "give_up"
    return "retry"

graph.add_conditional_edges("send", should_retry, {
    "retry": "send",       # ← 指回自己 = 迴圈
    "done": "finish",
    "give_up": "error",
})
  • "retry": "send" 就是迴圈關鍵
  • retry_count >= 3 是終止條件

Human-in-the-Loop：人工審核

為什麼要暫停

觀念

自動化很好，但有些決策不能全自動——「高風險告警發出去之前，主管要先看過」。

parse → classify → 高風險 → [暫停] → 主管核准 → 發告警
                                          → 主管拒絕 → 結束
  • Checkpointer：把當前狀態存起來，之後可恢復
  • interrupt_before：指定暫停點
  • thread_id：識別「哪一次執行」，像客服工單編號

Checkpointer 設定

範例 
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()  # 正式系統用 Redis/PostgreSQL

app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["send_alert"],  # 在這個 Node 之前暫停
)

暫停 → 審核 → 繼續

流程 
config = {"configurable": {"thread_id": "violation-001"}}

# Step 1：執行到暫停點
result = app.invoke(
    {"raw_event": "no_helmet", "status": "pending", "messages": []},
    config=config,
)
# → 跑了 parse → classify → [暫停]
print(f"暫停於：{result['status']}")

# Step 2：人工審核（更新 State）
app.update_state(config, {
    "status": "approved",
    "messages": ["主管已核准"],
})

# Step 3：繼續執行（傳 None = 從暫停處繼續）
result = app.invoke(None, config=config)
# → 繼續跑 send_alert → END
  • app.invoke(None, config) — 傳 None 代表從暫停處繼續
  • app.update_state(config, {...}) — 人工修改 State
  • 不同事件用不同 thread_id：互不干擾
五大概念串起來：State（共用白板）→ Node（工作站）→ Edge（路線）→ Conditional Edge（分岔/迴圈）→ Checkpointer（暫停/恢復）。Multi-Agent 跟 MCP 都是建在這五個概念之上。

MCP 入門 — Model Context Protocol

MCP 是什麼

觀念

MCP 是一個標準協議，讓你寫的工具可以被任何 AI 助手呼叫。就像 USB 是「讓任何裝置都能插上任何電腦」的標準，MCP 是「讓任何工具都能被任何 LLM 使用」的標準。

  • USB 類比：鍵盤 / 滑鼠 / 隨身碟 ─ USB 協議 ─ 任何電腦
  • MCP 對應：你寫的工具 ─ MCP 協議 ─ 任何 AI 助手（Claude / OpenAI / Gemini）
  • 核心價值：工具寫一次，到處都能用
解決的痛點：過去每個 LLM 框架有自己的 tool 接法（OpenAI Function Calling、LangChain @tool、Gemini Function Declarations），同一個功能要寫三次。MCP 統一介面後寫一次就好。

兩個角色

觀念

MCP 架構非常簡單，只有兩端：

角色誰做什麼
Server你寫的程式提供工具（tool）給別人用
ClientAI 助手 / 測試程式連上 Server，呼叫工具

安裝：

pip install mcp
python -c "import mcp; print(mcp.__version__)"

最簡單的 Server

範例

server.py — 用 FastMCP 建立 Server，@mcp.tool() 把函式註冊成工具：

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("自我介紹")

@mcp.tool()
def get_name() -> str:
    """回傳我的名字。"""
    return "小明"

@mcp.tool()
def get_age() -> str:
    """回傳我的年齡。"""
    return "25 歲"

if __name__ == "__main__":
    mcp.run()  # stdio 模式
  • docstring 很重要：LLM 靠這段文字決定何時呼叫 tool
  • stdio 模式：透過 stdin/stdout 跟 Client 溝通，直接跑會看到空白（在等 Client 連線）

Client 呼叫流程

範例

client.py — Client 會自動啟動 Server 子程序，不用自己跑 python server.py：

import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def main():
    server_params = StdioServerParameters(command="python", args=["server.py"])
    async with stdio_client(server_params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()                # 握手
            tools = await session.list_tools()        # 列出可用工具
            result = await session.call_tool("get_name", arguments={})
            print(result.content[0].text)

asyncio.run(main())
  • stdio_client：啟動 Server 子程序、接管 stdin/stdout，回傳 read/write 管道
  • ClientSession：「翻譯員」，把請求轉成 MCP 協議格式（JSON-RPC）
  • 離開 async with：自動斷線、關閉 Server 子程序

MCP 概念與架構

三層角色

觀念

完整 MCP 架構有三層，一個 Host 可以連多個 Server：

角色是誰做什麼
HostClaude Desktop / Cursor / 你的 LangGraph App使用者面對的應用，內建 MCP Client
ClientHost 內部模組負責跟 Server 建立連線、傳送請求
Server你寫的 Python 程式暴露 tools / resources / prompts

MCP vs Function Calling

比較
項目Function CallingMCP
綁定對象特定模型 API任何支援 MCP 的 Host
定義方式每次 API call 帶 SchemaServer 啟動時註冊
運行位置你的 app 內獨立 process
可重用性換框架要重寫寫一次到處用

類比：Function Calling 是「每家餐廳自己的菜單格式」，MCP 是「全國統一的菜單標準」。

Server 提供的三種能力

觀念
能力說明類比
Tools可被 LLM 呼叫的函式像 Function Calling 的 function
Resources可被讀取的資料像檔案系統，LLM 可以「打開」來看
Prompts預定義的 prompt 模板像快捷鍵，選了就送出固定 prompt
  • 傳輸方式 1 — stdio：Host 直接啟動 Server process，本地開發 / Claude Desktop 適用
  • 傳輸方式 2 — SSE：Server 跑在某 URL，Client 透過 HTTP 連線，遠端部署 / 多人共用

MCP + LLM 混合：一個 Server 多種 AI

三步驟流程

流程

把 MCP Server 接上 LLM，讓 LLM 自己決定該呼叫哪個 tool：

  • Step 1：從 MCP Server 拿 tool 定義 → 轉成 LLM 看得懂的格式
  • Step 2：把 tool 定義 + 使用者問題 丟給 LLM
  • Step 3：LLM 決定呼叫哪個 tool → 透過 MCP 執行 → 拿回結果
關鍵：每個 LLM 的 tool 格式不同，但 MCP 給你的 tool 定義是統一的。只需要做一次「格式轉換」。Server 完全不用改，換 LLM 只改 Client。

LangGraph + MCP 整合範例

範例

MCPToolClient 包裝，讓 Node 可以非同步呼叫 MCP Server：

class MCPToolClient:
    def __init__(self, command, args):
        self.server_params = StdioServerParameters(command=command, args=args)

    async def call_tool(self, tool_name, arguments):
        async with stdio_client(self.server_params) as (read, write):
            async with ClientSession(read, write) as session:
                await session.initialize()
                result = await session.call_tool(tool_name, arguments=arguments)
                return result.content[0].text

# 在 Node 內呼叫
async def lookup_regulation(state: SafetyState) -> dict:
    result = await regulation_client.call_tool(
        "search_regulation",
        {"category": state["category"]},
    )
    return {"regulation": result, "messages": [f"📋 法規查詢完成"]}

# 因為 Node 是 async，要用 ainvoke 而非 invoke
result = await app.ainvoke({"category": "高處墜落", "messages": []})
  • ainvoke vs invoke：有 async Node 必須用 ainvoke
  • State 共享：MCP 查詢結果寫回 state，下游 Node 可讀取

LangGraph Multi-Agent：Supervisor 模式

三種 Multi-Agent 模式

觀念
模式適合場景複雜度
Supervisor任務可明確拆分、需統一決策⭐⭐
SwarmAgent 互相討論、邊界模糊⭐⭐⭐
Hierarchical大型系統、多層管理⭐⭐⭐⭐
  • 單一 Agent 痛點：prompt 越塞越多、context window 爆掉、出錯整個壞
  • Multi-Agent 優勢：專家做專家的事、各自管理 context、可並行

Supervisor 的職責

觀念

Supervisor 是一個特殊 Node，負責：

  • 看 State：現在的任務是什麼？
  • 選 Agent：誰最適合處理？
  • 收結果：Agent 做完，結果寫回 State
  • 判斷：還有事要做嗎？做完就 END
重點：Multi-Agent 不是新框架，就是把 LangGraph 的 State / Node / Conditional Edge 組合起來——每個 Agent 是一個 Node，Supervisor 是路由 Node。

Supervisor 架構範例

範例

State 設計：每個 Agent 寫自己的欄位，避免互相覆蓋；Supervisor 透過 next_agent 控制流程。

class SupervisorState(TypedDict):
    event: dict
    regulation_result: str        # 法規 Agent 寫
    report_result: str            # 報告 Agent 寫
    notification_result: str      # 通知 Agent 寫
    next_agent: str               # Supervisor 決策
    completed_agents: list[str]
    messages: Annotated[list, add_messages]

def supervisor_decide(state) -> dict:
    completed = state.get("completed_agents", [])
    if "regulation" not in completed:    next_agent = "regulation"
    elif "report" not in completed:      next_agent = "report"
    elif "notification" not in completed: next_agent = "notification"
    else:                                 next_agent = "done"
    return {"next_agent": next_agent}

# 建構 Graph
graph.add_node("supervisor", supervisor_decide)
graph.add_node("regulation", regulation_agent)
graph.add_node("report", report_agent)
graph.add_node("notification", notification_agent)
graph.add_edge(START, "supervisor")
graph.add_conditional_edges("supervisor", lambda s: s["next_agent"], {
    "regulation": "regulation",
    "report": "report",
    "notification": "notification",
    "done": END,
})
# 每個 Agent 完成後回到 supervisor
graph.add_edge("regulation", "supervisor")
graph.add_edge("report", "supervisor")
graph.add_edge("notification", "supervisor")
  • 一次 invoke：Supervisor 自動跑多輪（4 次決策 + 3 次 Agent 執行）
  • 不用手動控制順序：由 Supervisor 根據 completed_agents 決定下一步

from mcp.client.sse import sse_client

async with sse_client("http://localhost:8000/sse") as (read, write):

LangSmith：除錯與觀測

為什麼需要

痛點

Multi-Agent / Graph 跑出來的結果不對，但不知道哪一步出問題？print 大法不夠用——Supervisor 跑好幾輪、每輪不同 Agent，根本看不完。

LangSmith 是 LangChain 團隊的觀測平台，自動記錄 LLM 應用的每一步：

功能說明
Trace完整執行軌跡（每個 Node 的輸入/輸出）
Latency每一步花了多久
Token Usage每次 LLM 呼叫用了多少 token
Error Tracking哪一步出錯、錯誤訊息

零侵入設定

設定

Step 1 — 註冊 smith.langchain.com，免費方案每月 5000 次 trace

Step 2 — 取得 API Key（Settings → API Keys）

Step 3 — 加入 .env：

LANGSMITH_API_KEY="ls-your-api-key"
LANGSMITH_TRACING=true
LANGSMITH_PROJECT="hello-world"

Step 4 — 不用改程式碼！

load_dotenv()
result = app.invoke({...})
# 執行過程已自動送到 LangSmith
設計哲學：零侵入觀測。LangGraph 內部每次 invoke 自動檢查 LANGSMITH_TRACING 是否為 true，是就送 trace，不是就跳過。拿掉環境變數就完全不送任何資料。

整合思考題 + HW

練習

題目：設計一個「工地安全 AI 助手」的完整系統架構。

完整系統設計：

(1) 感知層：YOLO 偵測安全帽 / 背心 → Event JSON（class / confidence / bbox / timestamp）

(2) 模型層：Gemini 分析圖片內容（Structured Output 保證 JSON 格式）；Embedding 將法規轉向量存入向量 DB

(3) 行為層：Function Calling 定義 3 個工具（check_regulation / send_alert / log_incident）；RAG 搜尋相關法規段落塞進 prompt

(4) 編排層：用 Chain 固定流程（偵測 → 查法規 → 發告警 → 記錄）而非 Agent（生產系統要穩定）；每步結果存入 SafetyState

(5) 狀態層：SafetyState 追蹤累積違規 / 告警次數 / 處理歷史。N 次違規升級告警嚴重度。

(6) API 層：FastAPI 接收圖片（POST /detect）→ Schema 驗證 → Service 處理 → Guardrail 防呆 → 回傳決策

(7) 防護：System prompt 設定角色邊界 + 輸入過濾攻擊模式 + 輸出檢查敏感資訊

技術選型：YOLO（感知）+ Gemini（多模態分析 + FC）+ RAG（法規搜尋）+ FastAPI（API 入口）+ Pydantic（Schema 驗證）+ LangChain Chain（流程編排）。

核心原則：LLM 不直接操作系統，透過 Tool 間接使用；生產系統用 Chain 不用 Agent；Schema 驗證在 handler 前執行；每個決策都是事件，都要記錄。
