Git 基礎與版本控制

分散式版本控制、暫存區操作、commit、reset、restore 與 revert

Part 1: Git 核心概念

分散式版本控制系統

核心

Git 是分散式版本控制系統 (DVCS)，每位開發者都擁有完整的儲存庫副本，不需要隨時連線到中央伺服器。

四大工作區域
區域 說明 對應指令
Working Directory 本地工作目錄，實際編輯檔案的地方 直接編輯檔案
Staging Area 暫存區 (Index)，準備提交的檔案清單 git add
Local Repository 本地儲存庫，已提交的版本歷史 git commit
Remote Repository 遠端儲存庫 (GitHub, GitLab 等) git push / git pull 
# 工作流程
Working Directory  --git add-->  Staging Area  --git commit-->  Local Repo  --git push-->  Remote Repo

檔案狀態

基礎

Git 會追蹤每個檔案的狀態變化，理解狀態轉換是使用 Git 的基礎。

狀態 說明 如何辨識
Untracked 新建檔案，尚未被 Git 追蹤 git status 顯示紅色
Tracked - Unmodified 已追蹤且無變更 不會出現在 git status
Tracked - Modified 已追蹤但有修改，尚未暫存 Changes not staged
Staged 已加入暫存區，準備提交 Changes to be committed 
# 查看檔案狀態
git status

# 簡潔模式
git status -s
# ?? = Untracked, M = Modified, A = Added, D = Deleted

Part 2: git add 暫存區操作

git add 指令總覽

常用

將工作目錄的變更加入暫存區 (Staging Area)，為下一次 commit 做準備。

指令 功能 適用情境
git add <file> 加入指定檔案 精確控制要暫存的檔案
git add . 加入當前目錄下所有變更 (含新檔案) 快速暫存所有修改
git add -u 只加入已追蹤檔案的修改與刪除 (不含新檔案) 只更新已知檔案
git add -A 加入整個專案所有變更 (新增、修改、刪除) 全部暫存，與目錄位置無關
基本操作範例
# 暫存單一檔案
git add index.html

# 暫存多個檔案
git add index.html style.css app.js

# 暫存特定資料夾
git add src/

# 使用萬用字元
git add *.html
取消暫存
# 取消暫存指定檔案 (保留修改)
git restore --staged index.html

# 取消暫存所有檔案
git restore --staged .

# 舊版寫法 (仍可用)
git reset HEAD index.html
提示： git add . 只影響當前目錄及子目錄，而 git add -A 會影響整個專案。建議養成習慣使用 git status 確認後再 add。

Part 3: git commit 建立版本

git commit 指令

核心

將暫存區的變更永久記錄到本地儲存庫，建立一個新的版本快照。

指令 功能
git commit -m "訊息" 直接附帶訊息提交
git commit 開啟編輯器撰寫詳細訊息
git commit -am "訊息" 跳過 add，直接提交所有已追蹤且修改過的檔案
git commit --amend 修改最近一次 commit 的訊息或內容 
# 基本提交
git commit -m "feat: add login page"

# add + commit 合併 (僅限已追蹤檔案)
git commit -am "fix: resolve login bug"

# 修改上一次 commit 訊息
git commit --amend -m "fix: resolve login validation bug"

# 將遺漏的檔案加入上一次 commit
git add forgotten_file.js
git commit --amend --no-edit
注意： --amend 會改寫歷史紀錄，若已 push 到遠端，需要 git push --force，務必謹慎使用。

約定式提交 (Conventional Commits)

推薦

一套標準化的 commit 訊息格式，讓團隊協作更有效率，也方便自動產生 changelog。

格式
<type>(<scope>): <description>

# 範例
feat(auth): add Google OAuth login
fix(api): resolve timeout issue on /users endpoint
docs: update README with installation guide
Type 類型一覽
Type 說明 範例
feat 新功能 feat: add dark mode toggle
fix 修復 Bug fix: correct date format
docs 文件變更 docs: add API docs
style 格式調整 (不影響邏輯) style: fix indentation
refactor 重構 (非新功能、非修復) refactor: extract helper
test 測試相關 test: add unit tests for auth
chore 雜務 (build 設定、CI 等) chore: update dependencies
perf 效能優化 perf: lazy load images
ci CI/CD 設定變更 ci: add GitHub Actions
build 建置系統或外部依賴 build: upgrade webpack 5
revert 撤銷先前的 commit revert: revert "feat: ..."

Part 4: git status & git diff

git status 狀態檢視

基礎

顯示工作目錄與暫存區的目前狀態，是最常使用的 Git 指令之一。

# 完整狀態
git status

# 簡潔模式
git status -s
# 輸出範例：
#  M README.md       ← 已修改 (未暫存)
# M  index.html      ← 已修改 (已暫存)
# MM app.js          ← 暫存後又修改
# ?? new_file.txt    ← 未追蹤
# A  added.txt       ← 新增到暫存區
# D  deleted.txt     ← 已刪除
  • 第一欄：暫存區與上次 commit 的比較
  • 第二欄：工作目錄與暫存區的比較

git diff 差異比較

常用

檢視檔案變更的具體內容差異，幫助確認修改細節。

指令 比較對象
git diff 工作目錄 vs 暫存區 (尚未 add 的變更)
git diff --staged 暫存區 vs 最新 commit (已 add 尚未 commit)
git diff HEAD 工作目錄 vs 最新 commit (所有未提交的變更)
git diff <commit1> <commit2> 兩個 commit 之間的差異
git diff branch1..branch2 兩個分支之間的差異 
# 查看未暫存的修改
git diff

# 查看已暫存的修改
git diff --staged

# 只看檔案名稱 (不看內容)
git diff --name-only

# 看統計摘要
git diff --stat

Part 5: git reset 重置

三種模式比較

重要

git reset 用於將 HEAD 指標移回先前的 commit，三種模式對暫存區和工作目錄的影響不同。

模式 HEAD Staging Area Working Directory 用途
--soft 移動 保留 保留 重新編輯 commit 訊息、合併多個 commit
--mixed (預設) 移動 清除 保留 重新選擇要暫存的檔案
--hard 移動 清除 清除 完全捨棄變更，回到指定版本
危險： git reset --hard 會永久刪除工作目錄中未提交的變更，無法復原！使用前務必確認。

HEAD 是什麼？

概念

HEAD 是一個指標，指向目前所在的 commit (通常是當前分支的最新 commit)。

語法 意義
HEAD 目前所在的 commit
HEAD~1 或 HEAD~ 上一個 commit
HEAD~2 往前兩個 commit
HEAD~n 往前 n 個 commit
ORIG_HEAD 執行 reset / merge 前 HEAD 的位置 (安全繩) 
# 查看 HEAD 指向
git log --oneline -5

# 查看 ORIG_HEAD (reset 前的位置)
git reflog

使用範例

實作

各模式的實際使用情境與指令。

--soft：保留所有變更在暫存區
# 回到上一個 commit，變更留在暫存區
git reset --soft HEAD~1

# 用途：合併最近 3 個 commit
git reset --soft HEAD~3
git commit -m "feat: combine three commits"
--mixed：保留變更在工作目錄 (預設)
# 回到上一個 commit，變更留在工作目錄
git reset HEAD~1
# 等同於 git reset --mixed HEAD~1

# 用途：重新選擇要加入暫存區的檔案
git add specific_file.js
git commit -m "feat: only commit selected file"
--hard：完全回到指定版本
# 完全回到上一個 commit (捨棄所有變更)
git reset --hard HEAD~1

# 回到指定 commit hash
git reset --hard a1b2c3d

# 誤用 --hard 的補救：用 ORIG_HEAD
git reset --hard ORIG_HEAD

Part 6: git restore 還原

git restore vs git checkout

背景

Git 2.23 版 (2019) 將 git checkout 的功能拆分為兩個更明確的指令。

舊指令 (checkout) 新指令 用途
git checkout -- <file> git restore <file> 還原檔案內容
git checkout <branch> git switch <branch> 切換分支
建議： 優先使用 git restore 和 git switch，語意更清楚，不容易混淆。

git restore 指令

常用

用於還原工作目錄或暫存區的檔案到指定狀態。

指令 功能
git restore <file> 捨棄工作目錄的修改 (還原到暫存區/最新 commit 狀態)
git restore --staged <file> 將檔案從暫存區移除 (保留工作目錄修改)
git restore --source=<commit> <file> 將檔案還原到指定 commit 的版本 
# 捨棄工作目錄中的修改
git restore index.html

# 捨棄所有未暫存的修改
git restore .

# 取消暫存 (保留修改)
git restore --staged index.html

# 還原到特定 commit 的版本
git restore --source=HEAD~2 index.html

# 同時還原暫存區和工作目錄
git restore --staged --worktree index.html
危險： git restore <file> 會永久捨棄工作目錄中未暫存的修改，無法用 Git 復原！與 git reset --hard 類似的風險。

Part 7: git revert 撤銷

git revert vs git reset

重要比較

兩者都能「回到過去」，但方式截然不同。

比較項目 git reset git revert
原理 移動 HEAD 指標，刪除 commit 建立新的 commit 來反轉變更
歷史紀錄 會改寫歷史 保留完整歷史
適合場景 本地端尚未 push 的 commit 已 push 到遠端的 commit
團隊協作 可能造成衝突 (需 force push) 安全，不影響其他人
可追溯性 撤銷的 commit 從 log 消失 撤銷紀錄清楚可見
原則： 已推送到遠端的 commit 一律用 git revert；僅在本地的 commit 才考慮用 git reset。

git revert 指令

實作

建立一個新的 commit，內容是指定 commit 的反向操作。

指令 功能
git revert HEAD 撤銷最新的 commit
git revert <hash> 撤銷指定的 commit
git revert -n HEAD~3..HEAD 撤銷最近 3 個 commit (不自動 commit)
git revert --continue 解決衝突後繼續 revert 
# 撤銷最新 commit
git revert HEAD

# 撤銷指定 commit
git revert a1b2c3d

# 撤銷多個 commit (不自動產生各別 commit)
git revert -n HEAD~3..HEAD
git commit -m "revert: undo last 3 commits"

# 撤銷但不自動 commit
git revert --no-commit HEAD

衝突處理步驟

注意

執行 git revert 時可能遇到衝突，需要手動解決。

處理流程
  • Step 1：執行 git revert <commit> 後出現 CONFLICT
  • Step 2：用 git status 查看衝突檔案 (標記為 both modified)
  • Step 3：開啟衝突檔案，手動編輯解決衝突標記
  • Step 4：用 git add <file> 標記衝突已解決
  • Step 5：用 git revert --continue 完成 revert
# 衝突標記格式
<<<<<<< HEAD
目前版本的內容
=======
revert 想回復的內容
>>>>>>> parent of a1b2c3d

# 解決後
git add conflicted_file.js
git revert --continue

# 如果想放棄 revert
git revert --abort
