Atlas Dashboard 視覺化（延伸）

Atlas Charts 功能介紹

觀念

Atlas Charts 可將資料庫資料直接轉換為圖表，無需撰寫任何程式碼，適合快速視覺化與管理報告。

  • 適用場景：管理階層數據報告、監控評分變化、展示資料分布
  • 操作流程：Atlas → Charts → Activate Now → Create Dashboard
  • 建立「美食地圖儀表板」Dashboard
  • Add Chart → 選擇資料來源 food_map.restaurants_raw
  • Charts 會自動偵測欄位類型，提供適合的圖表建議

建立圖表實作

實作

實作四種常用圖表類型，從不同角度分析餐廳資料。

  • 長條圖（各類型餐廳數量）：X 軸 = 類型，Y 軸 = COUNT，快速看出哪種餐廳最多
  • 圓餅圖（價格區間分布）：展示 $/$$/$$$ 各佔比例，了解市場定位
  • 散佈圖（評分 vs 評論數）：觀察評分與評論數量的關聯性
  • 地理熱力圖（餐廳位置分布）：利用 GeoJSON 座標在地圖上顯示餐廳密集區域

練習：Dashboard 設計

練習

設計一個完整的美食地圖儀表板，包含至少四種不同類型的圖表。

優秀回答：建立了 4 種圖表：(1) 長條圖顯示各類型餐廳數量——台式料理最多，咖啡廳次之 (2) 圓餅圖顯示價格區間分布——$$ 佔 60% 是主流 (3) 散佈圖發現高評分不一定高評論數，川富餐廳評分 4.5 但評論 256 則，森林咖啡評分 4.0 只有 89 則 (4) Dashboard 可分享給團隊，資料更新時圖表自動反映。Dashboard 適合快速原型和管理報告，但複雜分析需要 Python + 聚合管道。圖表選擇原則：比較用長條圖、佔比用圓餅、關係用散佈、趨勢用折線。

Python PyMongo 基礎

PyMongo 連線設定

實作

透過 PyMongo 套件從 Python 連線 MongoDB Atlas，建立資料操作的基礎環境。

  • 安裝套件：pip install pymongo
  • 建立連線：from pymongo import MongoClient
  • 連線字串：client = MongoClient("mongodb+srv://user:pass@cluster.mongodb.net/")
  • 選擇資料庫：db = client["food_map"]
  • 選擇集合：collection = db["restaurants_raw"]
注意：連線字串應存在環境變數或 .env 檔案中，不要寫死在程式碼裡！密碼中若有特殊字元（如 @、#）需進行 URL encode 處理。

Python CRUD 操作

實作

PyMongo 的 CRUD 方法與 mongosh 語法高度對應，但有 Python 特有的命名規範。

操作方法
新增單筆insert_one(dict)
新增多筆insert_many([list])
查詢多筆find(filter)
查詢單筆find_one(filter)
更新單筆update_one(filter, {"$set": {...}})
更新多筆update_many(filter, {"$set": {...}})
刪除單筆delete_one(filter)
刪除多筆delete_many(filter)
  • Python 使用 dict {} 而非 JSON 格式
  • 方法名使用底線命名：insert_one 而非 insertOne
  • find() 回傳 Cursor 物件，需用 for 迴圈或 list() 取出資料

練習：PyMongo CRUD

練習

使用 PyMongo 對 restaurants_raw 集合進行完整的 CRUD 操作練習。

優秀回答：PyMongo 與 mongosh 語法幾乎一致但有差異：(1) Python 用 dict 而非 JSON (2) 方法名用底線如 insert_one 而非 insertOne (3) find() 回傳 Cursor 物件需用 for 迴圈取出。範例：for doc in collection.find({'評分': {'$gt': 4}}): print(doc['店名']) 列出所有評分 > 4 的餐廳。insert_many 比逐筆 insert_one 效率高。update_one 要記得用 $set 否則整份文件被替換。連線字串最佳實踐：import os; uri = os.environ.get('MONGO_URI') 從環境變數讀取。

Python 聚合管道

在 Python 中執行聚合

實作

Python 的聚合管道就是 list of dict，與 mongosh 寫法直接對應，無需額外學習語法。

  • 呼叫方式：collection.aggregate([pipeline])
  • pipeline 是 list of dict，每個 dict 代表一個聚合階段
  • 範例 — 統計各類型平均評分：
pipeline = [
    {"$group": {"_id": "$類型", "avg": {"$avg": "$評分"}}},
    {"$sort": {"avg": -1}}
]
results = collection.aggregate(pipeline)
for doc in results:
    print(f"{doc['_id']}: {doc['avg']:.1f}")

Python 地理位置查詢

實作

結合 MongoDB 地理查詢與 Python folium 套件，在地圖上視覺化附近餐廳。

  • $near 查詢附近 1 公里內餐廳：
nearby = collection.find({
    "位置": {
        "$near": {
            "$geometry": {
                "type": "Point",
                "coordinates": [121.5398, 25.0173]
            },
            "$maxDistance": 1000
        }
    }
})
  • 搭配 folium 套件將結果繪製在互動式地圖上
  • 每筆結果用 folium.Marker 標記位置與餐廳資訊

練習：Python 聚合 + 地理查詢

練習

綜合運用 Python 聚合管道與地理位置查詢，實作餐廳分析與地圖視覺化。

優秀回答：Python 聚合管道就是 list of dict，與 mongosh 語法完全對應。統計各類型餐廳：pipeline = [{'$group': {'_id': '$類型', '平均評分': {'$avg': '$評分'}, '數量': {'$sum': 1}}}, {'$sort': {'平均評分': -1}}]。地理查詢配合 folium 可以在地圖上標示附近餐廳：import folium; m = folium.Map(location=[25.0173, 121.5398])；對每筆結果 folium.Marker([lat, lng]).add_to(m)。進階：用 $geoNear 聚合階段可同時取得距離值，方便做距離排序的推薦系統。

批次資料處理

大量資料匯入

實作

處理 20000+ 筆政府開放資料的實際匯入流程，掌握三種主要匯入方式。

  • Compass Import Data：一鍵匯入 CSV/JSON 檔案，最簡單直覺
  • Python 批次插入：讀取 CSV → 轉 dict → insert_many 批次插入
  • mongoimport 命令列工具：快速匯入大量資料，適合自動化腳本
  • 實際處理 20000+ 筆政府開放資料的完整流程

Python 批次資料增強

實作

用 Python 邏輯批次為文件添加計算欄位，比手動逐筆操作效率高 100 倍。

  • 讀取 restaurants_raw 全部文件
  • 用 Python 邏輯判斷添加類型、評分等欄位
  • 批次 update_many 更新所有符合條件的文件
  • 範例：地址包含「大安區」→ 自動標記區域欄位
  • 比手動逐筆操作效率高 100 倍

練習：批次處理

練習

比較三種批次處理方式的優劣，並實作 Python 批次資料增強。

優秀回答：批次處理三種方式比較：(1) Compass Import Data——最簡單，一鍵匯入 CSV，適合初次匯入 (2) Python insert_many——可在匯入前用 Python 邏輯預處理資料（清洗/轉換/增強） (3) mongoimport——命令列工具，適合自動化腳本和 CI/CD。Python 批次增強範例：for doc in collection.find(): if '大安區' in doc.get('商業地址',''): collection.update_one({'_id':doc['_id']}, {'$set':{'區域':'大安區'}})。但更好的做法是用 update_many 一次處理：collection.update_many({'商業地址': {'$regex': '大安區'}}, {'$set': {'區域': '大安區'}})——一行搞定。

本地建模預測

技術架構

觀念

捨棄雲端 AutoML 方案，改用 Scikit-learn 本地訓練，將 2 小時雲端流程縮短至 1 秒完成。

  • 雲端 AutoML 訓練需要約 2 小時等待時間
  • 本地 Scikit-learn 訓練僅需 1 秒——速度差 7200 倍
  • 無需等待雲端資源配置與模型部署
  • 適合教學場景與快速迭代驗證

關鍵技術流程

觀念

從特徵工程到模型持久化的完整機器學習流程。

  • 特徵工程：One-Hot Encoding 處理類別型資料（如行政區），將「大安區」轉化為 [1,0,0,...] 向量矩陣
  • 演算法選擇：隨機森林回歸（Random Forest Regressor），集成學習處理非線性關係
  • 模型持久化：joblib 序列化為 .pkl 檔案，一次訓練多次調用
  • Random Forest 優勢：不需大量調參、訓練快速、能處理非線性關係（如房價與坪數非簡單正比）

練習：ML 整合思考

練習

比較本地建模與雲端 AutoML 的優劣，分析 Random Forest 與 One-Hot Encoding 的選擇理由。

優秀回答：本地建模 vs 雲端 AutoML：(1) 本地 Scikit-learn 訓練 1 秒完成，雲端 AutoML 需 2 小時——速度差 7200 倍 (2) 本地模型駐留記憶體預測延遲 <1ms，雲端 REST API 需 100-500ms 網路延遲 (3) 本地無閒置費用，雲端 Endpoint 持續計費 (4) 但雲端適合大型模型和團隊協作。Random Forest 選擇理由：集成學習處理非線性關係（房價與坪數不是簡單正比）、不需大量調參、訓練快。One-Hot Encoding 將「大安區」變成 [1,0,0,...] 向量，讓數學模型理解類別差異。joblib 序列化讓模型可重複使用——訓練一次，部署多處。

MongoDB vs SQL 最終比較

觀念

經過三天深入學習後，全面比較 MongoDB 與傳統 SQL 資料庫的差異與適用場景。

比較項目MongoDBSQL 資料庫
資料模型文件導向（Document）表格（Table）
Schema彈性 Schema固定 Schema
擴展方式水平分片（Sharding）垂直擴展
查詢語法JSON 運算子SQL 語法
適用場景快速迭代 / 地理查詢 / 嵌套資料事務一致性 / 複雜 JOIN

整合思考題

練習

假設要建立一個「即時美食推薦 APP」，從資料庫設計到前端展示的完整方案。

優秀回答：完整方案設計：(1) 資料庫：MongoDB Atlas M0 免費方案，food_map 資料庫含 restaurants（嵌入式設計：基本資訊 + 位置 GeoJSON + 評分 + 推薦菜色陣列 + 評論子文件）和 users（收藏清單 + 搜尋歷史）。(2) 資料來源：政府開放資料 20000+ 筆用 Python 批次匯入 + 增強（添加店名/類型/評分）。(3) 後端 API：Python FastAPI + PyMongo，核心端點：GET /restaurants/nearby?lat=25&lng=121&radius=1000（用 $near + $match 組合查詢）、GET /restaurants/top（聚合管道 $group + $sort）、POST /reviews（用 $push 添加評論 + $inc 增加評論數）。(4) 前端：React/Vue + Leaflet.js 地圖，搜尋框 + 地圖 + 餐廳卡片列表。(5) 部署：前端 Cloud Storage 靜態託管，後端 Cloud Run 無伺服器，MongoDB Atlas 雲端。(6) 成本：Atlas 免費 + Cloud Run 閒置 $0，預估月費 <$5。
