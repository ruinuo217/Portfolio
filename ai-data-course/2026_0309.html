MongoDB 基礎與 CRUD

3/9 課程筆記 — MongoDB 文件模型、Atlas 雲端環境、Compass 操作、CRUD 四大操作、資料增強、進階查詢

MongoDB 簡介

什麼是 MongoDB

觀念

文件導向（Document-Oriented）NoSQL 資料庫，以靈活的文件模型取代傳統關聯式表格。

  • BSON 格式儲存：Binary JSON，支援更多資料型別（日期、二進位等）
  • 彈性 Schema：不需預先定義欄位，同一集合中的文件可有不同結構
  • 水平擴展：分散式架構 + 自動分片（Sharding），輕鬆應對大量資料
  • 豐富查詢：支援聚合管線（Aggregation Pipeline）、地理位置查詢等進階功能

SQL vs MongoDB 術語對照

觀念

從 SQL 世界過渡到 MongoDB，首先要對照核心術語的差異。

SQL 術語MongoDB 術語
DatabaseDatabase
TableCollection
RowDocument
ColumnField
Primary Key_id
IndexIndex
JOIN嵌入文件或參照（$lookup）
  • 資料結構：SQL 固定欄位 vs MongoDB 彈性欄位
  • 關聯處理：SQL 外鍵 + JOIN vs MongoDB 嵌入文件 / 參照
  • Schema 設計：SQL 必須 CREATE TABLE vs MongoDB 直接插入

文件模型概念

觀念

SQL 以表格（行列）儲存資料，MongoDB 以 JSON-like 文件儲存，每筆文件是一個獨立物件。

  • 文件範例（台北市政府餐廳資料）：
  • _id：自動產生的 ObjectId
  • 統一編號：唯一識別碼
  • 商業名稱：公司名稱
  • 商業地址：完整地址
  • Longitude / Latitude：經緯度，便於地圖查詢
觀察：政府真實資料中，統一編號為唯一識別碼，經緯度便於地圖查詢，_id 由 MongoDB 自動產生。

適用場景

觀念

MongoDB 與 SQL 各有擅長的應用場景，選擇取決於資料特性與需求。

MongoDB 適合SQL 更適合
資料結構多變結構固定
階層嵌套資料複雜多表 JOIN
水平擴展需求ACID 保證（財務/訂單）
地理位置查詢嚴格資料完整性
高寫入場景交易一致性要求高
本課程案例：台北市餐廳地圖（20,000+ 筆開放資料），資料結構多變且需要地理位置查詢，非常適合 MongoDB。

練習：MongoDB 概念

練習

思考 MongoDB Collection 與 SQL Table 的差異，以及校園美食地圖為何適合 MongoDB。

優秀回答：MongoDB 的 Collection 與 SQL 的 Table 主要差異：(1) Collection 不需要預先定義 Schema，同一個 Collection 裡的文件可以有不同欄位，而 Table 的每一列必須遵循相同結構。(2) Collection 的文件可以包含嵌套物件和陣列（如評論列表），SQL 需要建立額外的表格再用 JOIN 關聯。(3) Collection 新增欄位不需要 ALTER TABLE，直接插入含新欄位的文件即可。校園美食地圖適合 MongoDB：資料結構多變（不同餐廳欄位不同）、需要地理位置查詢（附近餐廳）、需要快速迭代開發。

環境建立 — MongoDB Atlas

Atlas 雲端環境優勢

觀念

MongoDB Atlas 是官方提供的雲端資料庫服務，大幅降低學習門檻。

  • 免安裝：不需在本機安裝 MongoDB Server
  • 跨平台一致：Windows / Mac / Linux 都能使用相同環境
  • 免費方案：M0 免費提供 512MB 儲存空間
  • 內建監控備份圖表：自動備份、效能監控、視覺化圖表

註冊與建立叢集

實作

從零開始建立 Atlas 雲端叢集，約 3-5 分鐘即可完成。

  • 註冊 Atlas 帳號（可用 Google 登入）
  • 驗證 Email
  • 選擇 Shared 免費方案
  • 雲端供應商選 GCP asia-east1（台灣）
  • 方案選 M0 免費（512MB）
  • 叢集命名為 Cluster0
  • 等待 3-5 分鐘部署完成

安全設定

實作

設定資料庫存取權限與網路白名單，取得連線字串。

  • Database Access：建立使用者 admin / Password / Read and write to any database
  • Network Access：Allow Access from Anywhere（0.0.0.0/0）
  • 取得連線字串：mongodb+srv://admin:<YOUR_PASSWORD>@cluster0.<CLUSTER_ID>.mongodb.net/
注意：務必記下密碼！生產環境應限制 IP，不應使用 0.0.0.0/0。

CRUD — 新增操作 (Insert)

insertOne 單筆新增

實作

透過 Compass 的 ADD DATA → Insert Document 插入單筆文件。

  • 範例：統一編號 00000624 / 烹派餐飲有限公司 / 南港區 / 座標
  • _id（ObjectId）：MongoDB 自動產生，由時間戳 + 機器 ID + 程序 ID + 計數器組成
  • ObjectId 可反推文件建立的大致時間

insertMany 批次新增

實作

使用陣列格式 [{...}, {...}] 一次新增多筆文件。

  • 範例：一次新增 3 筆（川富 / 晙碩 / 欣欣韓友）
  • 文件最大 16MB
  • 避免使用 $ 和 . 開頭的欄位名稱
  • _id 重複會導致插入失敗

練習：新增資料

練習

比較 insertOne 與 insertMany 的差異，理解 ObjectId 的結構。

優秀回答：使用 insertMany 批次新增比逐筆 insertOne 更有效率，減少網路往返次數。MongoDB 自動產生的 _id (ObjectId) 包含時間戳記，所以可以大致知道文件的建立時間。中文欄位名稱在 MongoDB 完全支援，與政府開放資料欄位保持一致有助於資料對應。insertMany 預設 ordered:true，遇到錯誤會停止；設 ordered:false 可跳過錯誤繼續插入其餘文件。

CRUD — 查詢操作 (Find)

find 基本查詢

實作

在 Compass 的 Filter 欄位輸入查詢條件，篩選符合的文件。

  • {} — 查詢全部文件
  • {"商業名稱":"烹派餐飲有限公司"} — 精確查詢
  • {"統一編號":"00000624"} — 按統編查詢
  • {"Longitude":{"$gt":121.55}} — 條件查詢
比較運算子意義範例
$gt大於{"age":{"$gt":25}}
$gte大於等於{"age":{"$gte":25}}
$lt小於{"age":{"$lt":25}}
$lte小於等於{"age":{"$lte":25}}
$eq等於{"age":{"$eq":25}}
$ne不等於{"age":{"$ne":25}}

投影、排序、分頁

實作

控制查詢結果的欄位、順序與數量。

  • Project（投影）：{"商業名稱":1, "商業地址":1, "_id":0} — 選擇顯示欄位
  • Sort（排序）：{"Longitude":1} 升序 / {"Longitude":-1} 降序
  • Limit + Skip（分頁）：第 2 頁每頁 10 筆 → Limit=10, Skip=10

CRUD — 更新操作 (Update)

updateOne 與 $set

實作

更新單筆文件，可在 Compass 中點編輯圖示直接修改，或使用程式化方式。

  • Compass：點擊編輯圖示直接修改欄位值
  • 程式化：updateOne(filter, {"$set": {...}})
重要：務必使用 $set！不用 $set 整個文件會被替換，其他欄位將遺失。

updateMany 批次更新

實作

一次更新所有符合條件的文件，適合批次資料處理。

  • 範例：為所有大安區餐廳新增「區域」欄位
  • updateMany({"商業地址":{"$regex":"大安區"}}, {"$set":{"區域":"大安區"}})
  • 結果：回傳 matchedCount（符合筆數）/ modifiedCount（實際修改筆數）

CRUD — 刪除操作 (Delete)

deleteOne 與 deleteMany

實作

刪除文件或整個集合，操作不可逆。

  • Compass：點擊垃圾桶圖示刪除單筆
  • deleteMany({}) — 清空整個集合（危險！）
  • drop() — 刪除整個集合
警告：刪除無法復原！刪除前務必先用 find() 確認要刪除的文件。

練習：CRUD 綜合

練習

整理 CRUD 四大操作的要點與 SQL 對照。

優秀回答：CRUD 操作要點：(1) Insert: insertMany 比逐筆更有效率 (2) Find: 比較運算子 $gt/$lt 用於數值篩選，$regex 用於模糊查詢 (3) Update: 務必用 $set 避免文件被替換，updateMany 批次處理效率高 (4) Delete: 刪除前一定先 find 確認。SQL vs MongoDB 對照：INSERT INTO → insertOne/Many, SELECT WHERE → find(filter), UPDATE SET WHERE → updateOne/Many + $set, DELETE WHERE → deleteOne/Many。MongoDB 優勢：不需 ALTER TABLE 就能新增欄位，彈性 Schema 讓資料增強更容易。

SQL vs MongoDB 操作對比

完整對照表

觀念

SQL 與 MongoDB 的 CRUD 操作完整對應，方便從 SQL 背景過渡。

操作SQLMongoDB
新增單筆INSERT INTO table VALUES(...)insertOne({...})
新增多筆INSERT INTO table VALUES(...),(...)insertMany([{...},{...}])
查全部SELECT * FROM tablefind({})
條件查詢SELECT * WHERE col = valfind({field: value})
排序ORDER BY col ASC/DESCsort({field: 1/-1})
限制筆數LIMIT nlimit(n)
更新單筆UPDATE SET col=val WHERE...updateOne(filter, {$set:{...}})
更新多筆UPDATE SET col=val WHERE...updateMany(filter, {$set:{...}})
刪除單筆DELETE FROM WHERE...deleteOne(filter)
刪除多筆DELETE FROM WHERE...deleteMany(filter)
  • 不需預定義 Schema：MongoDB 不需要 CREATE TABLE，直接插入即可
  • 更新需用運算子：必須使用 $set 等運算子，否則整份文件被替換
  • 新增欄位非常容易：直接在 update 中加入新欄位，不需 ALTER TABLE

真實資料處理與增強（Unit 2上）

政府開放資料 vs 應用需求

觀念

政府開放資料提供基礎欄位，但消費者導向應用需要更多資訊，因此需要「資料增強」（Data Enrichment）。

欄位政府資料應用需求
商業名稱豪派餐飲有限公司需要消費者友好店名
分類無需要餐廳類型
評價無需要評分 / 評論
價格無需要價格區間
營業資訊無需要營業時間 / 推薦菜色
地理資訊✅ 完整可直接使用

資料增強實作流程

實作

從 5 個欄位增強到 14 個欄位，讓原始政府資料變成消費者友好的應用資料。

  • 步驟 1：匯入 5 筆原始資料到 restaurants_raw
  • 步驟 2：新增「店名」欄位（Compass 逐筆編輯）
  • 步驟 3：批次新增「類型」（mongosh updateOne）
  • 步驟 4：批次新增「評分」「評論數」（updateMany 先設預設值再逐筆更新）
  • 步驟 5：新增「價格區間」「營業時間」「推薦菜色」「特色標籤」（陣列欄位）
  • 步驟 6：用 $push 新增「評論」（嵌入式文件陣列）
  • 步驟 7：驗證增強結果（從 5 欄位 → 14 欄位）

關鍵運算子

觀念

資料增強過程中常用的 MongoDB 更新運算子。

運算子功能範例
$set新增 / 修改欄位{"$set": {"店名": "好吃小館"}}
$push新增元素到陣列{"$push": {"評論": {內容}}}
$inc增加數值{"$inc": {"評論數": 1}}
$pull從陣列移除元素{"$pull": {"標籤": "過期"}}

進階查詢

比較與邏輯運算子

觀念

MongoDB 提供豐富的比較與邏輯運算子，對應 SQL 的 WHERE 條件。

類型運算子說明SQL 對應
比較$eq等於=
$ne不等於!=
$gt大於>
$gte大於等於>=
$lt小於<
$lte小於等於<=
邏輯$and多條件並列（預設 AND）AND
$or任一符合OR
$not反轉條件NOT
$nor都不符合NOT ... AND NOT ...
包含$in包含在列表中IN
$nin不包含在列表中NOT IN

正則表達式查詢

觀念

使用 $regex 進行模糊查詢，類似 SQL 的 LIKE。

  • {"店名":{"$regex":"小館"}} — 找含「小館」的店名
  • {"商業地址":{"$regex":"大安區"}} — 找大安區的餐廳
  • $options: "i" — 不區分大小寫
模式意義範例
^開頭匹配{"店名":{"$regex":"^好"}}
$結尾匹配{"店名":{"$regex":"小館$"}}
.*任意字元{"店名":{"$regex":"好.*館"}}

陣列與嵌入文件查詢

觀念

MongoDB 的文件可包含陣列與嵌入文件，查詢語法也相應豐富。

  • 陣列包含：{"特色標籤":"外帶"} — 陣列中是否含有該值
  • $all 同時包含：{"特色標籤":{"$all":["內用","外帶"]}}
  • $size 陣列長度：{"推薦菜色":{"$size":3}}
  • $elemMatch 複雜條件：{"評論":{"$elemMatch":{"評分":5}}}
  • 點記法查嵌入文件：{"座標.Longitude":{"$gt":121.54}}
  • $exists 檢查欄位存在：{"評論":{"$exists":true}}

練習：進階查詢

練習

整理進階查詢的技巧與常見錯誤。

優秀回答：進階查詢技巧整理：(1) 多條件預設是 AND，不需明確寫 $and；$or 需要明確寫 (2) $in 類似 SQL 的 IN 運算子，查詢多個值之一 (3) $regex 用於字串模糊查詢，$options:'i' 不區分大小寫 (4) 陣列查詢：直接寫值會搜尋陣列是否包含該值，$all 要求同時包含所有值 (5) $elemMatch 用於陣列元素是物件時的複合條件查詢 (6) 點記法 'parent.child' 查詢嵌入文件，整個路徑要用引號包起來 (7) $exists 在彈性 Schema 環境特別有用，因為不是所有文件都有相同欄位。常見錯誤：運算子忘記 $ 開頭、$in 值不是陣列、$regex 不能用在數字欄位。

錯誤排查指南

觀念

初學 MongoDB 時最容易犯的錯誤與解決方式。

常見錯誤錯誤寫法正確寫法
運算子缺 ${"age":{"gt":25}}{"age":{"$gt":25}}
點記法缺引號{座標.Longitude:121}{"座標.Longitude":121}

$regex 用在數字欄位{"age":{"$regex":"2"}}改用 $gt/$lt 等比較運算子
$set 忘記用updateOne(f, {"name":"X"})updateOne(f, {"$set":{"name":"X"}})
