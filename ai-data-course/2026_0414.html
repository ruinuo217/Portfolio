協作開發與虛擬化技術 III：ArgoCD 與 GitOps CI/CD

ArgoCD 安裝與設定、GitOps 概念、Sync / Auto-Sync / Self-Heal、GitHub Actions CI、CI/CD 完整串接、Git Rollback

Ch1 — 為什麼需要 ArgoCD

情境：傳統部署的痛點

觀念

團隊有 5 個工程師，K8S 上跑了 3 個服務，目前的部署流程是：

  1. 改程式碼
  2. docker build + docker push
  3. 改 Deployment YAML 的 image tag
  4. kubectl apply -f ...

問題開始出現：

  • 小明昨天部署了什麼版本？沒人知道
  • 小華在 prod 上手動 kubectl scale 改了 replicas，YAML 沒更新
  • 出事要退版，沒人記得上一版的 YAML 長什麼樣
  • 主管問「現在線上跑的是哪個版本」，你要登到機器上 kubectl describe 才知道
核心問題：問題不在 K8S，在於沒有一個「唯一的真相來源」(Single Source of Truth)。

GitOps 是什麼

觀念

一句話：Git repo 就是系統的唯一真相來源。

K8S 上跑的狀態，必須跟 Git repo 裡的 YAML 一致。有差異就是錯的，要修正。

  • 傳統部署（Push 模式）：開發者 → kubectl apply → K8S
    問題：誰 apply 的？apply 了什麼？沒有記錄。
  • GitOps（Pull 模式）：開發者 → git push → Git repo ← ArgoCD 持續監控 → K8S
    所有變更都在 Git 裡，有 commit、有 diff、有 blame。

ArgoCD 的角色

觀念

ArgoCD 是一個跑在 K8S 裡的 GitOps 工具。它做一件事：

持續比對 Git repo 裡的 YAML 和 K8S 上的實際狀態，發現差異就同步。

概念傳統 kubectlArgoCD
部署方式人手動 applyArgoCD 自動從 Git 拉
部署記錄沒有（除非你自己記）Git commit history
退版kubectl rollout undo（記在 K8S）git revert（記在 Git）
多人協作誰最後 apply 誰贏PR review → merge → 自動部署
狀態漂移有人手動改了不知道ArgoCD 偵測到差異，自動修正

跟之前學的對照

觀念
你已經會的在 ArgoCD 世界裡
kubectl apply -f deploy.yamlArgoCD 幫你做，你只需要 git push
kubectl rollout undogit revert + ArgoCD 自動同步
手動改 replicasArgoCD 偵測到跟 Git 不一致，改回來
記住「上次部署的是什麼」看 Git log

Ch2 — 安裝 ArgoCD

前置條件

觀念
  • Minikube 已啟動（minikube status 確認 Running）
  • kubectl 可以正常使用

Step 1 ~ 2：建立 namespace 並安裝

實作 
# Step 1：建立 ArgoCD namespace
kubectl create namespace argocd

# Step 2：安裝 ArgoCD
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# 等所有 Pod 都 Running（第一次可能要等 1~2 分鐘拉 Image）
kubectl get pods -n argocd

預期輸出——全部 Running 才算安裝完成：

NAME                                  READY   STATUS    AGE
argocd-server-xxx                     1/1     Running   ...
argocd-repo-server-xxx                1/1     Running   ...
argocd-application-controller-xxx     1/1     Running   ...
argocd-redis-xxx                      1/1     Running   ...
argocd-dex-server-xxx                 1/1     Running   ...

Step 3：存取 ArgoCD Dashboard

實作

ArgoCD 有 Web UI，用 port-forward 開通：

kubectl port-forward svc/argocd-server -n argocd 8080:443 &

開瀏覽器：https://localhost:8080

會出現 SSL 憑證警告（自簽憑證），點「繼續前往」即可。

Step 4：取得初始密碼

實作

帳號是 admin，密碼存在 Secret 裡：

kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

用這組帳密登入 Dashboard。

Step 5：安裝 ArgoCD CLI（選用）

實作 
# 下載 CLI
sudo curl -sSL -o /usr/local/bin/argocd \
  https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
sudo chmod +x /usr/local/bin/argocd

# CLI 登入
argocd login localhost:8080 --username admin --password <剛才的密碼> --insecure

Dashboard 看到空的 Applications 頁面，代表安裝成功。

Workshop：安裝 ArgoCD

Workshop
  • 在 Minikube 上安裝 ArgoCD
  • 用 port-forward 開啟 Dashboard
  • 取得初始密碼並登入
  • 截圖 Dashboard 空白頁面，貼到共筆

驗證指令：

kubectl get pods -n argocd
kubectl get svc -n argocd

練習回答

Q：安裝完 ArgoCD 後，如何確認所有元件都正常運行？

A：執行 kubectl get pods -n argocd，確認所有 Pod 的 STATUS 都是 Running，且 READY 欄位顯示 1/1。關鍵元件包含：argocd-server（Web UI / API）、argocd-repo-server（從 Git 拉取 YAML）、argocd-application-controller（比對 Git 與 K8S 狀態的核心）、argocd-redis（快取）、argocd-dex-server（SSO 認證）。如果有 Pod 卡在 ContainerCreating 或 ImagePullBackOff，通常是網路問題或 Image 還在下載中，等待或檢查網路即可。

Ch4 — Sync 與 Auto-Sync

三種同步模式

觀念

Application 建好了，第一次 Sync 也成功了。但每次改 Git repo 的 YAML，都要手動按 Sync——能不能自動？

模式行為適合場景
手動 Sync改了 Git，要自己按 Syncprod 環境，需要人工確認
Auto-Sync改了 Git，ArgoCD 自動同步dev 環境，快速迭代
Self-Heal有人手動 kubectl 改了 K8S，ArgoCD 自動改回 Git 的版本防止狀態漂移

Demo 1：手動 Sync

實作

Step 1：在 GitHub 上修改 YAML——把 deployment.yaml 的 replicas 從 2 改成 3。

Step 2：觀察 Dashboard——ArgoCD 會在幾分鐘內偵測到（預設每 3 分鐘檢查一次），狀態變成 OutOfSync。

# 或手動觸發檢查
argocd app get fastapi-app --refresh

Step 3：手動 Sync

argocd app sync fastapi-app

Pod 數量變更為新的 replicas 值。

Demo 2：開啟 Auto-Sync

實作 
# 用 CLI 開啟
argocd app set fastapi-app --sync-policy automated

或在 Application YAML 加上：

spec:
  syncPolicy:
    automated:
      prune: true       # Git 裡刪掉的資源，K8S 上也會自動刪除
      selfHeal: true    # 有人手動改 K8S，自動改回 Git 的版本
參數說明
automated開啟自動同步
prune: trueGit 裡刪掉的資源，K8S 上也會自動刪除
selfHeal: true有人手動改 K8S，自動改回 Git 的版本

測試：在 GitHub 上改 replicas 從 2 → 4，等幾分鐘，ArgoCD 自動部署，Pod 變成 4 個。

Demo 3：Self-Heal（自動修正狀態漂移）

實作

開啟 Self-Heal 後，手動改 K8S 會被自動修正。

Step 1：手動 scale 到 10

kubectl scale deployment fastapi-deploy --replicas=10
kubectl get pods

10 個 Pod 開始建立。

Step 2：ArgoCD 偵測到差異——幾秒後多餘的 Pod 開始 Terminating：

fastapi-deploy-bddb76dbc-5wxgs   1/1     Terminating   0          23s
fastapi-deploy-bddb76dbc-cpnql   1/1     Terminating   0          23s

ArgoCD 發現 K8S 上有 10 個 Pod，但 Git 寫的是 2 個，觸發 Self-Heal。

Step 3：確認回到 Git 定義的狀態

kubectl get pods
# 回到 2 個 Pod，跟 Git 一致
核心觀念：Git 是唯一真相，任何偏離都會被修正。

常用指令整理

實作
指令用途
argocd app get APP --refresh立刻檢查 Git 變更
argocd app sync APP手動同步
argocd app set APP --sync-policy automated開啟 Auto-Sync
argocd app set APP --self-heal開啟 Self-Heal
argocd app set APP --sync-policy none關閉 Auto-Sync

Workshop：Sync 模式實驗

Workshop
  • 在 GitHub 上修改 replicas，觀察 ArgoCD 顯示 OutOfSync
  • 手動 Sync，確認 Pod 數量變更
  • 開啟 Auto-Sync
  • 再次在 GitHub 上修改 replicas，觀察自動同步
  • 開啟 Self-Heal
  • 用 kubectl scale 手動改 replicas，觀察 ArgoCD 自動改回來

預期結果：

  • 手動 Sync：改完 Git 後按 Sync，K8S 更新
  • Auto-Sync：改完 Git 後不用操作，K8S 自動更新
  • Self-Heal：手動改 K8S 後，ArgoCD 自動改回 Git 的版本

練習回答

Q：Self-Heal 和 Auto-Sync 有什麼差別？什麼場景下要同時開啟？

A：Auto-Sync 處理的方向是 Git → K8S：當 Git repo 有新的 commit（YAML 被修改），ArgoCD 自動把變更部署到 K8S。Self-Heal 處理的方向是 K8S → Git：當有人繞過 Git 直接用 kubectl 修改 K8S（例如手動 scale），ArgoCD 會偵測到 K8S 跟 Git 不一致，自動把 K8S 改回 Git 定義的狀態。在 dev 環境中通常同時開啟兩者——Auto-Sync 讓開發者只需 git push 就能部署，Self-Heal 確保沒有人用 kubectl 偷改狀態導致環境跟 Git 不一致。在 prod 環境通常只開 Self-Heal（防漂移），不開 Auto-Sync（需要人工審核後才部署）。

Ch5 — GitHub Actions（CI）

情境：還有手動的步驟

觀念

目前流程中，還有三步是手動的：

改程式碼 → 手動 docker build → 手動 docker push → 手動改 YAML image tag → ArgoCD 自動部署
           ^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^^^^^^
           這三步還是手動的

GitHub Actions 可以自動化前面這段（CI = Continuous Integration）。

GitHub Actions 是什麼

觀念

GitHub 內建的自動化工具。在 repo 裡放一個 YAML 檔，GitHub 就會在指定事件（例如 git push）時自動執行。

.github/
└── workflows/
    └── ci.yaml    ← GitHub 看到這個檔案就知道要跑

為什麼用 commit hash 當 tag

觀念
tag 策略範例問題
latestapp:latest每次都一樣，K8S 不知道有新版
v1, v2app:v2要手動維護版號
commit hashapp:a1b2c3d自動產生，每次不同，可追溯到程式碼

第一個 Workflow：Build + Push Image

實作

.github/workflows/ci.yaml：

name: CI - Build and Push

on:
  push:
    branches: [main]
    paths:
      - 'app/**'

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and Push
        uses: docker/build-push-action@v5
        with:
          context: ./app
          push: true
          tags: ${{ secrets.DOCKER_USERNAME }}/docker-fastapi:${{ github.sha }}

Workflow 參數解說

觀念
區塊說明
on: push: branches: [main] + paths: ['app/**']main branch 的 app/ 目錄有變更時才觸發（改 YAML 不會觸發 CI）
runs-on: ubuntu-latest在 GitHub 提供的 Ubuntu 機器上跑
actions/checkout@v4拉 repo 程式碼
docker/login-action@v3登入 Docker Hub
docker/build-push-action@v5Build Image 並 Push
tags: ...:${{ github.sha }}用 commit hash 當 tag，每次都不同

設定 GitHub Repository Secrets

實作

到 GitHub 的 argocd-demo repo 頁面：Settings → Secrets and variables → Actions

在 Repository secrets 區塊，點 New repository secret，新增兩個：

NameValue
DOCKER_USERNAME你的 Docker Hub 帳號
DOCKER_PASSWORD你的 Docker Hub 密碼或 Access Token
注意：是 Repository secrets，不是 Environment secrets 或 Organization secrets。

Workshop：測試 CI

Workshop
  • 確認 argocd-demo repo 有 app/ 目錄和 .github/workflows/ci.yaml
  • 到 GitHub repo 的 Settings → Secrets → Actions，設定 DOCKER_USERNAME 與 DOCKER_PASSWORD
  • 修改 app/main.py（例如改回傳內容），git push
  • 到 repo 的 Actions 頁面觀察 CI 執行
  • 到 Docker Hub 確認新 Image 出現（tag 是 commit hash）

預期結果：git push 後 GitHub Actions 自動觸發，Actions 頁面顯示綠色勾勾，Docker Hub 出現新的 Image。

練習回答

Q：為什麼 CI workflow 的 on.push.paths 設定為 'app/**'？如果不設定 paths 會怎樣？

A：設定 paths: ['app/**'] 表示只有 app/ 目錄底下的檔案有變更時才觸發 CI。這是為了避免不必要的 build——例如只修改了 K8S 的 YAML 檔（不在 app/ 裡）或修改了 README，這些情況下程式碼沒變、Image 不需要重建。如果不設定 paths，每次 push 到 main 都會觸發 CI，即使只改了一個文件也會重新 build + push Image，浪費 GitHub Actions 的執行時間和 Docker Hub 的空間。

Ch6 — CI/CD 完整串接

完整 CI/CD 流程圖

觀念

上一章的 CI 會自動 build Image 並 push 到 Docker Hub，但部署到 K8S 還是要手動改 YAML 的 image tag。這一章把最後一段也自動化：

  開發者 git push（改程式碼）
    │
    ▼
  App Repo（程式碼 + Dockerfile + CI workflow）
    │ 觸發 GitHub Actions
    ▼
  GitHub Actions（CI）
    ├── 1. docker build
    ├── 2. docker push → Docker Hub
    └── 3. 更新 Config Repo 的 image tag
                         │
                         ▼
                    Config Repo（只放 K8S YAML）
                         │ ArgoCD 每 3 分鐘偵測
                         ▼
                    ArgoCD（CD）: OutOfSync → Auto-Sync
                         │ kubectl apply
                         ▼
                    K8S 部署新 Image，新版 Pod 上線
關鍵：全程只有第一步（git push）是人做的，其餘全自動。

架構：兩個 Repo

觀念
Repo內容誰在用
argocd-demo（App Repo）程式碼 + Dockerfile + CI workflow開發者 git push → 觸發 CI
argocd-config（Config Repo）K8S YAMLArgoCD 監控 → 自動部署
 
argocd-demo（App Repo）         argocd-config（Config Repo）
├── app/                        ├── deployment.yaml
│   ├── main.py                 └── service.yaml
│   ├── Dockerfile
│   └── requirements.txt
└── .github/workflows/
    └── ci.yaml

為什麼要分兩個 repo？App Repo 的 push 觸發 CI（build Image），Config Repo 的變更觸發 CD（ArgoCD 部署）。分開後不會有無限迴圈問題（CI 改了 Config Repo 不會重新觸發 CI）。

完整 CI Workflow（含更新 Config Repo）

實作

在 argocd-demo repo 的 .github/workflows/ci.yaml：

name: CI/CD - Build, Push, Deploy

on:
  push:
    branches: [main]
    paths:
      - 'app/**'

jobs:
  build-and-update:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and Push
        uses: docker/build-push-action@v5
        with:
          context: ./app
          push: true
          tags: ${{ secrets.DOCKER_USERNAME }}/docker-fastapi:${{ github.sha }}

      - name: Update Config Repo
        run: |
          git clone https://x-access-token:${{ secrets.CONFIG_REPO_TOKEN }}@github.com/4-learn/argocd-config.git
          cd argocd-config
          sed -i "s|image: .*/docker-fastapi:.*|image: ${{ secrets.DOCKER_USERNAME }}/docker-fastapi:${{ github.sha }}|" deployment.yaml
          git config user.name "github-actions"
          git config user.email "actions@github.com"
          git add deployment.yaml
          git commit -m "chore: update image to ${{ github.sha }}"
          git push

練習回答

Q：為什麼要把 App Repo 和 Config Repo 分成兩個？如果放在同一個 repo 會有什麼問題？

A：分開的核心原因是避免無限迴圈。如果程式碼和 K8S YAML 放在同一個 repo，CI workflow 在 build 完 Image 後會用 sed 修改 deployment.yaml 並 git push。這次 push 又會觸發 CI（因為 repo 有新的 commit），CI 又會 build Image、又改 YAML、又 push......形成無限迴圈。分成兩個 repo 後：App Repo 的 push 只觸發 CI（build Image），CI 改的是 Config Repo，而 Config Repo 沒有設定 CI workflow，所以不會再觸發任何 build。另外分開也有權限管理的好處——開發者只需要 App Repo 的 push 權限，Config Repo 的直接修改可以限制給 SRE/DevOps 團隊。

git revert HEAD --no-edit 是什麼意思？

部分說明
git revert產生一個新的 commit，把指定 commit 做的事反過來
HEAD指「最新的那個 commit」
--no-edit不跳出編輯器，直接用預設的 commit message
重要觀念：revert 不是「回到過去」，是「產生一個新 commit 來抵銷上一個 commit」。Git 歷史只會往前，不會刪東西。 
# revert 前：
  5c92ff0 chore: update image to v2        ← image 從 v1 改成 v2

# revert 後：
  d227984 Revert "chore: update image to v2"  ← 新 commit，把 v2 改回 v1
  5c92ff0 chore: update image to v2

練習回答

Q：為什麼在 GitOps 流程中不應該用 kubectl rollout undo 退版？git revert 和 git reset --hard 在退版上有什麼差別？

A：不用 kubectl rollout undo 的原因是它會跟 ArgoCD 打架。rollout undo 只改了 K8S 上的狀態，Git repo 裡的 YAML 還是指向 v2。如果開了 Self-Heal，ArgoCD 會發現 K8S（v1）跟 Git（v2）不一致，然後「修正」回 v2——等於你的退版被撤銷了。正確做法是在 Git 上操作，讓 ArgoCD 自動同步。

至於 git revert 和 git reset --hard 的差別：git revert 會產生一個新的 commit 來抵銷指定 commit 的變更，Git 歷史完整保留，適合已經 push 到遠端的情況。git reset --hard 則是直接把 HEAD 指標往回移，歷史記錄會被「刪掉」，需要 git push --force 才能推上去，在多人協作中非常危險（會覆蓋別人的 commit）。在 GitOps 流程中，應該永遠用 git revert——保留完整歷史，團隊可以追溯每一次變更和退版的原因。
