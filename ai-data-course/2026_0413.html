協作開發與虛擬化技術 II：Docker Compose 與 Kubernetes

Docker Compose 多容器編排、Docker/Compose/K8S 三層關係、Kubernetes 核心概念與實作（Pod / Deployment / Service / HPA）

Docker Compose — 一次啟動整個系統

為什麼需要 Docker Compose

觀念

情境：真實系統不只一個 Container。一個 AI 專案可能包含：

  • FastAPI：API 伺服器
  • PostgreSQL：關聯式資料庫
  • ChromaDB / Qdrant：向量資料庫
  • Worker：背景任務處理

用 docker run 多次的問題：

  • 啟動順序錯 → API 先跑但 DB 還沒好，直接噴錯
  • Port 撞 → 多個 Container 搶同一個 Port
  • 重開後不知道先跑哪個 → 每次都要記順序、手動輸入一堆指令
解法：Docker Compose — 用一個 YAML 檔定義多個 Container，一個指令全部啟動。

安裝 Docker Compose

實作

Docker Compose V2 以 plugin 形式安裝，指令從 docker-compose 變成 docker compose（空格取代連字號）。

# 安裝 Docker Compose V2
sudo apt install docker-compose-v2

# 確認版本
docker compose version

第一個 docker-compose.yml（FastAPI only）

實作

最簡單的範例——只有一個服務：

# docker-compose.yml
services:
  api:
    image: my-fastapi-app
    ports:
      - "8000:8000"

這樣就能用 docker compose up 啟動，不用每次打長長的 docker run -p 8000:8000 my-fastapi-app。

FastAPI + PostgreSQL 完整範例

實作

一個完整的兩服務 Compose 檔案：

# docker-compose.yml
services:
  api:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      - db
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb

  db:
    image: postgres:16
    environment:
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
      - POSTGRES_DB=mydb
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

volumes:
  pgdata:

docker-compose.yml 參數解說

觀念
參數用途範例
services定義系統中所有服務api、db、worker
build用 Dockerfile 建 imagebuild: .
image直接用現成 imageimage: postgres:16
ports主機 Port : 容器 Port"8000:8000"
depends_on啟動順序（先啟動依賴服務）depends_on: [db]
environment環境變數POSTGRES_USER=user
volumes資料持久化（Container 刪掉資料還在）pgdata:/var/lib/...

environment 怎麼知道設什麼

觀念

每個 image 支援的環境變數都不同，去 Docker Hub 查文件。

  • 搜尋 postgres → 找到 Environment Variables 區塊
  • 常見：POSTGRES_USER、POSTGRES_PASSWORD、POSTGRES_DB
  • MySQL：MYSQL_ROOT_PASSWORD、MYSQL_DATABASE
  • 不是自己亂猜，是官方文件寫好的

為什麼要 volumes

觀念

資料持久化：Container 預設是短暫的，刪掉就沒了。

  • 沒 volume → docker compose down 後資料庫資料全消失
  • 有 volume → 資料存在主機上，Container 重建後資料還在
  • Named Volume（如 pgdata）由 Docker 管理，不用自己指定路徑

服務間連線：用服務名稱當 hostname

觀念

Docker Compose 自動建立一個內部網路，服務之間可以用服務名稱互相連線。

  • API 連 DB → db:5432（不是 localhost:5432）
  • API 連 Redis → redis:6379
  • 服務名稱 = docker-compose.yml 裡定義的 key
# 在 API 的程式碼裡
DATABASE_URL = "postgresql://user:pass@db:5432/mydb"
#                                      ^^
#                              用服務名稱，不是 IP

Compose 常用指令

實作 
# 啟動所有服務（前景模式，看得到 log）
docker compose up

# 啟動所有服務（背景模式）
docker compose up -d

# 停止並移除所有服務
docker compose down

# 查看服務狀態
docker compose ps

# 查看 log
docker compose logs -f

Compose 與 Docker 指令對照

觀念
動作Docker 指令Docker Compose 指令
啟動docker run ...（每個服務各跑一次）docker compose up
停止docker stop container1 container2docker compose down
查看狀態docker psdocker compose ps
查看 logdocker logs container1docker compose logs
重新建置docker build + docker rundocker compose up --build

Workshop：FastAPI + DB 一鍵啟動

Workshop

建立 docker-compose.yml，用一個指令啟動 FastAPI + PostgreSQL。

  • 建立專案資料夾，準備 Dockerfile 與 docker-compose.yml
  • 執行 docker compose up -d
  • 用 curl http://localhost:8000 確認 API 正常
  • 用 docker compose logs -f 觀察兩個服務的 log
  • 用 docker compose down 一鍵停止全部

Docker → docker-compose → Kubernetes 三層關係

三層架構概觀

觀念
層級工具管什麼適用場景
Layer 1Docker單一服務本機開發、模型測試
Layer 2docker-compose一個系統（多服務）專案交付、團隊協作
Layer 3Kubernetes營運層級服務不中斷、自動擴縮、多機器
  • Docker：把一個程式包成 Container，在任何機器跑起來
  • docker-compose：把多個 Container 組成一個系統，一鍵啟動
  • Kubernetes：管理成千上萬個 Container，自動修復、擴縮、排程

為什麼「現在不急著教 K8S」

觀念

沒有規模就沒必要。

  • 你的專案只有 2-3 個服務 → docker-compose 就夠了
  • K8S 是為了「一台機器不夠」的時候準備的
  • 先學好 Compose，等遇到瓶頸再上 K8S
  • 但概念要先懂，面試會考

docker-compose 是 K8S 的「前置語言」

觀念

docker-compose 的概念可以直接對應到 K8S：

docker-composeKubernetes
servicesDeployment + Service
imagePod 裡的 Container image
portsService (NodePort / LoadBalancer)
depends_onInit Container / Readiness Probe
volumesPersistentVolume (PV) + PVC
environmentConfigMap / Secret

什麼時候需要 K8S

觀念

判斷準則：

  • 服務掛掉需要自動重啟 → 需要 K8S
  • 流量暴增需要自動擴展 → 需要 K8S
  • 需要多台機器分散負載 → 需要 K8S
  • 需要零停機更新（Rolling Update） → 需要 K8S
  • 只是本機開發 / 小型專案 → docker-compose 就夠
學習地圖：Docker（包東西） → docker-compose（組系統） → K8S（管營運）。先把前兩層做好，第三層的概念先理解、實作等需要時再深入。

Kubernetes 概念

K8S 是什麼

觀念

Kubernetes（K8S）是容器編排系統，由 CNCF（Cloud Native Computing Foundation）維護。

  • Google 內部系統 Borg 的開源版
  • 管理大量 Container 的部署、擴縮、修復
  • K8S = K + 8個字母 + S（Kubernetes 的縮寫）

Pod / Node / Cluster 核心概念

觀念
  • Pod：K8S 最小部署單位，裡面裝一或多個 Container
  • Node：一台機器（實體 or 虛擬），上面跑多個 Pod
  • Cluster：一群 Node 組成的集群，由 K8S 統一管理

類比：Pod = 員工、Node = 辦公室、Cluster = 整間公司。

K8S vs Docker vs Docker Compose

觀念
特性DockerDocker ComposeKubernetes
管理範圍單一 Container多個 Container（單機）多個 Container（多機）
自動修復無無有（Pod 掛掉自動重建）
自動擴縮無無有（HPA）
負載平衡無無有（Service）
滾動更新無手動自動（Deployment）
適用場景開發測試小型部署正式營運

K8S 解決的問題

觀念
  • 服務掛掉怎辦？ → K8S 偵測到 Pod 異常，自動重建新的 Pod
  • 機器壞了怎辦？ → K8S 把 Pod 搬到其他健康的 Node 上
  • 流量暴增怎辦？ → HPA 自動增加 Pod 數量分散負載
核心差異：Docker = 工具（你告訴它怎麼跑），K8S = 制度（你宣告期望狀態，它自己想辦法達成）。這就是「宣告式」(Declarative) vs「命令式」(Imperative) 的差別。

K8S 環境建置

kubectl 是什麼

觀念

kubectl 是跟 K8S 溝通的唯一窗口（CLI 工具）。

  • 所有操作都透過 kubectl 下指令
  • kubectl 會把指令送給 K8S 的 API Server
  • 類似 Docker 的 docker 指令
# 安裝 kubectl
sudo snap install kubectl --classic

# 確認安裝
kubectl version --client

實作

Minikube 在本機建立一個單節點的 K8S Cluster，用來學習和測試。

類型工具用途

正式環境EKS (AWS) / GKE (GCP) / AKS (Azure)正式營運、自動擴縮

curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64

Pod

Pod 是什麼

觀念

K8S 不直接管 Container，管的是 Pod。

  • Pod = K8S 最小部署單位，裡面裝一或多個 Container
  • 為什麼不直接用 Container？因為有時多個 Container 需要共享網路和儲存空間（容器群組概念）
  • 大多數情況：一個 Pod 裝一個 Container

Pod 是短命的

觀念

Pod 掛掉是正常的。K8S 的設計哲學就是假設 Pod 隨時會掛。

  • Pod 掛掉不會自動重啟（除非有 Deployment 管它）
  • Pod 可以換 Node——今天在 Node A，明天可能在 Node B
  • Pod 每次重建都是全新的（新 IP、新狀態）
重點：不要把 Pod 當寵物（小心呵護），要把它當牛（壞了就換一頭新的）。

Pod 操作指令

實作 
# 建立一個 Pod（用 nginx image）
kubectl run my-nginx --image=nginx

# 查看所有 Pod
kubectl get pods

# 查看 Pod 詳細資訊
kubectl describe pod my-nginx

# 刪除 Pod
kubectl delete pod my-nginx

# 再次查看——Pod 消失了，沒人救它
kubectl get pods

Workshop：建立 Pod → 刪掉 → 確認沒人救

Workshop
  • 用 kubectl run 建立一個 nginx Pod
  • 用 kubectl get pods 確認 Pod 正在運行
  • 用 kubectl delete pod 刪掉它
  • 再用 kubectl get pods 確認——Pod 消失了，沒有人自動補回來
  • 結論：裸 Pod 沒有自我修復能力，需要 Deployment

Deployment

Deployment 是什麼

觀念

Deployment 負責確保指定數量的 Pod 持續存在。

  • 你告訴 Deployment：「我要 3 個 nginx Pod」
  • Deployment 保證永遠有 3 個 Pod 在跑
  • Pod 掛掉 → Deployment 自動建新的補上
  • 這就是 K8S 的自我修復（Self-Healing）

Deployment YAML 範例

實作 
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-api
  template:
    metadata:
      labels:
        app: my-api
    spec:
      containers:
      - name: api
        image: nginx
        ports:
        - containerPort: 80
參數用途
apiVersionAPI 版本（Deployment 用 apps/v1）
kind資源類型（Deployment）
metadata.nameDeployment 名稱
spec.replicas要維持幾個 Pod
selector.matchLabels用 label 找到對應的 Pod
templatePod 的模板（要長什麼樣）
containersPod 裡的 Container 定義

Deployment 操作指令

實作 
# 套用 YAML 建立 Deployment
kubectl apply -f deployment.yaml

# 查看 Deployment
kubectl get deployments

# 查看 Pod（會看到 2 個 Pod）
kubectl get pods

# 刪掉一個 Pod → 觀察自動補新的
kubectl delete pod my-api-xxxxx
kubectl get pods  # 會看到一個新 Pod 自動產生

# Scale：修改 replicas 為 3
# 1. 編輯 deployment.yaml，把 replicas 改成 3
# 2. 重新套用
kubectl apply -f deployment.yaml
kubectl get pods  # 現在有 3 個 Pod

Workshop A：自我修復

Workshop
  • 建立 Deployment（replicas: 2）
  • 用 kubectl get pods 確認 2 個 Pod
  • 手動刪掉一個 Pod
  • 再查看 → Deployment 自動補了一個新的
  • 結論：Deployment 確保 Pod 數量永遠符合預期

Workshop B：1 份 → 2 份

Workshop
  • 修改 YAML 的 replicas: 1 → 2
  • 執行 kubectl apply -f deployment.yaml
  • 用 kubectl get pods 確認變成 2 個 Pod
  • 結論：Scale 就是改 replicas 數字再 apply

Service

為什麼需要 Service

觀念

Pod 每次重建 IP 會變，使用者連哪個？

  • Pod A 掛了 → Deployment 建了 Pod B → IP 不一樣了
  • 如果使用者直接連 Pod IP，每次 Pod 重建都要改設定
  • Service 提供穩定的連線入口，不管後面的 Pod 怎麼換，IP 都不變
  • Service 靠 label selector 找到對應的 Pod

Service YAML（NodePort）

實作 
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-api-service
spec:
  type: NodePort
  selector:
    app: my-api        # 找 label 為 app=my-api 的 Pod
  ports:
  - port: 80           # Service 的 Port
    targetPort: 80     # Pod 的 Port
    nodePort: 30080    # 對外暴露的 Port（30000-32767）
參數用途
type: NodePort讓外部可以透過 Node IP + Port 連進來
selector用 label 找到要轉發流量的 Pod
portService 本身的 Port
targetPortContainer 內部的 Port
nodePort外部存取的 Port（範圍 30000-32767）

Service 操作與測試

實作 
# 建立 Service
kubectl apply -f service.yaml

# 查看 Service
kubectl get svc

# 查看 Service 對應的 Pod（Endpoints）
kubectl get endpoints

# 取得 Minikube IP
minikube ip

# 用 curl 測試（Minikube IP + NodePort）
curl http://$(minikube ip):30080

# 刪掉一個 Pod → 服務不中斷
kubectl delete pod my-api-xxxxx
curl http://$(minikube ip):30080   # 還是能連！
重點：刪 Pod 服務不中斷！Service 會自動把流量導到其他存活的 Pod，Deployment 同時補新的 Pod 上來。

Workshop：建立 Service → curl 測試 → 刪 Pod 再 curl

Workshop
  • 確保 Deployment 有至少 2 個 Pod 在跑
  • 建立 Service（NodePort 30080）
  • 用 curl http://$(minikube ip):30080 測試
  • 手動刪掉一個 Pod
  • 再次 curl → 服務依然正常回應
  • 結論：Service + Deployment = 服務不中斷

HPA（Horizontal Pod Autoscaler）

HPA 是什麼

觀念

情境：不知道該開幾個 Pod？流量高峰開太少會掛，低峰開太多浪費錢。

  • HPA 只看 Pod 資源使用率（主要是 CPU）決定要幾個 Pod
  • CPU 使用率超過門檻 → 自動增加 Pod
  • CPU 使用率降低 → 自動減少 Pod

三元件分工

觀念
元件負責什麼類比
Service流量送給誰櫃台（統一入口）
Deployment維持幾個 Pod主管（確保人數）
HPA要不要調整 Pod 數量老闆（看業績決定加人減人）

HPA 設定步驟

實作

Step 1：啟用 metrics-server（Minikube）

minikube addons enable metrics-server

Step 2：修改 Deployment YAML 加上資源限制

# deployment.yaml（加上 resources）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-api
  template:
    metadata:
      labels:
        app: my-api
    spec:
      containers:
      - name: api
        image: nginx
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: "100m"       # 最少需要 0.1 CPU
          limits:
            cpu: "200m"       # 最多用 0.2 CPU

Step 3：建立 HPA

# 套用 Deployment
kubectl apply -f deployment.yaml

# 建立 HPA：CPU 超過 50% 就擴展，最少 1 個 Pod，最多 5 個
kubectl autoscale deployment my-api --cpu-percent=50 --min=1 --max=5

# 查看 HPA
kubectl get hpa

Demo：負載測試觀察自動擴展

實作

用 busybox 持續發送請求，觀察 Pod 數量自動增加。

# 開一個新終端，持續發送請求（模擬高流量）
kubectl run -i --tty load-generator --rm --image=busybox \
  --restart=Never -- /bin/sh -c \
  "while sleep 0.01; do wget -q -O- http://my-api-service; done"

# 在另一個終端觀察 HPA 變化
kubectl get hpa -w

# 觀察 Pod 數量變化

  • 負載開始 → CPU 使用率上升 → HPA 觸發擴展 → Pod 從 1 個變成 2、3、4...
  • 停止負載 → CPU 使用率下降 → HPA 縮減 Pod 數量（有冷卻時間）

YAML 基礎教學（附錄）

YAML 是什麼

觀念

YAML Ain't Markup Language — 一種人類容易讀寫的資料序列化格式。

  • 大小寫敏感：Name 和 name 不同
  • 用空格縮排（不能用 Tab），通常 2 個空格
  • # 為註解
  • 副檔名：.yml 或 .yaml 都行
  • Docker Compose、K8S、GitHub Actions 都用 YAML

YAML 資料類型

觀念

Object / Mapping（key-value）：

# Object（字典）
name: "FastAPI App"
version: 1.0
debug: true

Array / Sequence（列表）：

# Array（列表）
ports:
  - 8000
  - 8001
  - 8002

Scalars（純量）：

# 字串、數字、布林
name: "hello"     # 字串
count: 42         # 整數
ratio: 3.14       # 浮點數
enabled: true     # 布林

Python pyyaml

實作

用 Python 讀寫 YAML 檔案：

import yaml

# 讀取 YAML
with open("config.yaml", "r") as f:
    config = yaml.safe_load(f)

print(config)
# {'name': 'FastAPI App', 'version': 1.0, 'debug': True}

# 寫入 YAML
data = {
    "model": "gemini-2.0-flash",
    "temperature": 0.7,
    "tools": ["search", "calculator"]
}

with open("output.yaml", "w") as f:
    yaml.dump(data, f, default_flow_style=False, allow_unicode=True)

實用情境：AI 規則設定檔

實作

用 YAML 管理 AI 系統的規則設定：

# ai_rules.yaml
model:
  name: "gemini-2.0-flash"
  temperature: 0.3

safety_rules:
  - name: "安全帽偵測"
    condition: "未戴安全帽"
    severity: "high"
    action: "send_alert"

  - name: "反光背心偵測"
    condition: "未穿反光背心"
    severity: "medium"
    action: "log_warning"

thresholds:
  confidence: 0.8
  alert_cooldown: 300  # 秒
# Python 讀取設定檔
import yaml

with open("ai_rules.yaml", "r") as f:
    rules = yaml.safe_load(f)

for rule in rules["safety_rules"]:
    print(f"規則：{rule['name']}，嚴重度：{rule['severity']}")
重點：YAML 讓設定與程式碼分離。修改規則不用改程式碼，改 YAML 檔就好。Docker Compose 和 K8S 大量使用 YAML，先熟悉語法之後學更快。
